# v6.4-2 写作 Agent 功能清单

## 概述

当前 `script_writer` 的能力定位为**基础铺展**：将单条序列的场记表格展开为带对白与描写的剧本正文。v6.4 在其上注入三项专业写作能力，每一项都需要对 SKILL.md 正文做实质改写，同时可能涉及 reads 数组扩展或新增辅助 Skill。

---

## 能力一：序列 → 章节的跨切片整理与伏笔感知组织

### 现状

每调用一次 `script_writer(target_chapter=S1-1)` 只写 `sequences/S1-1.md` 对应的一个章节。切片间割裂——Writer 不清楚同幕其他序列写了什么、前章正文风格如何、伏笔在哪章铺设在哪章回收，各章之间缺乏叙事衔接。更严重的是，Writer 连已写好的前章**正文**都看不到，只看到序列节拍表（表格），第二章可能与第一章在文风、节奏、对白密度上完全不同。

### 目标

Writer 能感知同幕全局结构 + 前章实际正文文风，把连续的序列组整理为有节奏的「集/幕」单元。章节拆分不再是机械的"一个序列=一章"，而是考虑信息披露、伏笔布局、叙事节奏、文风延续后的有机组织。

### 具体能力

#### 1.1 跨序列上下文感知

- 当前 `reads` 只读了 `sequences/${normalizeToSequenceId(target)}.md` 一个序列（通过 `<current_sequence_beats>` 注入）
- v6.4 扩展为两重新信息源：
  - **同幕全部序列**：从 target 反推幕号（如 `S1`），将 `sequences/S1-*.md` 全部读入，聚合为 `<same_act_sequences>` 标签（每序列一个 `<slice id="...">` 子标签）。这比 ±1 相邻序列更完整——Writer 写 S1-3 时能感知 S1-1 和 S1-2 的全貌，而非仅 S1-2。
  - **前章正文**：找到已存在 `chapters/*.md` 中目标章节的前一章，读入全文注入为 `<previous_chapter_draft>`。这是 Writer 感知**实际文风**的唯一渠道——行为追踪摘要只管角色 OOC，管不了文风一致性。
- 由 `executeTool` 在 resolveExtraContext 阶段计算
- 仅首章边界：同幕只有 1 个序列时 `<same_act_sequences>` 只含当前序列本身；无前章时 `<previous_chapter_draft>` 为空（标签被自动过滤）

#### 1.2 伏笔落地核查前置化

- 当前 `<foreshadowing>` 只作为被动参照
- v6.4：SKILL.md 正文要求 Writer 在 reframe 阶段（写正文前）先扫描 `foreshadowing.md` 中所有标记了当前幕号（如 `S1`）的条目，明确：
  - 哪些 plant/payoff 落在同幕序列范围内
  - 哪些需要在本轮章节中处理
  - 哪些在前序章节已处理但不能忘记给 payoff 保留张力

#### 1.3 章节定位说明

每章开头用一句**章节定位说明**说明该章节在全剧/本幕中的叙事定位。示例：

> 「本章是 S1 的收束章节，在前三章建立的冲突基础上引爆 A-B 线的第一次正面交锋，同时为 S2 埋下 B 角色的动机转向伏笔（F-07 plant）」

### 接口变化

```yaml
# SKILL.md reads 无静态新增（frontmatter 不变）
# Engine 侧 resolveExtraContext 动态注入三枚标签：
# - <same_act_sequences>：同幕全部序列（聚合 XML）
# - <previous_chapter_draft>：紧前章节正文全文（CREATE 模式才有）
# - <character_behavior_tracking>：前章角色行为摘要（能力二产出）
```

## 能力二：角色对齐（防止 OOC）

### 现状

Writer 的 `reads` 包含 `characters.md`，但正文里只作为被动参考。LLM 在生成大量台词和动作时容易发生角色漂移——人物在不同章节间言行不一致（OOC），或者陷入"tell 而非 show"的模式。

### 目标

每个角色通过**事件驱动 + 具体言行**来塑造，而非设定描述。Writer 须维护角色言行一致性，对已有章节的角色模式保持记忆。

### 具体能力

#### 2.1 角色言行风格锚点提取

`characters.md` 中除了静态设定（年龄/职业/背景），应包含写作期可用的**言行锚点**：

| 锚点类型 | 说明 | 示例 |
|----------|------|------|
| 语言风格 | 句式偏好、用词习惯、语气 | "多用短句，习惯用反问，自称'老子'" |
| 身体语言 | 标志性动作/姿态 | "紧张时会摸左腕手表，愤怒时反而声音更轻" |
| 行为模式 | 压力下的典型反应 | "面对权威角色时回避眼神接触但坚持己见" |

这些锚点应通过 prompt 工程要求 Writer **在每个角色首次登场时按锚点塑造行动**，并在后续出场中保持一贯性。

> **不在 v6.4 范围内**：让 `characters` subagent 自动生成这些锚点。v6.4 只要求 Writer 的 SKILL.md 正文约定锚点格式与使用规范，锚点的填写暂时靠用户或 characters subagent 的后续升级手动完成。

#### 2.2 角色行为追踪（写作期内存）

writer 在单次会话（`sendMessage` 生命周期）内维护一个轻量级的**角色行为追踪表**，记录每个角色在本会话中已展现的核心行为：

```
林川：
  - 已表现：第 2 章拍桌怒斥王总 → 表明情绪底线在利益背叛
  - 已表现：第 3 章沉默离开会议室 → 习惯用克制表达失望
  - 待注意：第 4 章如要展现脆弱面，需铺垫
```

实现方式：

- 不在持久化文件层面额外写新文件
- 以 `TEMP_CHARACTER_TRACKING` memory 机制实现——engine 层在每次 script_writer 调用后将上次输出中识别出的角色行为摘要作为 `<character_behavior_tracking>` 标签注入当前上下文
- 简化实现：由 **SKILL.md 正文要求 Writer 在输出末尾附上一段 `<!-- BEHAVIOR_TRACK: 角色=行为摘要 -->` 注释**，engine 捕获后注入下一次调用

#### 2.3 OOC 自检指令

SKILL.md 正文末尾加入一条固定格式的**自检清单**，要求 Writer 落笔前快速过筛：

```
□ 所有登场的角色言行是否在 <characters> 范围内？
□ 是否有角色做了与之前章节矛盾的事情却未给出合理动机？
□ 每个角色的新行为是否有对应的触发事件？
□ 是否避免了"单纯描述角色心情"然后用说教性旁白解释？
```

### 接口变化

```yaml
# SKILL.md reads 无变化（characters.md 已存在）
# 增强点：SKILL.md 正文大幅扩充关于角色锚点 + 行为追踪 + OOC 自检的节
```

## 能力三：视听转化

### 现状

当前剧本正文以**对白主导 + 基础动作描写**为主体。心理活动、叙事说明、主题隐喻等抽象内容大量通过「他说」「她想」「旁白」直述。

### 目标

把不可见的内心世界和叙事手法转化为镜头语言、视听元素。类似「契科夫之枪——给特写」「紧张——眉头紧皱」「压抑——雨声 + 冷光」的视听化转译。

### 具体能力

#### 3.1 情绪/心理 → 视听转译规则

SKILL.md 正文包含一套**转译对照**（作为创作指南，非硬性规则表）：

| 抽象内容 | 视听化方案 |
|----------|-----------|
| 角色紧张焦虑 | 小动作（指甲掐掌心、舔嘴唇）、环境反射（雨声特写、钟摆声突出）、呼吸变化 |
| 隐瞒/说慌 | 台词语速异常、眼神回避、手势与话语不同步 |
| 权力关系 | 空间站位（坐/立、距离）、打断频率、谁先开口、称呼方式 |
| 道具重要性（契科夫之枪） | 特写镜头描写、角色互动时视线/触觉的额外停留 |
| 压抑感 | 色调暗示（冷光/阴影）、环境音过滤（只突出一种声源） |
| 释然/解脱 | 空间感开阔（描写视角从特写拉成全景）、环境声丰富回来 |

#### 3.2 叙事手法转译

| 叙事手法 | 视听表达 |
|----------|---------|
| 倒叙/闪回 | 画面切换标记（`*画面淡入·三小时前*`）、色调/光线变化 |
| 内心 OS | 保留但不滥用——仅在情绪峰值插入一句 **「（内心）** 标示 |
| 平行蒙太奇 | 场景交替描写 + `*同时，另一处……*` 过渡 |
| 象征/隐喻 | 通过环境元素实现：风雨预示冲突、镜子反射暗示双重人格、焚毁照片＝关系的终结 |

#### 3.3 场景渲染密度控制

- 章节开头/场景转换处允许较高密度视听描写（建立氛围）
- 对话中段保持克制，由台词驱动，纯描写不超过 2-3 行
- 情绪节拍点（转折/揭示/高潮）恢复描写密度用于定格关键瞬间

#### 3.4 避免"小说化"条约

明令禁止的写法：

- ❌ 长段无对白的纯散文式环境描写（超过 5 行无人物/对话）
- ❌ 全知叙述者心理分析（"他不知道，这是他最后一次见到她"）
- ❌ 直白情感标签取代行为（"她很伤心" → "她盯着杯沿冷凝水滑落的轨迹，没有说话"）

### 接口变化

```yaml
# 无 reads/writes/outputTags 变化
# 全部为 SKILL.md 正文的实质性扩充
```

## 四、三项能力的协同排布

三项能力不是三个独立的 skill，而是**同一 SKILL.md 内的三大章节**：

```
# 剧本正文写作（script_writer）v6.4
## 角色
...
## 上游参照区段（+ 同幕序列 + 前章正文 + 行为追踪）
## 双模式判定
## 核心能力

### 一、跨序列组织与伏笔感知
| 子节 | 内容               |
|------|-------------------|
| 1.1  | 章节定位说明       |
| 1.2  | 伏笔落地先检       |
| 1.3  | 同幕序列感知+前章正文衔接 |

### 二、角色对齐与反 OOC
| 2.1  | 言行锚点使用       |
| 2.2  | 行为追踪注释       |
| 2.3  | OOC 自检清单       |

### 三、视听转化
| 3.1  | 心理→视听转译      |
| 3.2  | 叙事手法转译       |
| 3.3  | 密度控制            |
| 3.4  | 禁"小说化"条约     |

## 字数控制
## 格式规范
## CREATE 形态示范
```

保持单一 SKILL.md 的**主要原因**：

1. 三项能力不是可选插件而是必须同时启用的写作规范——拆开会导致 LLM 遗漏
2. 没有多 skill 路由需求（写作期只有 script_writer 一个 active subagent）
3. 三项能力共享相同的 `reads/writes/outputTags`

### 为什么不动 skillLoader + SKILL.md 的 frontmatter 模式

- `reads` 不需要扩展新文件（相邻序列注入靠 engine 侧 resolveExtraContext，不入 frontmatter）
- `writes` 保持动态写靶模式
- `outputTags` 不变
- frontmatter 不出现 `subagent/owner/agent` 等禁用键

唯一的 engine 侧改动是 resolveExtraContext 阶段注入同幕序列标签 + 前章正文 + 行为追踪摘要。软校验在 executeTool 成功分支内追加 warning，不阻塞落盘。
