# 01 产品设计方案 — 7.0 用户化专项

## 1. 背景

StoryCrafter 3 当前是纯浏览器端 SPA（v6.9），存在五个核心痛点：

| 痛点 | 现状 | 影响 |
|---|---|---|
| 刷新即丢 | 资产存内存（`InMemoryFileManager`），无持久化 | 创作成果无法保留，关页面即清零 |
| key 暴露 | DeepSeek API key 焊死进浏览器 bundle，DevTools 可见 | 分享/部署不可行，key 易泄露 |
| 导入导出缺失 | 仅支持 `.md` 输入，无任何导出 | 无法接入已有 Word 大纲，无法导出成稿 |
| 单项目 | 无多项目概念，store 模块级单例 | 无法管理多个故事，互相污染 |
| 前端单调 | 单一暖色主题，0 个 `@media`，无暗色/无响应式 | 夜间用眼累，窄屏不可用 |

7.0 用户化专项针对这五个痛点提出 4 目标。

## 2. 目标（来自 `目标.md`）

1. **功能组件生态**：支持 Word 输入，支持 Word 和 MD 格式导出
2. **前端优化 3.0**：更美观
3. **存储生态**：支持多个对话框/项目，刷新不丢失，永久保留（除非用户手动改文件）
4. **后端层文件关系系统**

## 3. 版本规划

### v7.1：后端地基 + 持久化 + 多项目 + Word 导入导出
- **覆盖目标**：③ 存储生态 + ① Word 输入导出
- **核心**：引入 Node Express 后端，文件系统持久化，前后端分离
- **复用**：主体直接复用 `product_design_v6_后端专项/` 8 份设计文档的 6 阶段方案
- **粗估**：8–11 天

### v7.2：前端美化 3.0
- **覆盖目标**：② 前端优化
- **核心**：暗色主题 + 响应式断点 + 视觉调优
- **粗估**：2.5–3 天

### 文件关系系统（目标 ④）
- **本轮不实现功能**，仅在架构层预留 6 处扩展口子
- 未来版本按需启动（依赖图、影响分析、引用拓扑等）

> **版本顺序决策**：持久化与 Word 导入导出**同时开发**（v7.1 内），避免后端架构定稿后再加 Word 又要改；文件关系系统预留空间但不占独立版本；前端美化放最后（v7.2）。

## 4. 功能规格

### 4.1 v7.1 功能

#### 4.1.1 持久化与多项目（目标 ③）

**持久化**
- 所有资产落盘为真实 `.md` 文件：`server/data/projects/<projectId>/assets/*.md`
- 用户可用外部编辑器（VS Code 等）直接编辑 `.md`，刷新后改动生效——满足"除非用户手动改文件"
- 对话历史落 `chat_history.json`，执行日志落 `execution_log.jsonl`（追加写，崩溃安全）
- 项目元数据落 `metadata.json`

**多项目**
- 每个项目独立目录（projectId 用 UUID），物理隔离
- 支持项目列表、新建、切换、删除（二次确认硬删）
- 切项目时对话与资产全量重拉，互不串扰

**LLM key 保护**
- key 移到后端 `config.json`（`chmod 0600`），浏览器不暴露
- Settings 页可视化配置 provider（baseURL/apiKey/model）
- 支持测试连接

#### 4.1.2 Word 导入导出（目标 ①）

**Word 输入**
- 拖入 `.docx` 文件 → 后端解析为 markdown → 走现有 `input_normalizer` 归一化链路 → 分类为种子资产
- 复用现有 `_input_raw.md` 机制，归一化逻辑零改动

**MD 导出**
- 选中任意资产 → 一键下载 `.md`（纯前端 Blob，零后端依赖）

**Word 导出**
- 选中任意资产 → 后端生成 `.docx` 下载
- MVP 表格保真度降级为文本行，完整 fidelity 推后

### 4.2 v7.2 功能（目标 ②）

**暗色主题**
- HeaderBar 一键切换 ☀/🌙，localStorage 记住选择
- 全套色彩变量 + diff 高亮均有暗色版，WCAG AA 对比度

**响应式**
- 三档断点：
  - `wide`（≥1280px）：四栏现状（可拖拽分隔）
  - `medium`（768–1279px）：两栏（左对话+资产 Tab，右当前+基线 Tab）
  - `narrow`（<768px）：单栏 + 顶部 Tab
- 窄屏 diff 上下堆叠不丢失

**视觉调优**
- spacing scale + 字体层级变量化
- 空状态占位文案+图标
- 加载骨架（替代现状空白）
- 卡片 hover/状态圆点微动画

### 4.3 文件关系系统（目标 ④，本轮仅预留）

本轮**不实现任何关系功能**，仅在以下 6 处留结构口子，未来加功能不用改 schema 和路由命名空间：

1. `metadata.json` 预留 `relations?` 字段
2. `shared/src/types.ts` 预留 `AssetRelation` 类型
3. `server/src/routes/relations.ts` 预留 501 stub 端点
4. `web/src/api/relations.ts` 预留调用 stub
5. `assetStore` 的 `AssetState` 预留 `relations?`
6. `types/index.ts` 的 `AssetFileInfo` 预留 `relations?`

未来可实现：资产依赖图（改 worldbuilding 提示受影响下游）、影响分析、引用拓扑、关系可视化。

## 5. 用户场景

**场景 1（持久化+手编）**：创作者启动应用 → Settings 配置 LLM → 创作故事 → 刷新页面，资产仍在 → 用 VS Code 打开 `server/data/` 手改世界观 `.md` → 刷新，改动生效。

**场景 2（Word 导入）**：创作者有现成 Word 大纲 → 拖入 `.docx` → 自动归一化为世界观/角色/幕结构种子资产 → 继续创作。

**场景 3（Word 导出）**：创作者完成章节 → 选中章节资产 → 导出 Word → 发给编辑审阅。

**场景 4（多项目）**：创作者同时推进两个故事 → 新建"武侠长篇" → 切换到"都市短剧" → 两项目独立不串。

**场景 5（前端美化）**：夜间创作 → 切暗色主题护眼 → 笔记本窄屏 → 单栏 Tab 模式继续工作。

## 6. 本轮不做

**文件关系系统**（仅预留口子）：关系图 UI、自动依赖计算、关系可视化、影响分析、跨项目关系、关系变更历史。

**P3 远期项**：项目导出 zip、Skill marketplace、Anthropic/Gemini provider、Electron 打包、多用户/认证/云同步、版本快照/回滚、项目 fork、对话分页、`fs.watch` 自动刷新、Word 表格完整 fidelity、`.xlsx` 导入、多 profile 快速切换 UI、单 AssetCard 级导出、`~/.storycrafter/` 跨 clone 数据目录。

## 7. 与现有版本的关系

- v7.1 后端改造**保持四层框架**（Orchestrator → Subagent → Skill Router → Skill）不变，只改三个注入接口：`FileManager`（→ `HttpFileManager`）、`LLMClient`（→ 打后端）、`skillRegistry`（→ `GET /api/skills`）。
- `OrchestratorEngine` 业务内核（`validateOutput`/`assembleContext`/types）零改动。
- v6.9 的 writer pipeline、并发池、产品档案等能力全部保留。
