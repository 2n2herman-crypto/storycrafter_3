# StoryCrafter 3 — 开发规划

> **风格**：Vibe Coding — 每一步独立可运行、可预览  
> **步骤数**：9 步  
> **预估总工时**：约 10-12 天  
> **依赖**：Node.js 18+, DeepSeek API Key

---

## 前置约定

1. **每步提交即用**：每步完成时，代码处于可运行状态（`npm run dev` 能启动且无报错）
2. **前后步不矛盾**：后一步只新增/扩展文件，绝不推翻前一步的架构
3. **设计文档优先**：每次开始新一步前，先读 `设计文档目录与指南.md` 确认涉及哪些设计文档
4. **不重复造轮子**：设计文档已有的 TypeScript 接口、组件 Props、IPC 定义直接复用

## Step 4：AgentRegistry + Workflows + 通用调度引擎

**目标**：这是重构核心。实现 Agent 注册表、Workflow 编排定义、以及完全通用化的调度引擎（不再硬编码 Agent A/B）

**具体任务**：

4.1 实现 `src/orchestrator/agentRegistry.ts`：
  - 引用 `types/index.ts` 中定义的 `AgentSpec` 类型（Step 1 已定义），实例化 `AGENT_REGISTRY` 常量数组
  - `AGENT_REGISTRY` 先注册 MVP 的 2 个 Agent：
    - `subagent_a`（幕-序列编排师）：reads 4 文件（含 story_structure.md），writes 2 文件，outputTags 4 个（含 start + end tag）
    - `subagent_b`（场景-节拍精铸师）：reads 5 文件，writes 1 文件，outputTags 2 个（含 start + end tag）
  - 导出工具函数：`getAgentSpec(id)`, `getAgentsByPhase(phase)`, `getFilesByPhase(phase)`
4.2 实现 `src/orchestrator/workflows.ts`：
  - `WorkflowStage` 接口（agents[], parallel?, dependsOn?）
  - `PHASE_WORKFLOWS` 常量，定义 MVP 线性三步：
    - STATE_1_CONCEPT: []（无 Subagent，Orchestrator 直接 LLM 提炼）
    - STATE_2_MACRO: [{ agents: ['subagent_a'] }]
    - STATE_3_MICRO: [{ agents: ['subagent_b'] }]
  - 导出工具函数：`getWorkflow(phase)`, `getNextPhase(phase)`
  - `PHASE_ORDER` 映射常量移至 `stateMachine.ts`（供 AgentSpec.phase 与 Phase string 比较）
4.3 实现 `src/orchestrator/stateMachine.ts`：
  - `PHASE_TRANSITIONS` 流转表（从 Workflows key 读取）
  - `canAdvanceTo(phase)` 就绪条件改为从 Registry 动态计算：
    - Phase 1: 检查 worldbuilding.md / characters.md / plot_synopsis.md 三个直接产出文件是否已生成
    - Phase 2+: 查当前阶段所有 Agent 的 writes 文件是否已全部生成
  - 定义 `PHASE1_DIRECT_WRITES` 常量列表用于 Phase 1 就绪判断
  - `detectRollback(instruction)` 回退检测（关键词匹配）
4.4 实现 `src/orchestrator/contextAssembler.ts`：
  - `function assembleContext(agentSpec: AgentSpec, files: Record<string, string>): string`
  - 不硬编码 XML 标签，从 `agentSpec.reads` 自动生成：
    - 读取 reads 列表中的每个文件
    - 自动以 `<filename去掉.md>` 为标签包裹
4.5 实现 `src/orchestrator/outputValidator.ts`：
  - `function validateOutput(output: string, spec: { outputTags: string[], writes: string[] }): ValidationResult`
  - 通用化设计：接受任意 `outputTags + writes` 对，不限于 AgentSpec
  - 不硬编码校验 TAG，从 `outputTags` 读取：
    - 逐项检查输出中是否包含所有 start tag 和 end tag
    - 检查 start tag 和 end tag 之间是否包含合法 Markdown 内容
  - 提取逻辑：从 Agent 输出中定位 `<<<{PREFIX}_START>>>` 和 `<<<{PREFIX}_END>>>` 之间的内容，截取后写入对应 `writes` 文件
    - 前缀推导规则：`act_map.md → ACT_MAP`，`sequence_list.md → SEQUENCE_LIST`
  - 返回 `{ valid, missingTags, extracted }`
4.5a Phase 1 特殊处理（在 agentDispatcher.ts 中）：
  - Phase 1 没有 AgentSpec，但 LLM 输出同样需要按文件拆分
  - 定义 `PHASE1_OUTPUT_SPEC = { outputTags: [...], writes: ['worldbuilding.md', 'characters.md', 'plot_synopsis.md'] }`
  - 调用 `validateOutput(llmOutput, PHASE1_OUTPUT_SPEC)` 复用同一校验函数
  - Phase 1 的三个文件使用以下 tag：
    - `<<<WORLDBUILDING_START/END>>>`
    - `<<<CHARACTERS_START/END>>>`
    - `<<<PLOT_SYNOPSIS_START/END>>>`
4.6 实现 `src/orchestrator/agentDispatcher.ts`：
  - `class AgentDispatcher`
  - `dispatchForPhase(phase, instruction)`：
    - 从 workflows 查出当前阶段的 Agent 列表
    - 遍历 Agent 列表，对每个 Agent：
      ① 从 Registry 查 AgentSpec
      ② assembleContext(spec, files) → 组装 XML 标签上下文
      ③ 加载 spec.systemPromptFile → 构建完整 Prompt
      ④ call LLM → 拿到输出
      ⑤ validateOutput(output, spec) → 校验 + 重试(×3)
      ⑥ 提取表格 → 写入 spec.writes 文件
  - 重试时附加"格式错误，请严格遵循XML标识符和Markdown模板重新输出"
4.7 在 `chatStore.ts` 中接入 AgentDispatcher：
  - `sendMessage` 真实逻辑：phaseStore 判断 → agentDispatcher.dispatchForPhase → 更新 assetStore

**验证方式**：
- 验证 Registry：手动调用 `getAgentSpec('subagent_a')` 返回正确的身份卡
- 验证 Workflow：`getWorkflow('STATE_2_MACRO')` 返回 `[{ agents: ['subagent_a'] }]`
- 手动构造模拟的 Agent 输出，验证 outputValidator 能根据 outputTags 正确校验
- 测试重试逻辑：连续传格式错误的模拟输出，确认 3 次后停止

**涉及设计文档**：技术开发文档.md（第 4.0-4.5 节）、agent任务清单.md（全部）、全程的调度规则.md（第 1-4 节）、Agent-Orchestrator_system_prompt.md  
**新增文件**：`src/orchestrator/agentRegistry.ts`, `src/orchestrator/workflows.ts`, `src/orchestrator/stateMachine.ts`, `src/orchestrator/contextAssembler.ts`, `src/orchestrator/outputValidator.ts`, `src/orchestrator/agentDispatcher.ts`

## Step 8：全流程集成

**目标**：串联用户输入 → ChatStore → AgentDispatcher → LLM → FileManager → UI 刷新的完整闭环

**具体任务**：

8.1 在 chatStore.sendMessage 中接入真实 AgentDispatcher：
  - 用户发送消息 → chatStore.setProcessing(true)
  - → phaseStore 判断当前阶段
  - → intentPurifier 提纯意图
  - → 回退检测（detectRollback）
  - → agentDispatcher.dispatch(phase, instruction, files)
  - → 写入 fileManager
  - → assetStore.refreshAllFiles
  - → chatStore.addMessage(response)
  - → chatStore.setProcessing(false)
8.2 实现防抖逻辑：
  - chatStore.isProcessing = true 时，ChatInput disabled
  - 如果用户连续发送，ChatInput 显示"正在处理中..."
8.3 实现回退流程：
  - intentPurifier 返回 isRollback = true
  - 如果当前阶段是 STATE_COMPLETED → 前端弹出二次确认对话框："这将清空所有已创作内容（含基础设定文件），恢复至概念设定阶段。确定继续？"
  - 用户确认后 → phaseStore.rollbackToConcept()
  - → 从 Registry 查出当前阶段及之后的所有 Agent 的 writes 文件 → 清空
  - → 如果从 COMPLETED 回退，额外清空 Phase 1 直接产出文件（worldbuilding.md / characters.md / plot_synopsis.md）
  - → 清空对应的 *.approved.md
  - → 解除文件锁定状态
  - → assetStore.refreshAllFiles()
  - → chatStore.addSystemMessage("已重置到概念阶段")
8.3a 实现 COMPLETED 状态的"重新创作"按钮：
  - 在 STATE_COMPLETED 的 UI 中显示「重新创作」按钮（灰色，与「导出最终文件」并列）
  - 点击不经过意图提纯，直接弹出二次确认对话框
  - 确认后调用相同的 rollbackToConcept() 流程
  - 此按钮解决了 COMPLETED 状态下对话输入框禁用导致的回退无法触发问题
8.4 双模 Phase 推进逻辑：
  - PhaseSelector 下拉切换 → store.setViewingPhase() → 仅改变查看范围，不创建快照，不锁定
  - 如果在查看模式下用户发送修改指令 → 系统自动切回 currentPhase 再执行调度
  - ConfirmButton 点击 → saveSnapshot → setPhase(nextPhase)
  - → assetStore.refreshAllFiles()
  - → 自动选中下一阶段第一个文件
  - 推进后前一阶段所有卡片标记为 locked

**验证方式**：
- 完整的 Phase 1 流程：输入概念 → 生成 3 基础 MD → 确认 → Phase 2
- 处理中输入框禁用 → 完成后恢复（消息不排队）
- 输入"推翻重写" → 回退到 Phase 1（含二次确认对话框）
- PhaseSelector 下拉切换 → 只改变查看范围，不创建快照
- 查看模式下尝试发送 → 自动切回 currentPhase

**涉及设计文档**：全程的调度规则.md（全部）、Agent-Orchestrator_system_prompt.md  
**修改文件**：`src/store/chatStore.ts`, `src/store/assetStore.ts`, `src/store/phaseStore.ts`
