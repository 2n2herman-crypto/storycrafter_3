# 04 · 开发方案（v6.6 可执行实施计划）

> 本文是把 [03_产品设计文档.md](03_产品设计文档.md) 落到现有 v5.3/v6.1 代码基线的**分 Wave 施工图**：改哪些文件、影响面、类型改动、不做的事、风险。
> 代码锚点均来自当前基线（`orchestratorEngine.ts` / `skillLoader.ts` / `outputValidator.ts` / `scene_beats/structuralChecks.ts`）。

---

## 一、施工总原则（守既有不变式）

| 编号 | 不变式 | v6.6 的守法 |
|------|--------|------------|
| INV-1 | `outputValidator` 业务逻辑不动 | validator 档案化通过**传参**（ValidationSet）实现，不改校验主体逻辑 |
| INV-2 | `contextAssembler.assembleContext` / `buildAgentPrompt` 函数体不动 | `<product_profile>` 等新标签走 caller 侧 `appendExtraLabels`（已有机制） |
| INV-3 | `types/index.ts` / SkillSpec / SubagentSpec / frontmatter parser 零变更 | 产品类型新建 `src/types/product.ts`，不污染 index.ts；Pipeline 元数据仍嵌 `PIPELINE_REGISTRY` |
| INV-4 | Skill 归属靠目录、`FORBIDDEN_SKILL_KEYS` 不声明属主 | 4 个 writer + input_normalizer 沿用 `src/skills/<id>/<skillId>/SKILL.md` 目录约定注册 |
| INV-5 | 资产文件结构、表格骨架、ID 格式、TAG 不变 | 见 03 文档第七节红线 |

**改造边界**：所有产品差异收敛到 `ProductProfile` 一个对象 + 由它派生的注入/校验/可见性，**不在业务函数内散落 if(mode)**。

## 四、不做的事（明确排除，避免范围蔓延）

1. **不做并行批量成文**：短剧集间强因果，串行才能保 behaviorTrack/foreshadowingState 顺序一致。并行降级为后续可选优化（见风险 §五）。
2. **不做中途切换产品**：切换 = `reset_all` 重来。不实现"保留资产换产品"的迁移逻辑。
3. **不改文件结构/表格骨架/ID/TAG**：四产品共用同一套（守 INV-5）。
4. **不动 `contextAssembler.assembleContext` 函数体**：新标签一律走 `appendExtraLabels`（守 INV-2）。
5. **不引入 `js-yaml` 或第三方状态库**：沿用自研 frontmatter parser 与 Zustand。
6. **不做归一化的完美还原**：反向抽取是"尽力而为"建议稿。

## 五、风险与缓解

| 风险 | 等级 | 缓解 |
|------|------|------|
| 设计区 SKILL.md 去数字后 LLM 行为漂移（丢失原有隐性约束） | 高 | 逐 Skill 改造 + 人工跑四产品各一轮回归；`<product_profile>` 注入务必显式给区间，勿留"自行判断" |
| `orchestratorEngine` 改动密集引入调度回归 | 高 | 分 Wave 提交，每 Wave 后 `npx tsc -b` + 手动 smoke（`test_api.mjs`）；profileLock 为空时的降级路径要显式测试 |
| validator 多文件提取破坏 INV-1 | 中 | Wave 5 单独隔离；多文件提取写成 validator 外的**新函数**，不改 `validateOutput` 主体 |
| 长剧单集 5-10 场景超单次 LLM 稳定产出上限 | 中 | `long_drama_writer` 按场景分段续写（`proseSplitUnit:'scene'`，同文件 REFINE 追加） |
| **短剧一序列 8-15 集必超单次上限**（体量远大于长剧单集，END TAG 截断风险最高） | **高** | 短剧 `proseSplitUnit:'episode'` **默认按集分段**，非兜底而是内置成文路径；`batchProgress` 记录已完成集号断点续写（见 03 §5.3） |
| script_writer 删除后残留字面量分支（:382/461/547/608/843）漏改 | 高 | Wave 1/2 明确列出所有 `id==='script_writer'` 锚点；统一迁移到 `WRITER_IDS`；改后 grep 全仓确认无残留 |
| foreshadowingState 生产端（FORESHADOW 注释）漏在 writer 里输出 | 中 | Wave 2 把"输出 FORESHADOW 注释"列为四 writer 共享片段的**必含项**，与 BEHAVIOR_TRACK 并列 |
| 短剧 100 集 behaviorTrack 膨胀 | 中 | 按幕范围裁剪 + LRU 上限档案化；batchProgress 支持分批（7 批×15 集） |
| 无自动化测试兜底 | 高 | 本仓库无测试框架（已知缺口）；每 Wave 依赖 `npx tsc -b` + 人工四产品回归清单，改动前后对比资产 diff |

## 六、验收清单（四产品各跑一遍）

- [ ] 选定产品 → profileLock 落定 → 仅对应 writer 进 FC 面（其余三 writer 不可选）
- [ ] 设计区各层数量/语义随档案变化（短剧每幕恰 1 序列；长剧场景 5-10）
- [ ] validator 按档案切词库/列数（短剧四拍词库通过、5 类词库产品不误报）
- [ ] 任意文件输入 → input_normalizer 产出对应种子资产
- [ ] 成文风格互斥正确（小说有心理描写，剧本/长剧/短剧无）
- [ ] 短剧成文含 4 拍 + 镜头分解注释；长剧含集间钩子
- [ ] `reset_all` 释放 profileLock，可重选产品
- [ ] 全程 `npx tsc -b` 通过

## 七、推荐施工顺序

```
Wave 0（地基）→ Wave 1（锁定+可见性）→ Wave 2（四 writer）
   → Wave 4（validator 档案化）→ Wave 3（设计区去数字）
   → Wave 6（运行时记忆）→ Wave 5（输入归一化，独立可后置）
```

> Wave 4 先于 Wave 3：先让 validator 能吃档案，再放开设计区数字，避免设计区改完撞旧硬编码校验。Wave 5 输入归一化相对独立，可并入任意阶段或最后做。
