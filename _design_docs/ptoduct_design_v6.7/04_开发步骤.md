# v6.7 序列-场景-节拍链路重构 · 开发步骤

## Step 1 · 校验层（structuralChecks.ts）

**先做校验层，因为它是纯函数、零依赖、最易验证，且是引擎层的前置。**

1. 删除 `BEAT_TABLE_COLUMNS` 常量、`checkBeatTable` 函数（整表节拍校验）。
2. 保留 `SCENE_ID_REGEX`/`BEAT_ID_REGEX`/`SCENE_TABLE_COLUMNS`/`parseMarkdownTable`/`splitRow`/`findStrayPipe`/`checkSceneTable`/`extractSceneIds` 不动。
3. 新增 `BEAT_BLOCK_FIELDS` 常量、`BeatBlock` 接口、`parseBeatBlocks`、`checkBeatBlocks(md, sceneId, vset)`、`countBeatBlocks`（代码见 03 §3.3）。
4. 确认 `checkBeatBlocks` 的 `vset: ValidationSet` 类型 import 路径与 `checkSceneTable` 处一致。

**验证**：`npx tsc -b` —— 若报 `checkBeatTable` 仍被 `buildStructuralCheckForStep` 引用，是预期的，Step 4 会一起消解；此步可先只保证 structuralChecks.ts 自身无语法错。

## Step 2 · 引擎常量

1. 把 `PIPELINE_REGISTRY` 从 `{ steps: [...] }` 改为 `{ sceneStep, beatStep }` 结构（03 §2.1）。
2. **不新增 `PIPELINE_CONCURRENCY`**——偏差 ③（并行）本次不做，`runBatchPipeline` 走串行 `for...await`。
3. 此步会让所有引用 `.steps` 的旧代码报错——**预期**，Step 4-6 逐个替换。

## Step 3 · 抽公共重试块（重构，不改行为）

1. 从现有 `runPipeline` 里把「读 skill.reads → assembleContext → appendExtraLabels → sendMessage ×MAX_RETRIES → validateOutput → 返回 extracted 或 null」抽成私有 `runSingleStep(subagent, skill, seqId, instruction, history, opts)`，`opts = { structuralCheck, extras }`。
2. 加私有 `designExtras(priorDraft): ExtraLabel[]` —— 统一拼 `<current_target>`(priorDraft baseline) + `<product_profile>`(`renderProductProfileXml(profileLock)`）。
3. 此步先让 `runSingleStep` 存在但暂无调用方（或让旧 runPipeline 先改调它），保证 tsc 通过。

**验证**：`npx tsc -b`。此步是纯抽取，行为等价。

## Step 4 · runSequencePipeline（单序列三段式）

1. 把 `runPipeline` 重命名/改造为 `runSequencePipeline(subagent, pipe, seqId, instruction, history)`（03 §4.3）。
2. 段①建档：`writeFile(sequences/<seqId>.md, buildSequenceSkeleton(seqId))`。
3. 段②场景：`runSingleStep(...sceneSkill..., { structuralCheck: checkSceneTable, extras: designExtras })`。
4. 段③节拍：`extractSceneIds(scenesMd)` → for 循环逐场景 `runSingleStep(...beatSkill..., { structuralCheck: (x)=>checkBeatBlocks(x, sc, vset), extras: [prev_scenes, target_scene, ...designExtras] })`；失败降级为占位注释。
5. 收口 `assembleSequenceDoc` + 覆写 + 软校验（软校验函数 Step 7 再改，先传参占位）。
6. 加 `pipelineFail(subagent, seqId, stage, finalPath)` 辅助返回失败 ToolResult。

**验证**：`npx tsc -b`（此时 `buildStructuralCheckForStep` 若仍在可删——节拍校验已内联进 runSequencePipeline，场景校验直接调 `checkSceneTable`）。

## Step 5 · runBatchPipeline + 辅助

1. 新增 `runBatchPipeline(subagent, pipe, instruction, history)`（03 §4.2）：`parseSequenceIds` → **串行 `for...await`** 循环 `runSequencePipeline` → 汇总 ToolResult。
2. 新增 `parseSequenceIds`、`buildSequenceSkeleton`、`sliceSceneRow`（03 §4.5）。
3. **不新增 `runWithConcurrency`**——偏差 ③（并行）本次不做。

**验证**：`npx tsc -b`。

## Step 6 · assembleSequenceDoc + executeTool 分流

1. `assembleSequenceOutline`→`assembleSequenceDoc(seqId, scenesMd, sceneIds, beatBySc)`（03 §4.4）；删 `extractTableBlock`（确认无其他调用方）。
2. `executeTool` 的 PIPELINE 分支改分流（03 §4.1）：空靶→`runBatchPipeline`，合法靶→`runSequencePipeline`，非法靶→早退。

**验证**：`npx tsc -b`——此时应**全绿**（所有旧结构引用都已替换）。grep 确认 `extractTableBlock`/`checkBeatTable`/`.steps`/`assembleSequenceOutline` 零残留。

## Step 7 · 软校验改块计数

1. `runPipelineSoftValidation(scenesMd, beatBySc)`：场景数 `countTableDataRows(scenesMd)` 不变；节拍数改 `[...beatBySc.values()].reduce((n,md)=>n+countBeatBlocks(md),0)`；平均值区间判断逻辑不变（03 §4.6）。
2. 确认 `countTableDataRows` 仍被场景计数用到 → 保留；仅删对节拍整表的调用。

**验证**：`npx tsc -b`。至此引擎侧完成。

## Step 11 · 联调回归

1. `npm run dev`，选**短剧**产品。
2. 投喂/建好 `worldbuilding`/`characters`/`act_map`/`sequence_list`。
3. 说「把所有序列的场景节拍都铺出来」（空靶）：
   - **验偏差 ①（非表格 + 逐场景，防漏写 + 格式稳）**：随机抽查 2-3 个序列文件，确认场景表里**每个** SC-ID 下都有 `#### SC-ID` + 至少 1 个 `[BEAT ...]` 块，无跳过；观察执行日志 beat_writer 单场景调用基本一次过、不反复 retry；
   - **验偏差 ②（短剧资产单位）**：进写作期调 short_drama_writer，确认 `chapters/<序列号>.md` 为单位（非每集一文件），集号由 writer 内部 `## 第N集` 自管理；
   - **验偏差 ④（批量收敛）**：确认这是 **1 次 tool_call**（Orchestrator 视角原子），不占多 round 配额；资产面板逐张出现 sequences 卡片（串行，不并发）。
4. **精修**：「重写 S1-2 的场景节拍」→ 只 `sequences/S1-2.md` 变化，其余不动；若 S1-2 是旧表格格式，确认首次精修即转为块格式。
5. **写作期回归**：进写作期调对应 writer，确认 `<current_sequence_beats>` 注入新格式（外层 TAG + `# <seqId>` 未变），writer 正常产集。
6. **软校验**：故意让某序列场景数越界，确认 warning 经 `tool_complete.warnings` 透传 UI、不阻断落盘。
7. **降级**：模拟某场景节拍反复失败（可临时改 prompt 制造违规），确认该场景落 `<!-- 待补节拍 SC-xxx -->` 占位、整序列仍落盘、事件流有报告。

## 回滚与风险控制

- **分步提交**：Step 1-7（引擎/校验）一个 commit，Step 8-10（prompt）一个 commit，便于二分定位。
- **prompt 与引擎解耦验证**：Step 7 完成后引擎已能跑，但 beat_writer 仍输出旧表格 → `checkBeatBlocks` 会全部 retry 失败 → 降级占位。这是**预期中间态**，Step 8 重写 prompt 后才真正打通。故 Step 8 不可跳过或延后到联调。
- **墙钟时间**：串行批量铺 20+ 序列耗时较长（每序列 1 场景表 + 8-15 场景节拍次 LLM），属已知代价；若需压缩，未来加并发池（偏差 ③）而非本步处理。

## 不做的事（本轮边界）

- **偏差 ③（并行输出多序列）**：本次不做，串行批量替代；未来可加 `runWithConcurrency` + `PIPELINE_CONCURRENCY` 扩展。
- **偏差 ④ 的彻底解**（`MAX_ROUNDS`/`MAX_TOOLS` 扩容）——本次仅靠串行批量收敛缓解 scene_beats 阶段。
- **短剧 writer 覆写**——属写作模块，不在本次设计区。
- **不改场景层为非表格**——用户明确保留表格，仅节拍层改块。
- **不改 `types/index.ts`/`SkillSpec`/`SubagentSpec`/frontmatter parser**（守 INV-3）。
