# v5 Subagent 体系 — 技术框架变更计划

## Context

v4 有 14 个 Tool（6 gen + 6 refine + 1 reset + 1 orch），gen/refine 分开。v5 将 gen/refine 合并为单一 Tool，新增 3 个工具（伏笔追踪器、支线管理器、故事检查器），移除 plot_synopsis（职能被 act_map 吸收）。总计 9 个 Tool。

**关键架构决策**：拆除 dependsOn 硬闸门，所有 Tool 始终在 FC 列表中可见。结构感不靠代码拦截，而靠 prompt 引导 + reads 上下文的空标签机制自然维持。题材感知逻辑已确认去掉。

## Step 1: toolRegistry.ts — 重写注册表

### 1.1 替换 12 个 gen/refine ToolSpec 为 6 个合并 ToolSpec

**合并模式**：工具自身的文件放在 reads 中。自身文件不存在 → reads 为空 → LLM 自动创建模式；自身文件存在 → reads 有内容 → LLM 修改模式。

| 新 ID | reads | writes |
|-------|-------|--------|
| `worldbuilding` | `['worldbuilding.md']` | `['worldbuilding.md']` |
| `characters` | `['worldbuilding.md', 'characters.md']` | `['characters.md']` |
| `act_map` | `['worldbuilding.md', 'characters.md', 'act_map.md']` | `['act_map.md']` |
| `sequence_list` | `['worldbuilding.md', 'characters.md', 'act_map.md', 'sequence_list.md']` | `['sequence_list.md']` |
| `scene_beats` | `['worldbuilding.md', 'characters.md', 'act_map.md', 'sequence_list.md', 'scene_beat_outline.md']` | `['scene_beat_outline.md']` |

**dependsOn 字段保留但不做代码拦截**，转为元信息供 prompt 使用。

### 1.2 删除 2 个 ToolSpec

- `generate_plot_synopsis` / `refine_plot_synopsis` — 职能被 `act_map` 吸收
- `plot_synopsis.md` 不再是独立资产

### 1.3 新增 3 个 ToolSpec

| `story_checker` | 检查 | `['worldbuilding.md', 'characters.md', 'act_map.md', 'sequence_list.md', 'scene_beat_outline.md', 'foreshadowing.md', 'subplots.md', '_check_report.md']` | `['_check_report.md']` |

### 1.5 getAvailableTools() 全部放行（核心改动）

```typescript
// v4：硬闸门过滤
export function getAvailableTools(existingFiles: Set<string>): ToolSpec[] {
  return TOOL_REGISTRY.filter((tool) => {
    return tool.dependsOn.every((f) => existingFiles.has(f))
  })
}

// v5：全部工具始终可见

  return TOOL_REGISTRY
```

dependsOn 字段保留在 ToolSpec 中，仅作为 prompt 层面的"推荐顺序"提示，不再做代码拦截。

该函数已通用地遍历 TOOL_REGISTRY 聚合 writes 路径，加 Tool 后自动扩展。

### 2.2 新增 story_checker 特殊 result 注入

当前所有 tool_result 返回简短消息。story_checker 需要注入完整报告：

```typescript
if (toolSpec.id === 'story_checker' && result.success) {
  const report = await this.fileManager.readFile('_check_report.md')
  messages.push({
    role: 'tool',
    tool_call_id: toolCall.id,
    content: report,  // 完整报告
  })
} else {
  // 原有逻辑不变
  messages.push({
    role: 'tool',
    tool_call_id: toolCall.id,
    content: result.success
      ? `已成功执行 ${toolSpec.name}，输出已保存。`
      : `${toolSpec.name} 执行失败: ${result.error || '未知错误'}`,
  })
}
```

### 2.3 审计闭环 — 放在 FC 循环内部

FC 框架不变，让 orchestrator LLM 在看到 `_check_report.md` 内容后自主选择修复工具（scene_beats / sequence_list / subplot_manager）。

引擎层面只做两件事：

**① 审计轮次计数器** — 在 SchedulerState 中增加：

```typescript
interface SchedulerState {
  // ...原有字段不变
  auditRound: number       // 新增
  maxAuditRounds: number   // 新增 = 3
}
```

**② FC 循环头部超限检查**：

```typescript
while (state.currentRound < MAX_ROUNDS) {
  if (state.auditRound >= MAX_AUDIT_ROUNDS) {
    return {
      success: true,
      results: state.toolResults,
      response: '已完成检查和修复，部分问题可能需要你的进一步指导。',
    }
  }
  // ...原有循环逻辑
}
```

### 2.4 MAX_ROUNDS 不变

FC 循环保持 5 轮上限，审计独立 3 轮上限。二者不混用。审计修复占用 FC 轮次配额。

### Prompt 层面如何维持结构感（核心引导策略）

每个 Tool 的 prompt 中强调层级定位。以 `scene_beats.md` 为例：

```
你负责生成场景节拍（幕→序列→场景→节拍四层结构的第3-4层）。

生成原则：
- 如果上游文件（worldbuilding/characters/act_map/sequence_list）存在，参考它们做一致性对齐
- 如果上游文件不存在，基于 <user_revision_instruction> 直接生成，但输出必须保持
  幕ID → 序列ID → 场景ID → 节拍ID 的编号体系
- 输出表格必须包含所有 15 列（act_id, seq_id, 场景目标, 冲突与障碍, 场景结果, ...）
```

`orchestrator_v5.md` 中增加推荐顺序引导：

```
工具调用建议顺序（非强制，仅供参考）：
  世界观 → 角色 → 幕结构 → 序列 → 场景节拍 → 故事检查

你可以根据用户需求灵活调整顺序：
- 生成新故事 → 按推荐顺序逐步生成
- 修改特定内容 → 直接调对应工具
- 用户想看中间产出 → 跳过上游，直接调目标工具
```
