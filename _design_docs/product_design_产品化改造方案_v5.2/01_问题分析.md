# 01｜场景节拍 Agent 格式错误问题分析

## 一、现象

`scene_beats` 在用户实际使用中高频出现「格式错误」，导致场景节拍大纲无法正常写入 `scene_beat_outline.md`。

从代码链路看，错误在 Tool 执行阶段被归类为格式错误：

1. `orchestratorEngine.executeTool()` 调用 LLM；
2. `outputValidator.validateOutput()` 校验输出是否包含 ToolSpec 中声明的 START/END TAG；
3. 校验失败则追加通用格式提示重试，最多 3 次；
4. 3 次仍失败则返回 `输出校验失败` 或执行失败。

注意：当前 validator 只检查 TAG，不检查 Markdown 表格列数。因此用户看到的「格式错误」既可能是真正缺失 TAG，也可能是模型输出在长表格中被截断、被内容扰动、或未能完整收束到 END TAG。

## 二、旧版 `scene_beats` 的结构风险

旧版 prompt 要求输出「单张 Markdown 宽表」，包含 15 列：

1. 场景ID
2. 所属序列
3. 场景功能
4. 场景目标
5. 冲突与障碍
6. 场景结果
7. 时空边界
8. 视角人物
9. 节拍序号
10. 节拍类型
11. 动作-反应描述
12. 情绪/信息变化
13. 承接钩子
14. 关联伏笔
15. 关联支线

这个结构对 LLM 不稳定，原因如下。

### 1. 单表列数过多

Markdown 表格对 `|` 极度敏感。15 列宽表要求每一行稳定输出大量分隔符，任意一行漏列、串列、内容中误用管线符号都会破坏结构。

### 2. 行数随故事规模爆炸

Prompt 要求每序列 2-5 个场景、每场景 3-7 个节拍。若一个故事有 8 个序列，保守估算：

- 8 序列 × 3 场景 × 4 节拍 = 96 行；
- 96 行 × 15 列 = 1440 个表格单元格。

这已经超出模型稳定维护单一 Markdown 宽表的舒适区。

### 3. 场景信息被重复写入每个节拍行

同一场景的目标、冲突、结果、时空边界会在每个节拍行重复。重复不仅浪费 token，也增加不一致概率：

- 第一行写「办公室」，第二行可能漂移成「会议室」；
- 第一行结果是「主角被迫接受」，后续行可能变成「主角主动接受」；
- 同一场景内出现事实漂移，checker 后续也难判断。

### 4. 违背「场景 → 节拍」层级

参考 `Philosophy_story_structure.md`：

- 场景由目标、冲突、结果、时空硬边界定义；
- 节拍是场景内部的动作-反应与情绪/权力位移；
- 节拍没有独立时空边界。

旧表把场景和节拍压在同一行级结构中，导致模型更容易把节拍当成独立小场景写，或让节拍跨越场景边界。

### 5. 通用重试提示无法定位错误

`orchestratorEngine.ts` 中的重试提示是通用的：要求重新包含正确 TAG。它不知道具体是：

- 缺 START TAG；
- 缺 END TAG；
- 表格断裂；
- 输出被截断；
- 格式说明和内容生成互相干扰。

因此重试很可能重复失败。

---

## 三、v5.2 设计结论

本次不优先改 validator，而是先改 prompt 结构，因为：

1. 高频失败的主要诱因是输出协议过宽、过长、过扁平；
2. 改 prompt 即可降低生成结构难度，不影响现有 ToolRegistry；
3. 如果未来仍有失败，再增加 `scene_beats` 专用结构校验和恢复策略。

v5.2 将输出结构从：

```text
单张 15 列宽表：每行一个节拍，重复场景信息
```

改为：

```text
按序列分组
  ├─ 场景表：7 列，每场景一行
  └─ 节拍表：6 列，每节拍一行，引用所属场景
```

这样保留表格可读性，同时把每张表的列数控制在更稳定的范围内。
