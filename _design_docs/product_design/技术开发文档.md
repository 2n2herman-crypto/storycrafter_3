# StoryCrafter 3 — 技术开发文档

> **版本**：2.0（重构版 — 引入 Agent 注册表模式）  
> **日期**：2026-06-24  
> **开发策略**：先 Web 验证核心逻辑 → 再包装 Electron  
> **核心原则**：调度引擎通用化，Agent 信息注册表化，为 6+ Agent 预留扩展空间

---

## 一、项目全景

### 1.1 产品定位

StoryCrafter 3 是一个基于多 Agent 协作的**故事创作辅助工具**。用户通过与 AI 系统对话，逐步完成从世界观设定 → 宏观大纲 → 场景节拍精铸的完整创作流程。

远期扩展目标：**世界观开发、角色设计、支线设计、信息披露架构设计、伏笔设计、验收审阅**等专精 Agent 均可通过注册表声明式接入。

### 1.2 三阶段工作流（MVP）

| 阶段 | 输入 | 处理 | 输出文件 |
|------|------|------|---------|
| **STATE_1_CONCEPT** | 用户对话 | Orchestrator 直接提炼 | worldbuilding.md, characters.md, plot_synopsis.md |
| **STATE_2_MACRO** | 3 基础 MD + 修改指令 | → Subagent A | act_map.md, sequence_list.md |
| **STATE_3_MICRO** | 5 上游 MD + 修改指令 | → Subagent B | scene_beat_outline.md |

> 远期：每个阶段可以包含任意多个 Agent，Agent 之间可以是并行或串行依赖关系。  
> 当前 MVP 仅为线性 3 步，但调度引擎设计为通用的 DAG 执行器。

### 1.3 核心约束

1. **绝对不直接生成故事内容**：Orchestrator 只做调度和文件管理，不写剧本
2. **纯 Markdown 存储**：N 个 MD 资产文件 + `*.approved.md` 快照，无数据库
3. **XML 标签上下文隔离**：Agent 输入用 `<filename>` 标签包裹（标签名从注册表读取）
4. **正则输出校验**：Agent 输出必须含注册表中声明的 `<<<TAG>>>` 标识符
5. **严格单向状态机**：可自定义阶段数量，阶段间单向流转，推翻核心设定才回退
6. **Agent 权限声明式管理**：每个 Agent 的读/写/校验规则由 `AgentRegistry` 统一声明，调度引擎无硬编码

### 2.3 关键架构决策

| 决策 | 选择 | 理由 |
|------|------|------|
| **状态管理位置** | 前端 Zustand Store | 用户确认"前端管理为主"，状态机、调度逻辑均在渲染进程 |
| **Agent 信息管理** | AgentRegistry 声明式注册表 | 加 Agent = 加一条记录，不改调度引擎 |
| **编排方式** | Workflow DAG 定义 | 支持远期串行/并行/依赖关系 |
| **LLM 调用位置** | 前端直接调用（DeepSeek API） | DeepSeek 天然支持浏览器 CORS，Phase 1a 可直接调用；Electron 后移至主进程可选 |
| **数据持久化** | Web: 内存 Map / Electron: MD 文件系统 | 先验证逻辑，后加文件 IO |
| **前后端分离度** | 低（同一 Vite 项目） | 先确保核心闭环，不引入服务端复杂度 |
| **Electron 引入时机** | Phase 1 核心逻辑验证后 | 用 Web 模式跑通 Agent 调度 + UI 显示，再加壳 |

### 4.2 状态机 (stateMachine.ts)

状态定义改为从 Workflows 动态读取阶段列表：

```typescript
// MVP 4 个阶段。扩展新阶段时在此 union 追加即可（与 workflows.ts 的 key 保持同步）
type Phase = 'STATE_1_CONCEPT' | 'STATE_2_MACRO' | 'STATE_3_MICRO' | 'STATE_COMPLETED'

// Phase → 数字映射，用于 AgentSpec.phase（number）和 Phase（string）之间的比较
const PHASE_ORDER: Record<Phase, number> = {
  STATE_1_CONCEPT: 1,
  STATE_2_MACRO: 2,
  STATE_3_MICRO: 3,
  STATE_COMPLETED: 4,
}

const PHASE_TRANSITIONS: Record<Phase, Phase | null> = {
  STATE_1_CONCEPT: 'STATE_2_MACRO',
  STATE_2_MACRO: 'STATE_3_MICRO',
  STATE_3_MICRO: 'STATE_COMPLETED',
  STATE_COMPLETED: null,
}

// Phase 1 直接产出文件列表（Orchestrator 直接写入，非 Agent 产出）
const PHASE1_DIRECT_WRITES = ['worldbuilding.md', 'characters.md', 'plot_synopsis.md']

// 阶段就绪条件 → 从 Registry 动态计算（Phase 1 特殊处理）
function getRequiredFiles(phase: Phase): string[] {
  if (phase === 'STATE_1_CONCEPT') return PHASE1_DIRECT_WRITES  // Phase 1 直接产出文件
  const workflow = PHASE_WORKFLOWS[phase]
  if (!workflow || workflow.length === 0) return []
  return workflow.flatMap(stage =>
    stage.agents.flatMap(agentId => {
      const agent = AGENT_REGISTRY.find(a => a.id === agentId)
      return agent ? agent.writes : []
    })
  )
}

// 回退检测关键字（保持不变）
const ROLLBACK_KEYWORDS = ['推翻', '重写', '换主角', '改世界观', '重新设定', '重构']
```

### 4.3 Orchestrator 调度逻辑 (agentDispatcher.ts)

**从硬编码改为通用调度引擎**：

```
handleUserInput(userInput)
  │
  ├─ 1. 防抖检查
  │
  ├─ 2. 意图提纯 → 结构化指令
  │
  ├─ 3. 回退检测
  │
  ├─ 4. 从 workflow 读取当前阶段需要执行的 Agent 列表
  │
  ├─ 5. 遍历 Agent 列表（通用调度循环）:
  │     for each agentId in workflow[phase]:
  │       ① 从 Registry 查到 AgentSpec
  │       ② 读 AgentSpec.reads 中的文件 → 组装 XML 标签
  │       ③ 读 AgentSpec.systemPromptFile → 加载 System Prompt
  │       ④ call LLM(systemPrompt + 上下文)
  │       ⑤ 校验 AgentSpec.outputTags 是否都在输出中
  │       ⑥ 失败 & 重试<3 → 附加错误提示后重发
  │       ⑦ 成功 → 写入 AgentSpec.writes 中的文件
  │
  └─ 6. 返回处理结果 → UI 更新
```

核心代码结构：

```typescript
class AgentDispatcher {
  async dispatchForPhase(phase: Phase, instruction: string) {
    const stages = PHASE_WORKFLOWS[phase]
    if (!stages) return                                    // STATE_1 直接 LLM 提炼
    for (const stage of stages) {
      for (const agentId of stage.agents) {
        await this.dispatchAgent(agentId, instruction)
      }
    }
  }

  private async dispatchAgent(agentId: string, instruction: string) {
    const spec = getAgentSpec(agentId)                      // 从 Registry 查
    const context = await this.assembleContext(spec)         // 读 spec.reads 文件
    const prompt = this.buildPrompt(spec, context, instruction) // 加载 spec.systemPromptFile
    const output = await this.callWithRetry(prompt, spec)    // 校验 spec.outputTags
    await this.writeOutput(spec, output)                     // 写入 spec.writes
  }
}
```

### 4.4 通用上下文组装器 (contextAssembler.ts)

```typescript
function assembleContext(spec: AgentSpec, files: Record<string, string>): string {
  return spec.reads
    .map(file => {
      const tag = file.replace('.md', '')  // worldbuilding.md → worldbuilding
      return `<${tag}>${files[file] || ''}</${tag}>`
    })
    .join('\n')
}
```

未来加 Agent 只需在注册表的 `reads` 数组加文件名，无需改任何函数体。

### 4.5 通用输出校验器 (outputValidator.ts)

```typescript
function validateOutput(output: string, spec: AgentSpec): ValidationResult {
  const missingTags = spec.outputTags.filter(tag => !output.includes(tag))
  // 额外通用校验：输出必须不含 JSON/YAML frontmatter，必须是纯 Markdown
  // 提取：按 writes 文件名找对应的 start/end tag 对，截取中间内容
  const extracted: Record<string, string> = {}
  for (const file of spec.writes) {
    // 从文件名推导 tag 前缀：act_map.md → ACT_MAP
    const prefix = file.replace('.md', '').replace(/_/g, '_').toUpperCase()
    const startTag = `<<<${prefix}_START>>>`
    const endTag = `<<<${prefix}_END>>>`
    const startIdx = output.indexOf(startTag)
    const endIdx = output.indexOf(endTag)
    if (startIdx !== -1 && endIdx > startIdx) {
      extracted[file] = output.slice(startIdx + startTag.length, endIdx).trim()
    } else {
      extracted[file] = ''   // 提取失败，留空
    }
  }
  return {
    valid: missingTags.length === 0,
    missingTags,
    extracted,
  }
}
```

### 4.5a Phase 1 输出提取机制（特殊处理）

Phase 1（STATE_1_CONCEPT）由 Orchestrator 直接调用 LLM 生成三个基础文件，不走 Subagent。Orchestrator 的 System Prompt 要求 LLM 输出时使用与 Subagent 相同的 `<<<TAG>>>` 包裹协议：

```
LLM 预期输出（Orchestrator 阶段一调用）:
  <<<WORLDBUILDING_START>>>
  # 世界观设定
  ...
  <<<WORLDBUILDING_END>>>
  <<<CHARACTERS_START>>>
  # 角色设定
  ...
  <<<CHARACTERS_END>>>
  <<<PLOT_SYNOPSIS_START>>>
  # 剧情梗概
  ...
  <<<PLOT_SYNOPSIS_END>>>
```

提取逻辑在 `agentDispatcher.ts` 的 STATE_1_CONCEPT 分支中直接复用 `outputValidator` 的 tag 提取函数，但将 `outputTags` 和 `writes` 列表作为字面参数传入（而非从 AgentSpec 读取）：

```typescript
// agentDispatcher.ts — Phase 1 分支
const PHASE1_TAGS = {
  outputTags: ['<<<WORLDBUILDING_START>>>', '<<<WORLDBUILDING_END>>>',
               '<<<CHARACTERS_START>>>', '<<<CHARACTERS_END>>>',
               '<<<PLOT_SYNOPSIS_START>>>', '<<<PLOT_SYNOPSIS_END>>>'],
  writes: ['worldbuilding.md', 'characters.md', 'plot_synopsis.md'],
}
const result = validateOutput(llmOutput, PHASE1_TAGS)  // 复用通用校验器的 tag 扫描逻辑
if (result.valid) {
  for (const [file, content] of Object.entries(result.extracted)) {
    await fileManager.writeFile(file, content)
  }
}
```

这样 Phase 1 的文件拆分与 Subagent 阶段使用同一套校验机制，无需单独编写解析逻辑。

## 八、关键设计决策记录

| 决策点 | 方案 | 依据 |
|--------|------|------|
| Agent 信息管理 | **AgentRegistry 声明式注册表** | 加 Agent = 加一条记录，不改调度引擎 |
| Agent 编排 | **Workflow DAG** | 支持远期串行/并行/依赖 |
| 调度引擎 | **通用循环**（不再 if/else） | 从 Registry 读 Agent 列表，从 Workflow 读编排 |
| 上下文组装 | **从 AgentSpec.reads 自动生成 XML** | 加新 Agent 不需要写新组装代码 |
| 输出校验 | **从 AgentSpec.outputTags 通用校验** | 加新 Agent 不需要写新校验函数 |
| 状态管理位置 | 前端 Zustand Store | "前端管理为主" |
| 数据持久化 | Web: 内存 Map / Electron: 文件系统 | 先验证逻辑，后加文件 IO |
| 是否使用 AST 层 diff？ | **不**，文本层 `<ins>/<del>` | 保持 MD 表格结构完整性 |
| Electron 引入时机 | Phase 1 核心逻辑验证后 | 先 Web 后 Electron |
| **LLM 引擎** | DeepSeek API（OpenAI 兼容） | DeepSeek 原生支持浏览器 CORS，Phase 1a 可直接调用 |
| **Phase 1 输出拆分** | `<<<TAG>>>` 包裹协议（三文件共用一次 LLM 调用） | 复用 outputValidator 的 tag 扫描机制 |
| **Phase 1 就绪条件** | `PHASE1_DIRECT_WRITES` 常量（硬性文件列表） | Phase 1 无 Agent，需单独定义必需文件 |
| **COMPLETED 回退** | 「重新创作」按钮 + 清空 Phase 1 直接产出文件 | 解决输入框禁用无法触发回退的问题 |
