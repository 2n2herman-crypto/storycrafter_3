# v6.8 开发步骤

> 按此顺序落地，每步可独立编译验证。设计见 [02_开发方案.md](./02_开发方案.md)，产品逻辑见 [01_产品设计文档.md](./01_产品设计文档.md)。
> 原则：**先校验层（纯函数）→ 再并发（编排）→ 再 checker 约束（prompt + 引擎 Guard）→ 最后联调**。每步结束跑 `npx tsc -b`。
> 三块解耦，建议三段提交（① 校验 / ② 并发 / ③ checker 约束），便于二分定位。

## Step 9 · 引擎 Guard-3a + Guard-3b（dispatch 兜底）

1. 模块顶部加常量 `const UPSTREAM_DESIGN_IDS = ['worldbuilding','characters','act_map','sequence_list','foreshadowing_tracker','subplot_manager']`。

3. `processUserInput` 开头（[line 1192 附近](../src/orchestrator/orchestratorEngine.ts#L1192)）reset：`this.auditFixMode = false; this.auditScope = null`（不跨用户输入残留，避免"检查后用户改做别的"误拦上游）。

   const scope = scopeMatch?.[1] ?? null  // 'sequence_only' | 'has_upstream' | 'pass' | null(漏标)

## Step 10 · 联调回归

1. `npm run dev`，选**短剧**产品。
2. **校验放宽**（块①）：
   - 故意让 scene_designer 产「视角人物为空」的场景表（可临时改 prompt 或手改资产）→ 确认不再 retry、落盘正常。
   - 故意让 beat_writer 产「缺情绪字段」「相邻两拍同类型」「类型用词库近义词」的节拍 → 确认不再 retry、落盘正常。
   - 软校验仍报场景/节拍数越界 warning（不阻断）。
3. **并发**（块②）：建好 sequence_list（≥6 序列）→ 「铺所有序列」：
   - 资产面板短时间内集中出现多张骨架卡片（非逐张等待）；
   - 执行日志多序列 LLM 调用交错；
   - 墙钟对比 v6.7 串行近似减半；
   - 每序列每个 SC-ID 下仍有节拍块，无跳过（偏差①不退化）；
   - 仍是 1 次 tool_call（偏差④不丢）。
4. **429 模拟**：临时把 `PIPELINE_CONCURRENCY` 调到 10（或 mock sendMessage 抛 429）→ 确认退避重试生效、不崩；恢复后正常。
5. **部分失败**：临时让某序列 scene_designer 制造违规（如删场景表表格结构）→ 确认该序列失败降级、其余序列正常落盘、汇总 warnings 含失败清单。
6. **checker 精修**（块③）：
   - 制造单序列节拍问题（手改某序列节拍删除伏笔铺设）→ 调 story_checker → 确认报告「汇总发现」表标「序列 S1-3」+ 目标序列；
   - 确认 Orche 调 scene_beats 带 `target_sequence=S1-3` → 只 `sequences/S1-3.md` 变化、其余序列 diff 为空、上游设定文件不动；
   - 确认审计轮 FC 面无上游工具（Guard-3a 生效）。
7. **Guard-3a 上游放行**：制造 act_map 与 sequence_list 映射断裂 → checker 标「上游」→ 确认上游工具可见、Orche 可调对应上游工具修复。
8. **Guard-3b 拒收**：审计轮模拟 Orche 调 scene_beats 空靶 → 确认 executeTool 拒收、返回错误提示、不覆写任何序列。
9. **精修不进并发池**：带 `target_sequence` 走 `runSequencePipeline` 单序列路径（[line 531](../src/orchestrator/orchestratorEngine.ts#L531)），不进并发池。
10. **写作期回归**：进写作期调 short_drama_writer，确认 `<current_sequence_beats>` 注入正常、writer 产集。

## 回滚与风险控制

- **分块提交**：Step 1-3（校验放宽）一个 commit；Step 4-6（并发）一个 commit；Step 7-9（checker 约束）一个 commit。三块独立回滚。
- **AUDIT_SCOPE 字面对齐**：SKILL 输出的 `<!-- AUDIT_SCOPE: sequence_only -->` 字面必须与引擎扫描字符串逐字一致（半角空格、半角冒号）。Step 7 与 Step 9 联调时重点验证：手写两份 `_check_report.md`（含 `sequence_only` / 含 `has_upstream`）喂引擎，确认 Guard-3a 分别命中/放行。
- **`auditRound` 语义**：Guard-3a/3b 依赖 `auditRound > 0`。确认审计轮计数在 story_checker 首次成功后才置 >0，非审计轮（普通创作轮）为 0，避免误伤正常批量铺序列（空靶合法）。
- **并发墙钟**：若 429 频繁导致退避堆积，墙钟反劣于串行——联调观察；若如此先降 `PIPELINE_CONCURRENCY` 到 2，再考虑自动降并发（本轮不做）。
