# StoryCrafter 3 v5.2 场景节拍 Agent 稳定性改造

> 版本：v5.2  
> 目标：降低 `scene_beats` 在场景节拍大纲生成中的格式错误率，并让输出结构更符合「场景 → 节拍」的剧作层级概念。  
> 参考文档：`product_design_产品化改造方案_v5/Philosophy_story_structure.md`

---

## 一、改造背景

在当前用户使用中，`scene_beats` 是最容易因「格式错误」导致无法合理输出的 Agent。该错误通常表现为：

- 模型生成了不完整的 `<<<SCENE_BEAT_OUTLINE_START>>> / <<<SCENE_BEAT_OUTLINE_END>>>` 标签；
- 模型在长表格中漏列、串列、表格中断；
- 重试时只收到通用的「格式错误」提示，仍无法稳定恢复；
- 用户体验上表现为：其他资产已经完成，但微观场景节拍阶段断裂。

v5.2 的核心判断是：**这不是单纯的模型不听话，而是当前 `scene_beats` 输出协议本身过度脆弱。**

---

## 二、根因结论

当前 `scene_beats` 要求输出「单张 15 列宽表」，每一行代表一个节拍，并重复写入场景级信息。这个设计同时承担：

1. 场景边界定义；
2. 场景目标/冲突/结果；
3. 时空边界；
4. 节拍动作-反应；
5. 情绪/信息变化；
6. 伏笔与支线标记。

这导致一个序列下只要出现 3 个场景、每场景 5 个节拍，就会生成 15 行 × 15 列 = 225 个表格单元格。多序列生成时，单次输出很容易超过模型稳定维护 Markdown 表格结构的能力边界。

更重要的是，`Philosophy_story_structure.md` 明确指出：

- **场景是容器**：由目标、冲突、结果、时空硬边界定义；
- **节拍是场景内部单元**：由动作-反应和情绪/权力位移定义；
- 节拍没有独立时空边界，不能跨越场景。

但旧版 15 列宽表把「场景」和「节拍」压平成同一个行级结构，违背了层级概念，也增加了格式风险。

---

## 三、v5.2 改造目标

1. **降低格式错误率**：从单张超宽表改为分层 Markdown 结构，减少每张表的列数。
2. **恢复剧作层级**：场景先定义边界与动力结构，节拍再作为场景内部张力链展开。
3. **保留资产兼容性**：仍写入 `scene_beat_outline.md`，仍使用原有 TAG，避免改动 ToolRegistry 和前端资产系统。
4. **提升可读性**：用户能更清楚地看到每个序列的场景链与节拍链。
5. **降低一次生成压力**：使用「每个序列一组场景表 + 节拍表」的重复局部结构，而非全局巨表。

## 五、实际改动范围

本次 v5.2 先采用**低风险 prompt 改造**：

- 修改：`src/llm/prompts/scene_beats.md`
- 新增：`product_design_产品化改造方案_v5.2/` 方案文档
- 不修改：`toolRegistry.ts`、`outputValidator.ts`、`orchestratorEngine.ts`、前端 UI

原因：当前输出校验器只校验 TAG 是否存在；prompt 输出结构可以在不动代码的前提下从「单张宽表」改为「分层 Markdown 表格」。这能用最小工程风险先解决最影响用户体验的问题。
