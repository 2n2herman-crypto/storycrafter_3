# 01 问题分析

## 一、两类需求来源

StoryCrafter 的需求整理（`user_requirements_analyzer`）面对两类结构不同的需求：

### 1. 单轮判断得出型

用户在**一句话**里就把需求说清楚：

> "帮我写一个赛博朋克背景、女主角是黑客的短篇，结局要开放式。"

analyzer 拿到这一句就能提取出：世界观=赛博朋克、角色=女黑客主角、剧情方向=开放式结局。**当前实现已能处理**。

### 2. 多轮逐渐明晰型

需求跨越多条 message 才成型，尤其包含**指代性澄清**：

```
用户：写个科幻故事
系统：好的，已记录科幻背景。主角有什么设定吗？
用户：一个女科学家
系统：已记录。她的性格倾向？
用户：坚强一点，别太完美          ← "坚强""别太完美"指向上文的"女科学家"
用户：对，就按这个方向             ← 纯指代，脱离上文完全无意义
```

「坚强、有缺陷的女科学家主角」这条需求是**多轮累积**的结果。脱离对话上文，analyzer 无法把"坚强一点"关联到"女科学家"，**当前实现会丢失这条能力**。

## 二、当前记忆断点（代码级定位）

### 断点 1：`processUserInput` 每轮新建 messages

[orchestratorEngine.ts:216-218](../src/orchestrator/orchestratorEngine.ts)：

```ts
const messages: ChatCompletionMessageParam[] = [
  { role: 'user', content: userInput },   // 只有本次输入
]
```

函数结束后该数组丢弃，无跨轮持久化。

### 断点 2：chatStore 不回传历史

[chatStore.ts:83](../src/store/chatStore.ts)：

```ts
const result = await _engine.processUserInput(content, (event) => {...})
//                                            ^^^^^^^ 只传当前这一句
```

`chatStore.messages` 存了完整历史，但只用于 UI，从不进入引擎。

### 断点 3：后处理只更新状态，不合并内容

[orchestratorEngine.ts:273-286](../src/orchestrator/orchestratorEngine.ts) 的 `statusInstruction`：

> "根据本轮执行结果更新 user_requirements.md 的状态标记……仅更新状态标记，不修改需求内容。"

这是**状态标记模式**（把已实现需求标 ✅）。它不负责把用户本轮新说的需求合并进文档。而 analyzer 的**内容合并**只在 Orchestrator 主动通过 Function Calling 选择它时才发生——不保证每轮都发生。

### 断点 4：executeTool 只传孤立 instruction

[orchestratorEngine.ts:156](../src/orchestrator/orchestratorEngine.ts)：

```ts
let userContent = buildAgentPrompt(context, instruction)
```

`instruction` 是 Orchestrator 从当前输入里提取的片段，不含对话上文。analyzer 即使被调用也看不到指代所指。

---

## 三、结论

`user_requirements.md` 已具备"持久记忆载体"的地位（持久化 + 被下游 `reads`），缺的是**两个写入保障**：

1. 需求内容合并**不保证每轮发生**（依赖 Orchestrator 的选择）。
2. 合并时**看不到对话上文**（无法解析指代性澄清）。

v5.5 针对性补齐这两点，见 `02_设计方案.md`。
