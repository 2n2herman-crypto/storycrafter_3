# v6.7 序列-场景-节拍链路重构 · 改动环节清单

> 对应 [目前存在的偏差.md](./目前存在的偏差.md)：本次完成 **①（非表格范式）、②（短剧资产单位）、④（工具调用量）**，不做 **③（并行输出）**。
> 本文档只回答「要动哪些环节、为什么动」，具体设计/实现见 02/03/04。

## 一句话目标

把 scene_beats 从「一次调用 = 一个序列的两步流水线（场景表 → 整序列节拍总表 → 拼装）」重构为：

**一次调用 = 串行批量铺全部序列，每个序列走三段式【① 建档 → ② 写场景（保留表格）→ ③ 逐场景写节拍（改非表格块）】**，并保留「带 target_sequence 时只重做单个序列」的精修入口。

> 串行批量 = 一次 tool_call 内部 `for` 循环铺全部序列，对 Orchestrator 原子化为 1 次调用（不占多 round 配额，解偏差 ④）；不做并发（偏差 ③ 留作未来扩展）。

## 四条偏差如何被这次改动消解

| 偏差 | 根因 | 本次如何解 |
|---|---|---|
| ① 超长输出致格式崩坏，需改非表格范式（从场景节拍开始） | beat_writer 一次为整序列产 60-90 行大表，表格范式长输出易崩 + 模型偷懒漏写场景 | 节拍改**非表格字段块** + **逐场景独立 LLM 调用**（引擎按场景 ID 循环，每次只写一个场景的 4-6 拍）——机制上无法跳过、单场景短输出几乎不崩 |
| ② 短剧资产文件单位应为序列级（一序列含 8-15 集），总文件数 = 上游场景节拍输出数 | target 协议未强制序列级，Orche 误传集级 ID 时每集落独立文件 | **已落地**：`normalizeToSequenceId` 归约兜底（集级 ID → 序列号）+ `resolveWriteTarget`/`runPipeline` 落盘用归约 ID + prompt/SKILL 纠偏；本设计文档仅声明契约 |
| ③ 场景节拍支持并行输出多序列 | `runPipeline` 单序列、dispatch loop 串行 | **本次不做**；用串行批量（一次 tool_call 内 for 循环）替代，未来可扩展为并发池 |
| ④ 工具调用量不够（短剧 20+ 序列超单次对话总任务量） | 每序列各占 1 个 round/tool 配额 | 串行批量收敛：空靶 = 一次 tool_call 内部串行铺全部序列，不占多 round 配额 |

## 引擎侧改动细分（第 4 项展开）

现有 `runPipeline`（单序列两步）→ 拆为三个层级：

```
runBatchPipeline(subagent, instruction, history)         ← 新增：全序列串行批量编排器（空靶入口）
  ├─ 读 sequence_list.md，正则解析出全部序列 ID [S1-1, S1-2, ...]
  ├─ for (seqId of seqIds) { await runSequencePipeline(seqId) }   ← 串行循环，不并发
  └─ 汇总各序列成败 → 单个 ToolResult（writes = 全部成功的 sequences/*.md）

runSequencePipeline(subagent, seqId, instruction, history)  ← 由 runPipeline 改造：单序列三段式
  ├─ ① 建档：writeFile(sequences/<seqId>.md, 骨架占位)          ← 零 LLM
  ├─ ② 场景：scene_designer 一次 LLM → 场景表（内存）           ← 保留表格
  ├─ ③ 节拍：解析场景表得 SC-ID 列表，逐场景循环调 beat_writer  ← 逐场景、非表格块
  │        （每场景 ≤MAX_RETRIES；每个块经 checkBeatBlocks 单场景校验）
  └─ 收口：assembleSequenceDoc(seqId, 场景表, 各场景节拍块[]) → 覆写落盘
```

引擎侧函数增删：

- **新增** `runBatchPipeline`（全序列串行批量编排；`for` 循环 `await`，**无并发池**）
- **改造** `runPipeline` → `runSequencePipeline`（单序列三段式；带 target 精修入口复用它）
- **改造** `assembleSequenceOutline` → `assembleSequenceDoc`（拼装参数从「整表 beatsMd」改为「逐场景块数组」）
- **删除** `extractTableBlock`（仅 assemble 用，节拍不再是表格）
- **删除/改造** `countTableDataRows`：节拍计数改 `countBeatBlocks`；场景数仍数表格行（场景保留表格）
- **改造** `executeTool` 的 PIPELINE_REGISTRY 分流：空 target 走 `runBatchPipeline`，合法 target 走 `runSequencePipeline`（去掉「必须有 target」的早退）
- **改造** `runPipelineSoftValidation`：节拍数量用块计数

> 不新增 `runWithConcurrency` 与 `PIPELINE_CONCURRENCY`——偏差 ③（并行）本次不做。

## 决策记录（已与用户确认）

- **节拍粒度**：逐场景独立 LLM 调用（防跳过的机制性保证，非靠 prompt 约束）
- **批量边界**：引擎内一次 tool_call **串行**铺全部序列（`for` 循环 `await`，不并发），对 Orchestrator 是 1 次 tool_call
- **精修入口**：保留——带合法 `target_sequence` 时只重跑该序列（走 `runSequencePipeline`）
- **场景层**：保留 Markdown 表格（用户明确「参考之前的表格就行」）；仅节拍层改非表格块
- **偏差 ②**：已落地（代码侧 `normalizeToSequenceId` 兜底），本次设计文档仅声明契约

## 明确不在本次范围

- **偏差 ③（并行输出）**：本次不做，串行批量替代；未来可加并发池（`runWithConcurrency` + `PIPELINE_CONCURRENCY`）扩展
- 偏差 ④ 的彻底解（`MAX_ROUNDS`/`MAX_TOOLS` 扩容）——本次仅通过串行批量收敛缓解 scene_beats 阶段
- 场景表也改非表格——用户选择场景保留表格，仅节拍层改块
- 短剧 writer 覆写——属写作模块，不在本次设计区

## 不变式守恒（沿用 v6.x）

- **INV-1**：`validateOutput` 主体不改；`checkBeatBlocks` 仍走 `SkillSpec.structuralCheck` 钩子
- **INV-2**：`assembleContext`/`buildAgentPrompt` 函数体不动；`<target_scene>` 等新标签经 `appendExtraLabels` 注入
- **INV-3**：优先不改 `types/index.ts`；子进度事件复用现有 `message` 字段
- **INV-4**：scene_beats 三个 skill（scene_designer/beat_writer/建档无需 skill）沿用目录约定，不声明属主
