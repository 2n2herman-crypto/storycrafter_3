# v6.7 序列-场景-节拍链路重构 · 产品设计文档

## 1. 背景

对应 [目前存在的偏差.md](./目前存在的偏差.md)，本次完成 ①②④、不做 ③。用户手动跑短剧全流程时在「序列 → 场景节拍」环节发现的问题与偏差映射如下：

- **偏差 ①（超长输出致格式崩坏）**：表现为两类——
  1. **节拍漏写**：一个序列有 1-6 个场景，模型只给最后一个场景写了节拍，前几个被跳过（非高频但确实出现）。
  2. **输出不稳**：平均要失败 2 次才能稳定产出场景节拍文件（表格范式长输出崩坏）。
- **偏差 ②（短剧资产文件单位应为序列级）**：已落地（`normalizeToSequenceId` 归约兜底 + 落盘用归约 ID）。
- **偏差 ③（并行输出多序列）**：本次**不做**，用串行批量替代。
- **偏差 ④（工具调用量不够）**：20+ 序列逐个来撞满调用轮次上限。

根因是当前架构把「整个序列的所有场景节拍」压在**一次 LLM 调用产一张 60-90 行大表**里，既容易漏（模型偷懒）、又容易崩（长表格脆弱）；且每序列各占 1 次 tool_call 配额。

## 2. 设计目标

把 scene_beats 重构为一条**串行批量、防漏写、格式稳**的三段式链路，同时保留单序列精修能力。偏差 ③（并行）本次不做，留作未来扩展。

## 3. 新链路产品逻辑

### 3.1 用户视角

用户在设计期说「把所有序列的场景节拍都铺出来」（或点某个批量按钮）：

```
用户一句话
  → scene_beats 被调用（不带具体序列号 = 全量）
  → 系统读取序列清单，识别出全部 N 个序列
  → N 个序列文件【串行循环】生成（一次 tool_call 内部 for 循环，不占多 round 配额）
  → 每个文件内部：先有场景表，再为每个场景铺节拍
  → 全部完成，资产面板逐张出现 N 张序列卡片（建档骨架先现，再逐步填充）
```

用户若只想重做某一个序列：「重写 S1-3 的场景节拍」→ 带序列号调用 → 只重跑该序列，其余不动。

### 3.2 每个序列文件的三段式生产

一个 `sequences/<序列ID>.md` 的诞生分三段：

| 段 | 名称 | 谁来做 | 产物 |
|---|---|---|---|
| ① | **建档** | 引擎代码（零 LLM） | 先落一个带标题骨架的 `sequences/<ID>.md`，占位声明「本序列场景/节拍待填」 |
| ② | **写场景** | scene_designer（1 次 LLM） | 该序列的**场景表**（沿用现有 7 列表格：场景ID/功能/目标/冲突/结果/时空/视角） |
| ③ | **写节拍** | beat_writer（**逐场景**多次 LLM） | 对场景表里的**每一个场景**单独调用一次，产出该场景的节拍（**非表格字段块**） |

段③是本次防漏写的核心：引擎从段②的场景表解析出场景 ID 列表 `[SC-S1-1-01, SC-S1-1-02, ...]`，**按列表逐个**调 beat_writer，每次只把「这一个场景」丢给模型让它写节拍。模型每次面对的是单场景小任务，机制上不可能"从最后一个场景开始漏掉前面"。

### 3.3 为什么这样分能同时解决三个问题

- **防漏写**：段③循环由引擎控制场景遍历，不是让模型自己决定给几个场景写节拍。场景数 = 节拍调用次数，一一对应。
- **格式稳**：段③每次输出只有一个场景的 4-6 拍（约 6-10 行块），远小于原来的整序列大表；且节拍改用**字段块**而非表格，无 `|` 分隔崩坏风险。
- **批量收敛（解偏差 ④）**：段①②③封装在「单序列流水线」里，全部序列由引擎 `for` 循环串行跑完，对上层是 1 次 tool_call，不占多 round 配额。偏差 ③（并行）本次不做——串行墙钟时间较长属已知代价，未来加并发池缓解。

## 4. 节拍非表格范式

### 4.1 旧（表格，废弃）

```
| 所属场景 | 节拍序号 | 节拍类型 | 动作-反应描述 | 情绪/信息变化 | 关联伏笔 |
|---------|---------|---------|-------------|-------------|---------|
| SC-S1-1-01 | B-SC-S1-1-01-1 | 铺垫 | 机械处理邮件 → 自觉三年原地踏步 | 麻木透厌倦 | — |
```

### 4.2 新（字段块，采纳）

每个场景的节拍以「块」为单位，块内每字段一行：

```
[BEAT B-SC-S1-1-01-1]
类型: 铺垫
动作: 机械处理邮件 → 自觉三年原地踏步
情绪: 麻木透厌倦
伏笔: —

[BEAT B-SC-S1-1-01-2]
类型: 触发
动作: 小赵随口问周末安排 → 林川答不上来愣神
情绪: 自觉生活失焦
伏笔: F02
```

> 因为段③是**逐场景**调用，节拍块的「所属场景」由引擎在调用时已知（`<target_scene>` 注入），无须模型在每块重复填写 SC-ID；块头 B-ID 里已内嵌场景 ID，校验时反查即可。这比旧表格每行都要填「所属场景」列更省、更不易错。

### 4.3 选型理由

- **块范式 vs 表格**：块无 `|` 列分隔依赖，长内容不会因单元格错位崩溃；字段缺失可精准定位到「哪个块缺哪个字段」。
- **块范式 vs XML/JSON/YAML**：`[BEAT ...]` 方括号块在 Markdown 里不渲染成链接/标题，与现有 `<prev_scenes>` 等 XML 上下文标签不混淆；无 JSON 转义负担；不与 Markdown `---` 分隔线冲突。
- **单场景 scope**：一次只写一个场景的节拍，是块范式收益最大化的前提——输出短到几乎不可能崩。

## 5. 最终落盘文件形态

`sequences/S1-1.md`（三段拼装后）：

```markdown
<<<SCENE_BEAT_OUTLINE_START>>>
# S1-1

### 场景表

| 场景ID | 场景功能 | 场景目标(Objective) | 冲突与障碍 | 场景结果(Outcome) | 时空边界 | 视角人物 |
|--------|---------|---------------------|-----------|-------------------|---------|---------|
| SC-S1-1-01 | ... | ... | ... | ... | ... | 林川 |
| SC-S1-1-02 | ... | ... | ... | ... | ... | 林川 |

### 节拍

#### SC-S1-1-01

[BEAT B-SC-S1-1-01-1]
类型: 铺垫
动作: ...
情绪: ...
伏笔: —

[BEAT B-SC-S1-1-01-2]
...

#### SC-S1-1-02

[BEAT B-SC-S1-1-02-1]
...
<<<SCENE_BEAT_OUTLINE_END>>>
```

关键：**外层 `<<<SCENE_BEAT_OUTLINE_START/END>>>` 与 `# <序列ID>` 标题保留不变**——下游四个 writer 通过 `<current_sequence_beats>` 整段注入消费该文件，是纯文本注入不解析结构，故契约不破。段标题由 `### 场景表 / ### 节拍表` 调整为 `### 场景表 / ### 节拍`，节拍段内按场景用 `#### <SC-ID>` 分组。

## 6. 精修与批量的边界

| 调用形态 | target_sequence | 行为 |
|---|---|---|
| 批量 | 留空 | 读 sequence_list，全序列**串行**三段式（一次 tool_call 内 for 循环），产 N 个文件 |
| 精修 | `S1-3` | 只对 S1-3 跑三段式，覆写 `sequences/S1-3.md`，其余不动 |
| （兜底）| 集级/场景级 ID | `normalizeToSequenceId` 归约到序列级后按精修处理 |

## 7. 校验策略（防漏写 + 容错）

- **场景层**（scene_designer）：沿用现有 `checkSceneTable`（列数=7、SC-ID 格式、视角非空），不变。
- **节拍层**（beat_writer，单场景）：新 `checkBeatBlocks`——
  - 至少 1 个块；
  - 每块 B-ID 格式 `B-SC-{幕}-{序}-{nn}-{n}` 且内嵌 SC-ID = 当前 `<target_scene>`；
  - 必需字段齐全（类型/动作/情绪/伏笔）；
  - 类型主词 ∈ `<product_profile>` 词库（允许后缀变体）；
  - 相邻块不同类型（按档案开关，小说关闭）；
  - **容错**：块间空行/注释容忍，仅块内无法识别的行才报错——不因轻微格式偏差就拒收烧 retry。
- **防漏写兜底**：段③循环结束后，引擎断言「已为场景表中每个 SC-ID 都产出了非空节拍块」，缺失则该序列标记部分失败并在事件流报告具体缺哪个场景。

## 8. 数量软校验（非阻塞）

沿用 v6.6：场景数、平均每场景节拍数越出 `<product_profile>` 区间时产 warning（不 retry、不阻断），经 `tool_complete.warnings` 透传 UI。计数从表格行改为块计数。

## 9. 兼容性

- **旧表格格式的 sequences 文件**（v6.6 及之前生成）：作精修 baseline 注入时，模型读旧表格、输出新块格式，**首次精修即转格式**。校验只认新块，baseline 是参考非被校验对象。
- **下游 writer**：零改动（纯文本注入契约不破）。
- **story_checker / UI / foreshadowingState**：零改动（均纯文本消费 sequences 文件）。
- **偏差 ②（短剧资产单位）**：已落地——`chapters/<序列号>.md` 为最小单位（一序列含 8-15 集），集号由 writer 内部 `## 第N集` 自管理；引擎 `normalizeToSequenceId` 把 Orche 误传的集级 ID（如 `S1-1-03`）归约到序列号兜底，`resolveWriteTarget`/`runPipeline` 落盘均用归约 ID。本设计文档仅声明此契约，代码侧已完成。

## 10. 风险

| 风险 | 缓解 |
|---|---|
| 逐场景调用使单序列 LLM 次数上升（1 场景表 + N 节拍）| 单次输出极短、几乎不 retry，净耗时可控 |
| 串行批量墙钟时间较长（20+ 序列 × (1 场景表 + 8-15 场景节拍) 次 LLM）| 属已知代价；偏差 ③（并行）本次不做，未来加并发池可大幅压缩墙钟时间 |
| 批量下部分序列失败、部分成功 | 单序列失败不拖垮整批；ToolResult 汇总成败清单，失败序列可单独精修重跑 |
| 段③某场景节拍反复失败 | 该场景节拍块降级为占位注释 `<!-- 待补节拍 SC-xxx -->` 不阻断整序列落盘，事件流报告 |
