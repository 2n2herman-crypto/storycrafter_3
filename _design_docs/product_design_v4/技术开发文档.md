# StoryCrafter 3 v4 — 技术开发文档

> **版本**：4.0（Tool 调度 + Function Calling 架构）  
> **日期**：2026-06-27  
> **核心变更**：从 Phase 驱动 → Tool 调度，从意图提纯 → Function Calling，从 5 态状态 → 3 态  
> **核心原则**：Orchestrator 只做调度不写内容，所有 Tool 从 ToolRegistry 声明式读取

---

## 一、项目全景

### 1.1 产品定位

StoryCrafter 3 是一个基于多 Tool 协作的**故事创作辅助工具**。用户通过与 AI 系统对话，逐步完成从世界观设定 → 宏观大纲 → 场景节拍精铸的完整创作流程。

**交互模型**：用户输入 → Orchestrator LLM（Function Calling）→ 选择并调用 Tool → Tool 执行创作 → 继续或回复

### 1.2 工作流（v4）

v4 取消了阶段概念。所有流程通过 Tool 的 `dependsOn` 隐式表达：

```
用户输入 "写一个太空歌剧"
  → Orchestrator 调用 generate_worldbuilding（无依赖，始终可用）
  → Tool 产出 worldbuilding.md

用户输入 "再生成几个角色"
  → Orchestrator 调用 generate_characters（dependsOn: worldbuilding.md → 已满足）
  → Tool 产出 characters.md
  ...以此类推
```

Tool 的执行顺序不是预定义的，而是由 Orchestrator LLM 根据用户意图动态决定。

### 1.3 核心约束

1. **Orchestrator 绝不自己编故事内容** — 只做调度和工具决策
2. **Tool 输出必须是纯 MD 表格 + `<<<TAG>>>` 包裹** — 不用 JSON/YAML，不输出前言
3. **Tool 上下文隔离** — 每个 Tool 只能看到其 `reads` 声明的文件
4. **校验规则从 ToolSpec.outputTags 动态读取** — 不硬编码 TAG 列表
5. **Processing 状态下不排队** — 新消息直接丢弃
6. **所有 Tool 信息从 ToolRegistry 声明式读取** — 加新 Tool 不改调度引擎
7. **上游修改提示级联** — 修改上游文件后，Orchestrator 提示用户下游可能需同步

### 2.2 核心流程对比（v3 → v4）

| 维度 | v3 | v4 |
|------|-----|-----|
| **调度驱动** | Phase 推进 + 按钮确认 | Function Calling 自动调度 |
| **Agent 封装** | AgentSpec + Workflow DAG | ToolSpec + dependsOn |
| **意图理解** | IntentPurifier 二次 LLM 调用 | FC 原生 tool_choice='auto' |
| **上下文管理** | 一次性注入，无窗口限制 | 5 轮循环 + CONTEXT_LIMIT 摘要压缩 |
| **状态管理** | 5 态（加 approved/locked） | 3 态（pending/generated/modified） |
| **回退** | StateMachine 检测关键词 → rollback | reset_all Tool（writes+outputTags 空 = 清空） |
| **依赖链** | 阶段锁定（推进后上游不可改） | 提示级联（不锁定，但提示用户） |

### 2.3 关键架构决策

| 决策 | 选择 | 理由 |
|------|------|------|
| **调度模式** | Function Calling | LLM 原生工具选择能力，无需自定义意图分类 |
| **Tool 信息管理** | ToolRegistry 声明式数组 | 加 Tool = 加一条记录 |
| **编排方式** | dependsOn 隐式依赖 | 无需显式 Workflow，由 LLM 动态决策顺序 |
| **重置机制** | reset_all Tool（协议约定） | 不引入特殊 case 代码，writes+outputTags 空=清空 |
| **上下文窗口** | CONTEXT_LIMIT ≈ 22K + 摘要压缩 | 防止多轮 FC 循环溢出 token 限制 |
| **依赖链处理** | 提示级联（非自动重写） | 不替用户做创作决策，但让用户知情 |
| **状态持久化** | Web: 内存 Map / Electron: 文件系统 | 先验证逻辑，后加文件 IO |
| **LLM** | DeepSeek API（OpenAI 兼容） | 支持 FC，浏览器 CORS |

## 四、核心模块详细设计

### 4.0 ToolRegistry（取代 v3 的 AgentRegistry）

所有 Tool 的身份信息集中声明，调度引擎全通用化。

```typescript
// src/orchestrator/toolRegistry.ts

// ── Tool 身份卡 ──
interface ToolSpec {
  id: string                    // 唯一标识，FC function.name
  name: string                  // 人类可读名
  description: string           // FC 描述（最关键，LLM 据此判断调用）
  systemPromptFile: string      // 提示词路径
  reads: string[]               // 上下文隔离边界
  writes: string[]              // 写入文件
  outputTags: string[]          // [START, END] 对
  group: string                 // 前端分组
  dependsOn?: string[]          // 前置文件依赖
}

// ── 核心工具函数 ──
function getTool(id: string): ToolSpec | undefined
function getAvailableTools(existingFiles: Set<string>): ToolSpec[]
function buildFunctionSpec(tool: ToolSpec): object  // → FC 参数

// ── FC 映射规则 ──
ToolSpec.id          → function.name
ToolSpec.description → function.description
ToolSpec.id 的 instruction 参数 → function.parameters
```

ToolRegistry 提供 14 个 Tool，分 4 组：

| 分组 | Tool 列表 | 特点 |
|------|-----------|------|
| **基础设定** | generate_worldbuilding, generate_characters, generate_plot_synopsis + 对应 3 个 refine | 最上游，dependsOn 链起点 |
| **大纲结构** | generate_act_map, generate_sequence_list + 对应 2 个 refine | 中游，依赖上游全部 |
| **微观精铸** | generate_scene_beats + refine_scene_beats | 最下游，依赖全部上游 |
| **系统** | reset_all | writes+outputTags 空 = 清空协议 |

### 4.1 OrchestratorEngine（取代 v3 的 agentDispatcher + stateMachine + intentPurifier + workflows）

这是 v4 的核心调度循环：

```
processUserInput(userInput)
  │
  ├── ① 计算可用工具（getAvailableTools）
  │     读取 FileManager → 获取已有文件集合 → 按 dependsOn 过滤
  │
  ├── ② 加载 System Prompt（注入 {available_tools_json}）
  │     字符串替换：将可用工具的 FC 定义 JSON 注入提示词
  │
  ├── ③ 进入 FC 调度循环（最多 5 轮）
  │     │
  │     ├── 调 LLM（messages, tools=可用工具, tool_choice='auto'）
  │     │
  │     ├── 检查 finish_reason
  │     │   ├── 'tool_calls' →
  │     │   │     串行执行每个 Tool（最多 3 个/轮）
  │     │   │      ① 解析 FC 参数（instruction）
  │     │   │      ② 查 ToolRegistry → 获取 ToolSpec
  │     │   │      ③ executeTool → 读 reads → 组上下文 → 调 Subagent → 校验 → 写入
  │     │   │      ④ 将结果追加到 messages 历史
  │     │   │     ⑤ 重算可用工具（新文件生成后更多 Tool 可用）
  │     │   │     ⑥ 继续循环
  │     │   │
  │     │   ├── 'stop' → 返回 LLM 文本 → 结束
  │     │   ├── 'length' → 返回截断警告 → 结束
  │     │   └── 'content_filter' → 提示调整 → 结束
  │     │
  │     └── 超 5 轮 → 强制结束
  │
  └── ④ 返回 DispatchResult { success, results[], response }
```

**上下文窗口管理**：

```
当前 context 在 messages[] 中不断增长（每轮最多 +6 条消息）。
在每轮开始前检查估算 token 数（字符数 / 2）。

if (estimateTokens(messages) > CONTEXT_LIMIT_CHARS) {
  // CONTEXT_LIMIT_CHARS ≈ 22_000（32K 窗口的 ~70%）
  压缩策略：
  ① 保留 System Prompt
  ② 保留最近 2 轮完整 tool_call / tool_result
  ③ 更早的轮次替换为摘要："[此前已完成工具调用：世界观生成、角色生成]"
}
```

**Tool 执行（executeTool）**：

```
executeTool(tool, instruction)
  │
  ├── (特例) reset_all 协议
  │     if tool.writes.length === 0 && tool.outputTags.length === 0
  │     → fileManager.clearAll() → 直接返回
  │
  ├── ① 读 reads 文件（逐个 tryRead，不存在则用空串）
  ├── ② contextAssembler.assembleContext → XML 格式
  ├── ③ loadSystemPrompt(tool.systemPromptFile) → 加载提示词
  ├── ④ buildAgentPrompt(context, instruction) → 组装 user content
  │
  ├── ⑤ LLM 调用 + 重试（最多 3 次）
  │     for attempt = 0; attempt < 3; attempt++:
  │       output = llm.sendMessage(systemPrompt, userContent)
  │       result = outputValidator.validateOutput(output, tool)
  │       if result.valid → 提取内容 → 写入 writes → 返回 success
  │       if 失败 → 追加格式提示 → 重试
  │
  └── 返回 ToolResult { success, writes[], error?, output? }
```

### 4.2 上下文组装（contextAssembler.ts）

```typescript
function assembleContext(tool: ToolSpec, files: Record<string, string>): string {
  // worldbuilding.md → <worldbuilding>内容</worldbuilding>
  return tool.reads
    .map(filePath => {
      const tagName = filePath.replace(/\.md$/, '')
      const content = files[filePath]
      if (content) return `<${tagName}>\n${content}\n</${tagName}>`
      return `<${tagName}></${tagName}>`  // 空文件也出空标签
    })
    .join('\n\n')
}

// 组装后格式：
// <worldbuilding>
// ## 世界设定
// ...
// </worldbuilding>
// 
// <user_revision_instruction>
// 用户的具体修改指令
// </user_revision_instruction>
```

### 4.3 输出校验（outputValidator.ts）

v4 简化为单 TAG 校验（每个 Tool 只输出一个 TAG 对）：

```typescript
function validateOutput(output: string, tool: ToolSpec): ValidationResult {
  const [startTag, endTag] = tool.outputTags

  // 无 TAG 的 Tool（reset_all）跳过校验
  if (!startTag || !endTag) return { valid: true, missingTags: [], extracted: {} }

  const missingTags: string[] = []
  if (!output.includes(startTag)) missingTags.push(startTag)
  if (!output.includes(endTag)) missingTags.push(endTag)

  const extracted: Record<string, string> = {}
  if (missingTags.length === 0) {
    const content = extractBetween(output, startTag, endTag)
    if (content !== null) extracted[tool.writes[0]] = content
  }

  return { valid: missingTags.length === 0, missingTags, extracted }
}
```

## 七、依赖链与上下文管理

### 7.1 文件依赖图

```
worldbuilding.md ←── characters.md ←── plot_synopsis.md
                    ↓                    ↓
               act_map.md ←─────── sequence_list.md
                                      ↓
                                 scene_beat_outline.md
```

箭头方向表示"消费关系"：上游文件在下游的 `reads` 中。

### 7.2 依赖链失效处理

当用户通过 `refine_*` 修改上游文件时，Orchestrator 在回复中提示用户下游可能需同步更新。**不自动重写**。

实现方式：Orchestrator System Prompt 中有一条规则——

```
## 工具调度规则
...
7. 当你修改了上游文件（如世界观），请在回复中提示用户下游文件（如角色、幕结构）
   可能需要同步更新。让用户决定是否更新，不要自动重写下游。
```

### 7.3 上下文窗口管理

| 阶段 | 策略 | 触发条件 |
|------|------|----------|
| Level 1 | 文件全量注入 reads 中的文件 | 默认（文件 < 22K chars） |
| Level 2 | 按 instruction 关键词提取相关段落 | 文件 > 4000 chars |
| 历史压缩 | 保留最新 2 轮，摘要更早轮次 | messages > CONTEXT_LIMIT_CHARS |

