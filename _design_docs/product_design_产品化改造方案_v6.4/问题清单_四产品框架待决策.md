# 问题清单 · 四产品框架的现状、缺口与待决策

> **本文性质**：诊断 + 待决策清单，不是设计定稿。
> 用于在继续修订 v6.4.1 / v6.5 PRD 之前，盘清「已实现的代码基线」「尚未实现的 PRD」「未描述清楚的边界」三者的关系。
>
> **重要前提校正**：
> - v5.3 / v6.1 的**四层框架、scene_beats 流水线、script_writer** 是**已落地的代码基线**。
> - v6.4 / v6.4.1 / v6.5 **只是 spec/PRD 文档，尚未开发**——其中引用的 `useModeStore`、`ModeParameters`、variant skill、并行批处理等**都还不存在**，属于「待实现设想」而非「已有实现的 bug」。
> - 本文讨论的目标不再是「短剧 vs 中长剧」两模式，而是**四个产品方向**：**小说 / 剧本 / 长剧脚本 / 短剧脚本**。

---

## 零、核心结论（一句话）

现有 PRD 把**短剧的叙事单位映射**讲得最细，却从未定义**其余三个产品**（小说、剧本、长剧脚本）在同一四层骨架下各层的语义与量级；而代码基线是**隐式按一套中长剧参数写死的**（幕数、场景数 3-6、节拍 12-30、字数 2000-4000、单目标、无模式概念）。要支持四产品，缺的不是"再加一个短剧分支"，而是**一套「骨架共享、层语义可变、量级参数化、成文层特化」的统一框架定义**——这套定义目前一处都还没写。

---

## 一、已实现的代码基线（作为参照系）

> 这些是**真实存在的代码**，也是隐含的"默认中长剧"基线。

| 层 | 载体 | 写死的参数 | 是否有模式/产品概念 |
|----|------|-----------|------|
| Orchestrator | `orchestratorEngine.ts` + `orchestrator_v5.md` | 单 target、`TARGET_ID_REGEX` 仅单目标、强制"一次一序列" | ❌ 无 |
| act_map | `act_map/SKILL.md` | 3-12 幕、10 列、占比 25/50/25 | ❌ 无 |
| sequence_list | `sequence_list/SKILL.md` | 每幕 2-N 序列、11 列、`S{幕}-{序}` | ❌ 无 |
| scene_beats | `scene_designer` + `beat_writer` + 代码拼装 | 场景 **N∈[3,6]**、每场 **3-7 拍**、总 **12-30 拍**、类型词库 5 类硬校验 | ❌ 无 |
| script_writer | `script_writer/SKILL.md` | 单章 **2000-4000 字**、单 target、**拒接批量** | ❌ 无 |

关键事实：
- 全代码库**无** `mode` / `episode` / `集` / `产品` 概念。
- validator（列数、SC-ID/B-ID 正则、类型词库）**硬编码**，任何量级/语义偏移都会撞校验。
- `CLAUDE.md` 把 scene_beats 记为"四步 Pipeline（含 sequence_decompose）"，但实际代码是 **2 步 LLM + 1 步代码拼装**，`sequence_decompose` 不存在——**文档表述与代码不符**（PRD 引用时需注意）。

## 二、四产品的结构性主矛盾（用户确认：三个都是主矛盾）

### 主矛盾 A — 成文输出层必须按产品特化，而非一个 script_writer 通吃
- 小说：需要心理描写、叙述声音、人称视角、场景可"慢"。
- 剧本 / 长剧脚本 / 短剧脚本：需要视听化、禁大段内心独白、对白驱动。
- **现有 `script_writer` 的"视听转化 + 禁心理描写"规则对小说是反的**。
- **拟定方向（用户）**：选定产品模式后，成文层**只暴露该产品方向的特化 Subagent**（如 `novel_writer` / `screenplay_writer` / `long_drama_writer` / `short_drama_writer`），彼此互斥可选。
- **待决策**：这些特化 Subagent 是并列新建，还是在 `script_writer` 下按产品挂多 Skill？（涉及矛盾 §四的路线之争）

### 主矛盾 B — 叙事单位映射混乱，需靠"可输出量级参数化"约束
- 小说 / 剧本：无"集"概念；长剧：一序列 = 一集；短剧：一序列 = 一幕 = 多集。
- 序列↔集↔场景↔资产文件的映射，在四产品里各不相同，目前无统一契约。
- **拟定方向（用户）**：用**参数化定义各层可输出的量级**来约束，例如：
  - 短剧：**每幕只能有 1 个序列**（强约束）；序列内场景/节拍量级参数化设定。
  - 其余产品：幕/序列/场景/节拍的数量区间各自参数化。
- **待决策**：这套参数表（每层数量的上下限、约束关系）需要为四产品各出一列——目前仅短剧有零散描述。

### 主矛盾 C — 层语义可变（骨架共享，语义按产品变）
- 用户确认：**四层骨架（幕-序-场-节）共享，但每层语义可随产品变化**。
- 例：短剧的"幕"= 全剧大阶段；长剧的"幕"= 传统戏剧幕；小说的"场景"语义 ≠ 剧本的"场景"。
- **待决策**：每一层在四产品下的语义定义表（这是补齐"中长剧结构没描述清楚"的核心动作）。

## 三、各层未描述清楚的边界（逐层，四产品视角）

> 用户提醒：**以字数为边界太粗糙**；序列由多个场景聚合，创作层问题（信息披露、伏笔、情绪曲线、戏剧问题归属）都要一并考虑。以下边界都应以**叙事/创作语义**为主、字数为辅。

### 幕层（act_map）
- 现有只有"3-12 幕 + 占比"，**未定义各产品幕的语义与集数/序列跨度**。
- 待定义：小说幕=？剧本幕=？长剧幕含几集/几序列？短剧幕=全剧大阶段且**恰含 1 序列**。

### 序列层（sequence_list）
- 现有"序列=一件事、回答一个戏剧问题"。
- **冲突点**：长剧"一序列=一集"，但一集常含多线多戏剧问题，与"只讲一件事"语义打架。
- 待定义：序列在四产品里对齐"一件事"还是对齐"一集/一幕"？每幕序列数区间（短剧=1，其余=?）。

### 场景层（scene_designer）
- 现有 **N∈[3,6]**，但 `短剧逻辑.md` 自陈中长剧"一集 5-10 场景"——**现有上限对中长剧就偏低**（既存局限，非短剧引入）。
- 待定义：四产品各自的"每序列场景数"参数；小说的"场景"是否等同剧本场景。
- 创作层：场景聚合成序列的逻辑（因果闭环 Outcome→Objective）在各产品是否一致。

### 节拍层（beat_writer）
- 现有每场 3-7 拍 / 总 12-30，类型词库 5 类硬校验。
- 待定义：短剧要 4 拍微循环 + 钩子/摩擦/尖峰/钉词库；中长剧一集若 5-10 场 ×6-10 拍 → 30-100 拍，**超现有上限**。
- 创作层：节拍类型词库按产品可扩展，validator 需按产品切换校验集合（机制未设计）。

### 成文层（writer）
- 现有单章 2000-4000 字、视听化、禁心理描写、单目标、拒批量。
- 待定义：四产品各自的成文规范（小说反着来）、单位产出粒度（一集？一幕？一章节？）、是否分段多次调用产整集。
- **字数不应作为主边界**——应以"覆盖某序列/某集的完整叙事单元"为界，字数是副产物。

## 四、实现路线之争（用户否决了前两版，需重议）

已被否决的两条路线：
1. ❌ **variant 热插拔**（每 Skill 挂 `_short` 副本）——文件翻倍、维护成本高。
2. ❌ **纯参数注入**（同一 SKILL.md + Engine 注数值）——被否，可能因为无法承载"层语义可变 + 成文层特化"的深度差异。

用户给出的新线索（需据此重构路线）：
- 成文层：**按产品选特化 Subagent**（模式选定后互斥可选）→ 倾向"多 Subagent 特化"而非"一个 Skill 参数化"。
- 中上游层（幕/序/场/节）：**量级参数化 + 层语义可变** → 倾向"同骨架 + 参数 + 语义槽"。
- **待决策**：是否采用**分层混合路线**——上游层走参数化/语义槽，成文层走产品特化 Subagent？这是当前最需要拍板的架构问题。

## 五、v6.5 并行批处理遗留的一致性边界（待验证是否仍要）

- 并行写多章时，`behaviorTrack`（角色行为追踪）串行依赖被打破 → 角色一致性失控。
- 伏笔 plant/payoff 顺序在并行下无法协调 → 可能重复 plant / 提前 payoff。
- `behaviorTrack` Map 并发写存在竞态。
- 判定串/并行依赖尚不存在的 `modeStore`。
- **待决策**：四产品重构后，并行批处理是否仍是 v6.5 目标？还是先聚焦四产品框架、并行降级为后续优化？
