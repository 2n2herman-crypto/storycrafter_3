## 一、toolRegistry.ts — 重大重写

### 变更 1：替换 12 个 gen/refine ToolSpec 为 6 个合并 ToolSpec

**v4（12 个规格）：**
```
generate_worldbuilding  +  refine_worldbuilding
generate_characters     +  refine_characters
generate_plot_synopsis  +  refine_plot_synopsis
generate_act_map        +  refine_act_map
generate_sequence_list  +  refine_sequence_list
generate_scene_beats    +  refine_scene_beats
```

**v5（6 个合并规格）：**
```
worldbuilding           （自身在 reads 中）
characters              （自身在 reads 中）
act_map                 （自身在 reads 中，吸收 plot_synopsis 职能）
sequence_list           （自身在 reads 中）
scene_beats             （自身在 reads 中）
```

**合并模式原理**：
- `reads` 数组中包含自身输出的文件（如 worldbuilding 的 reads 包含 `worldbuilding.md`）
- 自身文件不存在时 → 引擎注入空内容 → LLM 自动创建
- 自身文件存在时 → 引擎注入完整内容 → LLM 自动修改

```typescript

  reads: [],
  writes: ['worldbuilding.md'],
  dependsOn: [],
}
{
  id: 'refine_worldbuilding',
  reads: ['worldbuilding.md'],
  writes: ['worldbuilding.md'],
  dependsOn: ['worldbuilding.md'],
}

// v5 合并模式
{
  id: 'worldbuilding',
  reads: ['worldbuilding.md'],   // 自身在 reads 中
  writes: ['worldbuilding.md'],
  // 不再需要 dependsOn 做硬闸门
}
```

### 变更 3：getAvailableTools() — 全部放行（核心变化）

```typescript
// v4：硬闸门过滤——只有上游文件存在时工具才可见
function getAvailableTools(existingFiles: Set<string>): ToolSpec[] {
  return TOOL_REGISTRY.filter(tool => {
    return tool.dependsOn.every(dep => existingFiles.has(dep))
  })
}

// v5：全部工具始终可见——结构感由 prompt 引导，不走代码拦截
function getAvailableTools(): ToolSpec[] {
  return TOOL_REGISTRY
}
```

**为什么这样做**：
1. 用户可能跳过上游先写下游内容（"先写第3幕的关键场景"）
2. dependsOn 硬拦截会阻止这种灵活的工作流
3. 结构感（幕→序列→场景→节拍）由每个 Tool 的 prompt 定义其层级位置和输出模板，不需要代码拦截
4. Tool 执行时 reads 中的缺失文件自动变为空标签，LLM 自然理解"没有上游参照"

**dependsOn 字段保留**，但仅作为 prompt 层面的"推荐顺序"元信息，不再做代码拦截。

### 变更 4：getAllAssetPaths() 更新

```typescript
// v4 返回 6 个路径（硬编码）
const DEFAULT_ASSET_PATHS = [
  'worldbuilding.md', 'characters.md', 'plot_synopsis.md',
  'act_map.md', 'sequence_list.md', 'scene_beat_outline.md',
]

// v5 从 TOOL_REGISTRY 自动聚合（推荐做法）
// 遍历所有 tool.writes，去重后返回
// 自动包含：worldbuilding.md, characters.md, act_map.md,
//          sequence_list.md, scene_beat_outline.md,
//          foreshadowing.md, subplots.md,
//          draft_history.md, _check_report.md
export function getAllAssetPaths(): string[] {
  const paths = new Set<string>()
  for (const tool of TOOL_REGISTRY) {
    for (const w of tool.writes) {
      paths.add(w)
    }
  }
  return Array.from(paths)
}
```

> `draft_history.md` 不由任何 Tool 直接写入。如果有引擎级写入需求（如审计轮次记录），可在 engine 代码中写入。当前计划中列为预留路径，可以暂不加入资产列表。

### 变更 2：审计修改闭环（新增）

在 FC 循环框架内。由 orchestrator LLM 看到 `_check_report.md` 内容后自主选择修复工具。

```
FC 第 N 轮 → orchestrator 调用 story_checker
  → tool_result 返回完整报告内容（非简短消息）
  → orchestrator 看到报告中的 FAIL 项
  → orchestrator 自主决定修复层级：
    ├── 序列层问题 → 下一轮 FC 调 sequence_list（修改模式）
    ├── 场景层问题 → 下一轮 FC 调 scene_beats（修改模式）
    └── 支线层问题 → 下一轮 FC 调 subplot_manager（合并模式）
  → 引擎计数审计轮次，≥3 轮强制结束
```

**引擎层面改动**：

```typescript
// ① SchedulerState 增加审计计数器
interface SchedulerState {
  currentRound: number
  maxRounds: number
  toolsCalled: string[]
  toolResults: ToolResult[]
  auditRound: number       // ← 新增
  maxAuditRounds: number   // ← 新增，恒为 3
}

// ② story_checker 的 tool result 注入完整报告
if (toolSpec.id === 'story_checker') {
  const report = result.success
    ? await this.fileManager.readFile('_check_report.md')
    : `检查失败: ${result.error}`
  messages.push({
    role: 'tool',
    tool_call_id: toolCall.id,
    content: report,
  })
  state.auditRound++
} else {
  // 原有简短消息逻辑
  messages.push({
    role: 'tool',
    tool_call_id: toolCall.id,
    content: result.success
      ? `已成功执行 ${toolSpec.name}，输出已保存。`
      : `${toolSpec.name} 执行失败: ${result.error || '未知错误'}`,
  })
}

// ③ FC 循环头部超限检查
while (state.currentRound < MAX_ROUNDS) {
  if (state.auditRound >= state.maxAuditRounds) {
    return {
      success: true,
      results: state.toolResults,
      response: '检查和修复已达 3 轮上限，部分问题可能需要你的进一步指导。',
    }
  }
  // ...原有循环逻辑
}
```

### 变更 3：MAX_ROUNDS 策略

```typescript
// v4
const MAX_ROUNDS = 5

// v5（不变）
const MAX_ROUNDS = 5       // FC 调度轮次
// 审计修复独立上限 = 3（放在 SchedulerState 中）
```

FC 循环 5 轮上限不变，审计独立 3 轮上限。二者不混用。
注意：审计修复占用 FC 轮次配额——story_checker 发现问题后，orchestrator 在下一轮 FC 中调 scene_beats 修复，这占 1 轮 FC 配额。

## 三、contextAssembler.ts — 不需要改动

`assembleContext()` 已通用地遍历 tool.reads 构建 XML 上下文。新增的文件路径会在 reads 中声明，自动被组装。

**关键机制**：当文件不存在时，`fileManager.readFile()` 返回空字符串 → `assembleContext` 输出 `<tagName></tagName>`（空标签）。这正好用于"跳过上游"场景——Subagent 看到空标签，知道"没有上游参照，基于用户描述直接生成"。

### Prompt 结构感引导策略

每个 Subagent Prompt 中明确其层级位置。示例（scene_beats.md）：

```
你负责生成场景节拍（幕→序列→场景→节拍四层结构的第3-4层）。

生成原则：
- 如果上游文件存在，参考它们做一致性对齐
- 如果上游文件不存在，直接基于用户描述生成
- 输出必须保持 幕ID → 序列ID → 场景ID → 节拍ID 的编号体系
- 表格列不可省略（15列全量输出）
```

Orchestrator Prompt 中增加推荐顺序引导（orchestrator_v5.md）：

```
工具调用建议顺序（非强制，仅供参考）：
  世界观 → 角色 → 幕结构 → 序列 → 场景节拍 → 故事检查

你可以根据用户需求灵活调整顺序：
- 生成新故事 → 按推荐顺序逐步生成
- 修改特定内容 → 直接调对应工具
- 用户想看中间产出 → 跳过上游，直接调目标工具
```
