# Orchestrator 全程调度规则 (Routing & Dispatch Rules)

Orchestrator 的代码层必须实现以下硬性调度逻辑，不依赖 LLM 进行路由判断。  
**通用原则**：校验规则不硬编码，从 `AgentRegistry` 中读取 `AgentSpec.outputTags` 动态执行。

## 1. 意图提纯规则 (Intent Purification)
*   **规则**：在阶段二及之后，用户的输入往往是口语化的（如”我觉得第二幕节奏太慢了，加点动作戏”）。Orchestrator 必须先调用一个轻量级 LLM 步骤（或内部 Tool），将其转化为结构化的 `<user_revision_instruction>`（如：”修改要求：加快第二幕节奏，在序列S2-1中增加动作对抗场景”）。
*   **目的**：防止下游 Agent 被用户的口语化情绪干扰。

## 2. 防抖与并发控制 (Debouncing)
*   **规则**：当系统正在调用任何 Sub-Agent 时（即处于 `processing` 子状态），若用户在前端继续发送消息，Orchestrator 必须拦截并返回”正在生成/修改中，请稍候...”，**严禁**中断当前调用或将新消息追加到当前 Prompt 中。
*   **实现策略**：输入框 disabled + 显示”正在处理中...”。**不做消息排队**——用户在 Processing 期间输入的消息被丢弃（不是入队）。用户等处理完成后自行重新输入。理由：MVP 为单人单次创作场景，排队带来的复杂度（队列存储、出队时机、顺序保证）与收益不匹配。后续阶段可根据用户反馈重新评估。

## 3. 产物校验与重试机制 (Validation & Retry)
*   **规则**：接收到 Sub-Agent 或 Phase 1 LLM 的输出后，Orchestrator 必须执行校验。**校验规则不硬编码**，而是从 `AgentRegistry` 中对应 Agent 的 `AgentSpec.outputTags` 读取。
*   **具体流程**：
    1. 查出当前调用的 Agent 的 `AgentSpec.outputTags`（如 `['<<<ACT_MAP_START>>>', '<<<SEQUENCE_LIST_START>>>']`）
    2. 对于 Phase 1（Orchestrator 直接调用 LLM），使用预定义的文件级 tag 列表：
       `['<<<WORLDBUILDING_START>>>', '<<<CHARACTERS_START>>>', '<<<PLOT_SYNOPSIS_START>>>']`
    3. 逐项检查输出中是否包含每个 TAG
    4. 检查 TAG 内部是否包含合法的 Markdown 内容
*   **动作**：若校验失败，Orchestrator 自动将原 Prompt 附加”格式错误，请严格遵循XML标识符和Markdown模板重新输出”后重试，最多重试 3 次。

## 4. 状态回退机制 (Rollback)
*   **规则**：如果用户在阶段二或之后要求”推翻核心设定”（如更换主角、改变世界观），Orchestrator 必须识别此意图，将状态机强制回退至 `STATE_1_CONCEPT`。
*   **回退范围**：**动态计算**——清空当前阶段及之后的所有 Agent 产出的 MD 文件（即 `AgentSpec.phase >= 当前阶段` 的所有 Agent 的 `writes` 文件）。
    *   当回退由 `STATE_COMPLETED` 触发时，额外清空 Phase 1 直接产出文件（`worldbuilding.md / characters.md / plot_synopsis.md`），因为这些文件在 COMPLETED 状态下也应重置。
*   **STATE_COMPLETED 的可回退性**：`STATE_COMPLETED` 同样可触发回退。触发方式有两种：
    1. 用户点击前端 **「重新创作」按钮**（COMPLETED 状态下始终可见）
    2. 意图提纯检测到”推翻核心设定”意图
    *   回退前前端弹出二次确认对话框：”这将清空所有已创作内容（含基础设定文件），恢复至概念设定阶段。确定继续？”
    *   用户确认后执行回退流程。
*   **回退清理范围（含 Phase 1 文件）**：
    ```typescript
    // 回退时清空的文件列表
    async function rollback(sincePhase: number) {
      // 1. 清空 Agent 产出文件
      // 当从 COMPLETED (phase=4) 回退时，清空所有 Agent 产出文件
      // 当从 STATE_2/STATE_3 回退时，清空当前阶段及之后的 Agent 产出文件
      const agentFiles = AGENT_REGISTRY
        .filter(a => sincePhase >= 4 || a.phase >= sincePhase)
        .flatMap(a => a.writes)
      // 2. 如果回退起点是 COMPLETED (phase=4)，额外清空 Phase 1 直接产出文件
      if (sincePhase >= 4) {
        agentFiles.push(...PHASE1_DIRECT_WRITES)
      }
      // 3. 清空对应的快照文件
      for (const file of agentFiles) {
        await fileManager.clearFile(file)
        await fileManager.clearSnapshot(file)
      }
    }
    ```
*   **前端联动**：回退时前端必须一并清空对应快照文件（`*.approved.md`），重置资产卡片状态为”待生成”。

## 5. 前端调度联动规则 (Frontend Dispatch Rules)

### 5.1 阶段推进方式（双模设计）

前端提供**两种**阶段切换方式，行为不同：

**方式一：PhaseSelector 下拉切换（查看模式）**
- 用户可以自由选择任意 Phase 查看已生成的卡片内容
- 切换时不创建快照，不锁定文件
- 如果当前选中的 Phase 低于实际确认进度，卡片可见但不允许修改
- 用于回顾历史阶段的资产内容

**方式二：ConfirmButton 确认推进（推进模式）**
- 点击确认按钮 → 创建快照（`*.approved.md`）→ 推进到下一阶段
- 推进后，前一阶段的所有卡片锁定（只读）
- 推进不可逆（除非触发回退）

两种方式的协同规则：
- 如果用户通过 PhaseSelector 回看旧阶段，然后通过 ChatInput 发送修改指令 → 系统自动将 Phase 切回当前最新阶段，再执行调度
- 不允许在非当前阶段发送修改指令
- 下拉选择器始终显示所有 Phase 选项（包括已锁定和未解锁的），但未解锁的 Phase 会在用户选择时提示"请先完成当前阶段的确认"

### 5.2 防抖规则的前端体现

*   防抖由 Orchestrator 控制，前端表现如下：
    *   用户发送创作指令后，对话输入框立即显示”正在处理中...”状态
    *   在 Orchestrator 返回结果前，发送按钮禁用，输入框锁定
    *   用户在 Processing 期间输入的消息**直接丢弃**，不排队
    *   用户需等待当前处理完成后再发送新消息
    *   Orchestrator 返回 `processing` 状态码时，前端显示加载指示器

### 5.3 资产卡片可见性规则

前端根据当前 Phase 控制右侧栏资产卡片的显示与锁定。**卡片列表不硬编码**，从 `AgentRegistry` 动态读取。  
Phase 与数字的映射通过 `PHASE_ORDER` 工具表计算（在 `stateMachine.ts` 中定义，见 技术开发文档.md §4.2）：

```
PHASE_ORDER: { STATE_1_CONCEPT: 1, STATE_2_MACRO: 2, STATE_3_MICRO: 3, STATE_COMPLETED: 4 }
可见范围 = AgentRegistry 中 AgentSpec.phase <= PHASE_ORDER[当前 Phase] 的 writes 文件
锁定范围 = AgentRegistry 中 AgentSpec.phase < PHASE_ORDER[当前 Phase] 的 writes 文件
```

当前 MVP 的映射效果：

| Phase | 可见文件 | 锁定文件 |
|-------|---------|---------|
| STATE_1_CONCEPT | worldbuilding.md, characters.md, plot_synopsis.md | 无 |
| STATE_2_MACRO | 上 + act_map.md, sequence_list.md | 阶段一 |
| STATE_3_MICRO | 全部 6 个 | 阶段一 + 阶段二 |
| STATE_COMPLETED | 全部 6 个 | 全部 |

> 远期加 Agent 后此表自动扩展，不需手动维护。

### 5.4 卡片状态前端映射

每个资产卡片在右侧栏显示状态圆点：

| 后端状态 | 前端圆点颜色 | 标签 |
|---------|------------|------|
| `pending` | 灰色 | 待生成 |
| `generated` | 蓝色 | 已生成 |
| `approved` | 绿色 | 已确认 |
| `modified` | 黄色 | 已修改 |
| `locked` | 灰色 | 已锁定 |

### 5.5 确认按钮的显隐规则

确认按钮的文案和下一阶段目标**从 Workflow 定义中动态读取**：

*   当前阶段不是最后一个阶段 → 显示"确认{当前阶段名}，进入{下一阶段名}"按钮
*   当前阶段是最后一个阶段 → 显示"完成创作"按钮
*   确认按钮在当前阶段必需文件未全部生成时禁用（必需文件 = 当前阶段所有 Agent 的 `writes` 文件）
*   PhaseSelector 下拉选择器不会触发确认/快照，它只改变前端的查看范围。Phase 的实际推进必须通过点击 ConfirmButton 完成。
*   点击 ConfirmButton 后：创建快照 → 推进 phaseStore.currentPhase → 锁定前一阶段卡片 → UI 自动刷新。

当前 MVP 效果：

| 阶段 | 按钮文案 | 目标 |
|------|---------|------|
| STATE_1_CONCEPT | 确认设定，进入大纲构建 | STATE_2_MACRO |
| STATE_2_MACRO | 确认大纲，进入场景细化 | STATE_3_MICRO |
| STATE_3_MICRO | 完成创作 | STATE_COMPLETED |
| STATE_COMPLETED | 已完结（禁用） | — |
