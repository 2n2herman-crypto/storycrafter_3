# StoryCrafter 3 — 设计文档目录与阅读指南

> **目的**：防止 AI 跳过设计文档胡乱开发。  
> **规则**：每次开始一个新 Step 前，强制阅读该 Step 涉及的所有设计文档。  
> **本文档本身也要读**，它是所有文档的总索引。

## 二、强制阅读规则

### 2.1 通用规则（每次开发都生效）

| # | 规则 | 违反后果 |
|---|------|---------|
| 1 | **必须读完本文档**才能开始任何编码 | 不知道文档分布，漏读关键约束 |
| 2 | **必须先读涉及的设计文档，再写一行代码** | 自行发明架构与设计矛盾 |
| 3 | **IPC 接口定义以 design-doc.md 第 9 节为准**，不要自己发明 | 前端与主进程通信不兼容 |
| 4 | **Agent System Prompt 以 subagent1/2_system_prompt.md 为准**，不可自行改写 | Agent 行为失控 |
| 5 | **输出格式校验以全程调度规则.md 第 3 节为准** | Agent 输出无法被下游解析 |
| 6 | **状态机流转以状态机.md 为准**，不可发明新状态 | 阶段逻辑混乱 |

## 三、关键设计文档重点摘要

> 以下摘要是为了让 AI 快速回忆，但**不要替代原文阅读**。每份文档都有精确的细节，摘要可能遗漏。

### 3.1 Agent-Orchestrator_system_prompt.md

**Orchestrator 四大铁律**：
1.  绝不自己编撰故事内容
2.  绝不把原始用户对话直接透传给 Subagent
3.  在 STATE_2/STATE_3 绝不绕过 Subagent 直接修改 MD
4.  绝不理解 MD 表格业务逻辑，只做读写校验拼接

### 3.2 全程的调度规则.md

| 规则 | 要点 |
|------|------|
| 意图提纯（§1） | 阶段二/三必须将口语化输入提纯为结构化指令 |
| 防抖（§2） | Processing 状态下拦截用户新消息，严禁中断当前调用 |
| 校验重试（§3） | Agent A/B 输出正则校验，失败最多重试 3 次 |
| 回退（§4） | 发现"推翻核心设定"意图，强制回退到 STATE_1，清空后续文件 |
| 前端联动（§5） | Phase→卡片可见性/锁定/状态圆点/确认按钮的精确映射 |

### 3.3 状态机.md

```
STATE_1_CONCEPT → STATE_2_MACRO → STATE_3_MICRO → STATE_COMPLETED
   ↑                     ↑               ↑               ↑
  用户确认设定         用户确认大纲     用户完成创作     终点锁定
```

- 回退：STATE_2/STATE_3 直接跳回 STATE_1_CONCEPT
- 快照时机：每次用户点击确认按钮时生成 `*.approved.md`
- 回退时清空：阶段二/三的文件 + 对应的快照

### 3.4 数据接口契约文档.md

**6 个核心资产文件**：

| 阶段 | 文件名 | 生产者 | 消费者 |
|------|--------|--------|--------|
| 一 | worldbuilding.md | Orchestrator | Agent A, B |
| 一 | characters.md | Orchestrator | Agent A, B |
| 一 | plot_synopsis.md | Orchestrator | Agent A, B |
| 二 | act_map.md | Agent A | Agent B |
| 二 | sequence_list.md | Agent A | Agent B |
| 三 | scene_beat_outline.md | Agent B | 最终交付 |

**IPC 接口**：design-doc.md 第 9 节有完整的 `window.storyAPI` 定义。

### 3.5 Agent System Prompt（绝对不可修改）

**Subagent A** 输出模板（`subagent1_system_prompt.md`）：
```

## 幕级地图
| 幕编号 | 功能定位 | 占比 | 情绪目标 | 核心冲突升级 | 必须解决叙事项 |

| 序列ID | 所属幕 | 功能定位 | 必须完成事件 | 可选钩子 |
```

### 3.6 前端卡片状态映射

```
pending(灰) → generated(蓝) → approved(绿) → modified(黄)
                                              → locked(灰+锁)
```

| Phase | 可见卡片 | 锁定卡片 |
|-------|---------|---------|
| STATE_1_CONCEPT | 阶段一 3 个 | 无 |
| STATE_2_MACRO | 阶段一 + 二 5 个 | 阶段一 |
| STATE_3_MICRO | 全部 6 个 | 阶段一 + 二 |
| STATE_COMPLETED | 全部 6 个 | 全部 |

## 四、常见 AI 开发误区

> 以下是从设计文档中提取的**最容易踩的坑**，AI 开发时必须警惕。

| 误区 | 正确的做法 | 出处 |
|------|-----------|------|
| 自己写了一份新的 Agent Prompt | 严格从 `subagent1/2_system_prompt.md` 复制 | Agent 设计文件 |
| 输出格式用了 JSON 而非 MD 表格 | Agent 输出必须是纯 Markdown 表格，用 `<<<TAG>>>` 包裹 | subagent1_system_prompt.md |
| 直接在代码里改了状态机流转规则 | 严格遵从 `状态机.md` 的单向流转 + 回退规则 | 状态机.md |
| 自己发明了 IPC 通道名 | IPC 通道名必须与 `数据接口契约文档.md` 第 3.2 节一致 | 数据接口契约文档.md |
| 在 Renderer 进程直接 `fs.readFile` | 必须通过 Preload 的 `window.storyAPI.*` 调用 | design-doc.md §3 |
| 自行设计 diff 策略（如行级 diff） | 使用 `diff.diffWords` 词级 diff，文本层注入 `<ins>/<del>` | design-doc.md §4 |
| 在 State 1/2 绕过 Subagent 直接写了 MD 内容 | Orchestrator 只做调度，不能自己生成故事内容 | Agent-Orchestrator_system_prompt.md |
| 修改了 State 1 已确认的 MD 文件 | 已确认 = 锁定，只能由用户主动触发重新生成 | agent任务清单.md |
| 没有做防抖导致并发请求 | Processing 状态下必须拦截新请求 | 全程的调度规则.md §2 |
| 只重试 1 次 | Agent 输出校验失败必须重试最多 3 次 | 全程的调度规则.md §3 |

## 五、如何在开发中使用本文档

### 5.1 开发前

```
1. 打开 开发规划.md → 确认当前在哪个 Step
2. 查看上表 2.2 → 阅读该 Step 涉及的所有设计文档
3. 阅读本节 3.x → 确认该 Step 涉及的关键约束
4. 阅读本节 4 → 避开常见误区
5. 开始编码
```

### 5.2 遇到不确定的问题时

```
1. 搜索本节 3.x 的关键摘要
2. 回到原始设计文档阅读完整上下文
3. 如设计文档无明确规定 → 遵循最小改动原则，不发明新功能
```

### 5.3 提交/合并前

```
1. 检查本节 4 的常见误区，确认没有踩坑
2. 确认代码实现与设计文档的接口定义一致
3. 确认输出格式与 Prompt 模板一致
4. 确认状态机逻辑与设计一致
```
