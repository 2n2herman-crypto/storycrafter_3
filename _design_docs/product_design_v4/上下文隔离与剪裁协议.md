# 上下文隔离与剪裁协议

> **核心原则**：每个 Tool 只能看到其 `reads` 声明的文件，Orchestrator 负责将用户意图拆解并精准传递指令。

---

## 一、设计目标

1. **上下文隔离（Context Isolation）** — 每个 Tool 执行时，只读取其 `reads` 列表中的文件。不同 Tool 不能访问对方的"私有"文件。
2. **指令剪裁（Instruction Trimming）** — Orchestrator 将用户复杂需求拆解为每个 Tool 的具体指令，去掉无关内容。
3. **渐进式扩展** — 从文件级隔离（Level 1）起步，设计层面预留内容级剪裁（Level 2）和渐进式上下文（Level 3）的扩展点。

---

## 二、上下文隔离原则

### 2.1 隔离边界

```
ToolRegistry:
  generate_act_map:
    reads: [worldbuilding.md, characters.md, plot_synopsis.md]
    writes: [act_map.md]

  generate_scene_beats:
    reads: [worldbuilding.md, characters.md, plot_synopsis.md, act_map.md, sequence_list.md]
    writes: [scene_beat_outline.md]
```

- `generate_act_map` → 只能读取 worldbuilding.md、characters.md、plot_synopsis.md
- `generate_scene_beats` → 可以读取 act_map.md（因为在其 reads 中）
- `generate_act_map` → 不能读取 scene_beat_outline.md（不在其 reads 中）

### 2.2 文件写入隔离

- Tool 只能写回其 `writes` 声明的文件
- 不在 `writes` 列表中的文件不可被该 Tool 修改
- 多个 Tool 可以声明相同的 `writes`（如 `refine_worldbuilding` 和 `generate_worldbuilding` 都写 `worldbuilding.md`）— 由调度顺序决定最终内容

### 2.3 依赖检查

在 Tool 执行前，检查其 `dependsOn` 文件是否存在：

```typescript
function checkToolAvailability(tool: ToolSpec, existingFiles: Set<string>): boolean {
  if (!tool.dependsOn || tool.dependsOn.length === 0) return true
  return tool.dependsOn.every(f => existingFiles.has(f))
}
```

如果依赖文件不满足，Tool 不注册到 Function Calling 的工具列表中，LLM 不会看到它。

## 三、上下文组装格式

### 3.1 基本格式

```xml
<worldbuilding>
[worldbuilding.md 的完整内容]
</worldbuilding>

<characters>
[characters.md 的完整内容]
</characters>

<user_revision_instruction>
[裁剪后的指令]
</user_revision_instruction>
```

### 3.2 标签命名规则

- 文件名 `xxx.md` → 标签名 `xxx`（去掉 `.md` 后缀）
- 标签使用 `<xxx>` 和 `</xxx>` 包裹
- 如果文件不存在或为空：输出 `<xxx></xxx>` 空标签
- 指令使用 `<user_revision_instruction>` 包裹

### 3.3 格式示例

```
<worldbuilding>
## 世界名称
艾泽拉斯

## 物理法则
魔法与科技并存...
</worldbuilding>

<characters>
## 主角
姓名：阿尔萨斯
...
</characters>

<user_revision_instruction>
完善第二幕的冲突升级，增加阿尔萨斯和乌瑟尔的对抗戏份
</user_revision_instruction>
```

## 四、剪裁策略分档

### Level 1：文件级隔离（默认）

```
输入：ToolSpec.reads = [worldbuilding.md, characters.md]
输出：<worldbuilding>全量内容</worldbuilding>
      <characters>全量内容</characters>
```

- 只读取 `reads` 列表中的文件
- 每个文件**全量注入**
- 不在这列表中的文件对 Tool 完全不可见

### Level 2：内容级剪裁（进阶）

```
输入：ToolSpec.reads = [worldbuilding.md]
      instruction = "完善第二幕的冲突升级"

输出：<worldbuilding>
        [只提取与"第二幕"、"冲突"相关的段落]
      </worldbuilding>
```

- 对大型文件（> N 行/字符），提取与 `instruction` 相关的段落
- 提取策略：
  - **标题匹配**：找到指令中提到的章节标题
  - **关键词匹配**：提取包含指令关键词的段落
  - **段落边界**：以 `##` 或 `###` 标题为段落边界做切割

### Level 3：渐进式上下文（高阶）

```
直接相关文件（文件名匹配 instruction 关键词） → 全量注入
间接相关文件（reads 中有但非直接相关）       → 摘要注入（前 N 行/自动摘要）
不相关文件                                  → 不注入
```

- 根据 `instruction` 内容动态判断哪些文件"直接相关"
- 直接相关文件全量注入
- 非直接但仍在 `reads` 中的文件，注入摘要（如文件的前 30 行，或通过 LLM 自动摘要）

## 五、指令传递机制

### 5.1 指令裁剪

Orchestrator 在调用 Tool 之前，将用户输入裁剪为该 Tool 相关的部分：

```
用户输入:
"帮我完善世界观设定，把主角改成女性，然后生成新的幕结构"

Tool 1 (refine_worldbuilding) 收到的 instruction:
"完善世界观设定"

Tool 2 (refine_characters) 收到的 instruction:
"把主角改成女性"

Tool 3 (generate_act_map) 收到的 instruction:
"根据更新后的设定生成新的幕结构"
```

### 5.2 MVP 简化方案

MVP 阶段，指令裁剪可以简化为将完整用户输入作为 `instruction` 传给每个 Tool，让 Tool 自行忽略无关部分。更精准的裁剪是后续优化项。

---

## 六、上下文大小管理

由于 Token 窗口有限（`max_tokens=8192`），需要管理上下文大小：

| 策略 | 触发条件 | 做法 |
|------|----------|------|
| 跳过空文件 | 文件不存在或无内容 | 输出空标签 `<xxx></xxx>` |
| 文件截断 | 文件 > 2000 字符 | Level 1 全量保留，LLM 自行处理 |
| 内容裁剪 | 文件 > 4000 字符 | Level 2 启用，只提取相关段落 |
| 渐进摘要 | reads 中文件数 > 3 | Level 3 启用，非核心文件摘要化 |

---

## 七、依赖链失效传播机制

> **问题**：v4 没有阶段锁定，用户可以在任何时候修改任何文件。但如果修改了上游文件（如世界观），下游依赖它的文件（如角色、幕结构）可能变得不一致。

### 7.1 文件依赖图

```
worldbuilding.md ─────────────────────────────────────────────────┐
    │                              characters.md ────┐            │
    │                                  │             │            │
    │                                  ▼             ▼            ▼
    └────────────────────────→ plot_synopsis.md ──→ generate_act_map.md
                                                          │
                                                          ▼
                                                  sequence_list.md
                                                          │
                                                          ▼
                                                  scene_beat_outline.md
```

每个箭头表示"消费关系"：上游文件是下游文件的 `reads` 依赖。

### 7.2 级联影响判定

当用户调用 `refine_*` 修改某个文件时，系统需要判定哪些下游文件可能受到影响：

| 被修改的文件 | 可能受影响的下游 | 影响程度 |
|-------------|-----------------|---------|
| worldbuilding.md | characters.md, plot_synopsis.md, act_map.md, sequence_list.md, scene_beat_outline.md | 🔴 **全部下游** |
| characters.md | plot_synopsis.md, act_map.md, sequence_list.md, scene_beat_outline.md | 🟠 大部分 |
| plot_synopsis.md | act_map.md, sequence_list.md, scene_beat_outline.md | 🟠 中 |
| act_map.md | sequence_list.md, scene_beat_outline.md | 🟡 部分 |
| sequence_list.md | scene_beat_outline.md | 🟢 少量 |
| scene_beat_outline.md | 无 | ⚪ 无 |

### 7.3 处理机制

修改上游文件时，系统**不会自动重新生成**下游文件（用户意图优先），但会通过 Orchestrator 在回复中提示用户：

```
用户: "把世界观改成末世废土风格"
  → Orchestrator 调用 refine_worldbuilding → 执行成功
  → Orchestrator 回复：
    "世界观已更新为末世废土风格。注意：角色设定、剧情概要、
     幕结构等下游内容基于旧世界观生成，可能需要同步更新。
     告诉我是否需要调整它们。"
```

**实现方式**：Orchestrator 的 System Prompt 中有一条规则，要求它在 Tool 修改了上游文件后，提示用户下游文件可能需要更新。这不是代码级别的自动级联，而是通过 LLM 的推理能力实现的**提示级联**。

### 7.4 显式级联同步（用户主动触发）

如果用户想主动触发级联同步，可以直接要求：

```
用户: "世界观改了，把我的角色和剧情也基于新世界观更新一下"
  → Orchestrator 依次调用:
    → refine_characters(instruction="根据更新后的世界观同步更新角色设定")
    → refine_plot_synopsis(instruction="根据更新后的世界观同步更新剧情概要")
```
