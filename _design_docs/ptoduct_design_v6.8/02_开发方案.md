## 2. 改动 ②：并发多序列生成（吸收 v6.7.5）

### 2.1 runWithConcurrency（新增私有方法）

```ts
/** 有界并发池：最多 limit 个 worker 同时跑，超出排队。out[i] 与 items[i] 下标对齐。 */
private async runWithConcurrency<T, R>(
  items: T[], limit: number, worker: (item: T) => Promise<R>,
): Promise<R[]> {
  const out: R[] = new Array(items.length)
  let next = 0
  const runners = Array.from({ length: Math.min(limit, items.length) }, async () => {
    while (next < items.length) {
      const i = next++
      out[i] = await worker(items[i])
    }
  })
  await Promise.all(runners)
  return out
}
```

```ts
/** 全序列并行并发池上限。默认 3(保守防 429)。 */
```

### 2.4 sendWithRateLimit（429 退避，新增私有方法）

```ts
/** 429 指数退避重试(复用 classifyLLMError 识别 rate_limit);非 429 直接抛交原流程。 */
private async sendWithRateLimit(systemPrompt: string, userContent: string): Promise<string> {
  const MAX_429_RETRIES = 3
  for (let i = 0; i < MAX_429_RETRIES; i++) {
    try {
      return await this.llm.sendMessage(systemPrompt, userContent)
    } catch (e) {
      if (classifyLLMError(e).type !== 'rate_limit' || i === MAX_429_RETRIES - 1) throw e
      const backoff = 1000 * Math.pow(2, i) // 1s → 2s → 4s
      this.emit('tool_retry', {
        toolId: '', toolName: '', skillId: '', skillName: '',
        attempt: i + 1, maxAttempts: MAX_429_RETRIES,
        message: `429 限流，${backoff}ms 后重试 (${i + 1}/${MAX_429_RETRIES})`,
      })
      await new Promise((r) => setTimeout(r, backoff))
    }
  }
  throw new Error('unreachable')
}
```

## 3. 改动 ③：checker 精修范围约束

### 3.1 story_checker SKILL 报告结构化

[story_checker/SKILL.md](../src/skills/story_checker/story_checker/SKILL.md) 「汇总发现」表加「层级」「目标序列」两列：

```
| 严重等级 | 层级 | 目标序列 | 问题描述 | 所在文件 | 建议修复方向 |
|---------|------|---------|---------|---------|------------|
| critical | 序列 | S1-3 | 伏笔 F-02 缺失... | sequences/S1-3.md | 在 S1-3 中增加铺设 F-02 的场景 |
| minor | 上游 | — | 序列 S2-2 缺少钩子 | sequence_list.md | 建议补充可选钩子 |
```

「核心原则」补一条 **层级判定规则**：

- **序列级**：问题根源在某个 `sequences/<ID>.md` 的场景/节拍内容（节拍动作、场景目标、伏笔铺设位置等），`目标序列` 必填 `S{幕}-{序}`。
- **上游级**：问题根源在上游设定文件（worldbuilding/characters/act_map/sequence_list/foreshadowing/subplots）的矛盾或缺失，`目标序列` 填 `—`。
- 判定原则：**能通过精修单序列解决的，一律标「序列」**；只有上游设定本身矛盾（如 act_map 与 sequence_list 映射断裂、foreshadowing 规划与 sequence_list 冲突）才标「上游」。

输出格式示例段同步加列。

**机器可读标记（引擎 Guard-3 扫描用，关键）**：报告末尾（`<<<CHECK_REPORT_END>>>` 之前）必须输出一行 HTML 注释标识本轮审计范围：

```
<!-- AUDIT_SCOPE: sequence_only -->   <!-- FAIL 且全部为序列级问题 -->
<!-- AUDIT_SCOPE: has_upstream -->    <!-- FAIL 且含上游级问题 -->
<!-- AUDIT_SCOPE: pass -->            <!-- PASS/WARNING,无需修复 -->
```

引擎**不解析表格单元格**（「层级」列是给人看的），只扫描该注释字符串。`auditFixMode` 由标记判定（`sequence_only`/`has_upstream` → true，`pass`/漏标 → false）；只有 `sequence_only` 时 Guard-3a 才拦截上游工具；`has_upstream`/漏标时上游可见（默认放行，避免误拦上游真问题）。checker SKILL 必须稳定输出该标记（prompt 强约束）。

### 3.2 orchestrator_v5.md 审计闭环改写

[orchestrator_v5.md:132-142](../src/llm/prompts/orchestrator_v5.md#L132) 审计闭环段「审查未通过」步骤改写：

```
- **审查未通过**(总体结论 FAIL):
  1. 逐条读取「汇总发现」表的「层级」与「目标序列」列
  2. **序列级问题**:必须调 `scene_beats` 带 `target_sequence=<目标序列>` 精修该单序列
     - 禁止调其他序列的 scene_beats
     - 禁止空靶(全量覆写)——引擎会在审计轮拒收空靶
  3. **上游级问题**:才允许调对应上游工具(仅报告点名的那个文件对应的工具)
  4. **报告全是序列级问题时**:禁止调任何上游设定工具(worldbuilding/characters/act_map/sequence_list/foreshadowing/subplots)——引擎会在审计修复轮拒收上游工具调用
  5. 修复后再次调用 story_checker,重复不超过 3 轮
  6. 超 3 轮后无论结果如何都回复用户
```

### 3.3 引擎 Guard-3a：审计修复轮上游拦截（dispatch 兜底）

**为何不在 visibleSubagents 裁剪**：`visibleSubagents` 与 `toolSpecs` 在 FC 循环**外**一次性算定（[line 1208-1225](../src/orchestrator/orchestratorEngine.ts#L1208)），循环内不重算；而审计修复轮发生在 FC 循环内的后续轮（`state.auditRound` 在 [line 1467](../src/orchestrator/orchestratorEngine.ts#L1467) 递增）。FC 面裁剪对审计轮无效。且 `state.auditRound` 是 `processUserInput` 局部变量，`executeTool` 访问不到。故 Guard-3 走与 Guard-2 同款的 **executeTool dispatch 兜底 + 实例字段**路径。

**实例字段（承载审计状态，跨 executeTool 可见）**：

```ts
private auditFixMode = false        // story_checker 报 FAIL 时置 true;PASS/WARNING 置 false
private auditScope: 'sequence_only' | null = null  // checker 标 sequence_only 时设,否则 null
```

- `processUserInput` 开头 reset：`this.auditFixMode = false; this.auditScope = null`（新一轮用户输入不继承上轮审计模式——避免"检查后用户改做别的"被误拦上游）。
- story_checker 成功钩子（[line 1460-1467](../src/orchestrator/orchestratorEngine.ts#L1460)）设值：读 `report`，扫总体结论与 `<!-- AUDIT_SCOPE: ... -->` 标记：

```ts

  const report = await this.fileManager.readFile('_check_report.md')
```

## 6. 验证方法

1. `npx tsc -b` 通过（删 `ValidationSet` import / `BEAT_BLOCK_FIELDS` / vset 形参无残留引用）。
2. **校验放宽**：`npm run dev` → 故意让 scene_designer 产「视角人物为空」的场景表、beat_writer 产「缺情绪字段」「相邻同类型」的节拍 → 确认不再 retry、正常落盘；软校验 warning 仍报越界。
3. **并发**：建好 sequence_list（≥6 序列）→ 「铺所有序列」→ 资产面板短时间内集中出现多张骨架卡片；执行日志多序列 LLM 调用交错；墙钟对比 v6.7 串行近似减半。
4. **429 模拟**：临时把 `PIPELINE_CONCURRENCY` 调到 10 或 mock sendMessage 抛 429 → 确认退避重试生效、不崩。
5. **checker 精修**：制造单序列节拍问题 → checker 报告标「序列 S1-3」→ 确认 Orche 调 scene_beats 带 `target_sequence=S1-3`、只 S1-3 变化、其余序列 diff 为空、上游设定文件不动。
6. **Guard-3a 验证**：报告全序列级时 → 确认审计轮 FC 面无上游工具；制造 act_map 与 sequence_list 映射断裂 → checker 标「上游」→ 确认上游工具可见且可调。
7. **Guard-3b 验证**：审计轮模拟 Orche 调 scene_beats 空靶 → 确认 executeTool 拒收、返回错误提示。
8. **精修不受并发影响**：带 `target_sequence` 走 `runSequencePipeline` 单序列路径，不进并发池。
9. **写作期回归**：并行产出的 sequences 文件被 writer `<current_sequence_beats>` 正常消费。

## 7. 回滚

三块解耦：

- ① 校验放宽回滚 = `checkSceneTable`/`checkBeatBlocks` 恢复删掉的校验块 + 恢复 vset 形参。
- ② 并发回滚 = `runBatchPipeline` 换回 `for...await`、sendMessage 换回直调、删 `runWithConcurrency`/`sendWithRateLimit`/`PIPELINE_CONCURRENCY`。
- ③ checker 约束回滚 = SKILL 表去列 + prompt 审计闭环恢复旧文 + 删 Guard-3a/3b。

每块可独立回滚（一个提交）。
