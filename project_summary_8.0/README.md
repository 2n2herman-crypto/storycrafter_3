# StoryCrafter 3

基于多 Agent 协作的 AI 故事创作辅助平台。用户通过对话驱动一套 **Orchestrator → Subagent → Skill Router → Skill** 四层 Agent 框架，从零散想法逐步生成世界观、角色、结构大纲，直至完整的小说 / 剧本 / 长剧 / 短剧正文，并可导出为 Markdown 或 Word 文档。

## 这是什么

传统的 AI 写作工具通常是"一句提示词生成一段文本"，缺乏对长篇故事结构一致性的把控。StoryCrafter 3 把创作过程拆解为多个专业分工的 Agent（世界观设定、角色设计、幕结构、序列大纲、场景节拍、伏笔追踪、支线管理、一致性审查、正文写作等），由 Orchestrator 根据用户输入自动调度合适的 Agent 协同工作，每个 Agent 只读写它职责范围内的资产文件，最终在页面上以卡片 + Diff 的形式呈现所有创作资产的变化。

## 主要功能

- **对话式创作驱动**：用自然语言描述创作需求，系统自动理解、拆解任务并调用对应的创作模块，无需手动切换"功能页面"。
- **四产品类型支持**：小说、电影剧本、长剧、短剧四种产品档案，每种档案的结构层级（幕-序-场-节）、字数规范、校验规则均不同，一经选定会话内锁定，保证创作风格统一。
- **结构化故事资产体系**：自动生成并维护世界观设定、角色设定、幕结构图、序列清单、场景节拍表、伏笔追踪表、支线管理表等结构化 Markdown 资产，各资产间存在读写依赖关系，前置资产变化会驱动下游资产更新。
- **正文自动写作**：基于已确认的结构化设定，按目标章节/集数分段生成小说、剧本、长剧、短剧正文，支持角色行为轨迹与伏笔埋设/回收的自动标注追踪。
- **一致性检查闭环**：内置故事一致性审查 Agent，可对已有资产做多维度检查（设定冲突、伏笔遗漏、节奏问题等），并触发"审查 → 修改"的自动闭环。
- **设计期 / 写作期阶段闸门**：创作分为"设计期"（设定与结构确定）和"写作期"（正文生成）两个阶段，进入写作期后关键设定资产自动锁定，避免正文生成过程中设定被意外回改。
- **资产卡片与 Diff 对比**：所有生成/修改的资产以卡片形式在界面中展示，支持查看修改前后的词级 Diff，直观回溯每一次 AI 修改。
- **文件导入导出**：支持将现有 Word（.docx）文档导入并由 AI 自动归一化拆解为结构化资产种子；创作产出可导出为 Markdown 或 Word 文档。
- **多项目与历史持久化**：支持创建和切换多个故事项目，每个项目的资产文件、对话历史、执行日志均持久化保存，刷新或重启后不丢失。
- **LLM 配置管理**：内置设置页面，可配置多个 LLM 服务商/模型档案并测试连通性，API Key 保存在后端、不暴露在浏览器中。
- **Skill 热插拔扩展**：创作能力以 Skill 文件夹形式组织，支持运行时上传自定义 Skill 扩展 Agent 能力，无需修改代码或重新构建。

## 用户可以用它做什么

- 从一句话的故事灵感开始，逐步与 AI 对话完善出完整的世界观、人物关系和故事结构。
- 针对已经想好的大纲，让 AI 按幕/序/场/节的层级结构展开细化，生成可执行的分场景节拍表。
- 选择小说、剧本、长剧或短剧中的一种产品形态，让 AI 按该形态的行业规范（如短剧的强反转节奏、长剧的多线叙事与集间钩子）产出对应正文。
- 导入已有的 Word 文档素材（如已写的设定或草稿），让 AI 自动识别归类为结构化创作资产，在此基础上继续创作。
- 在创作过程中随时让 AI 做一致性审查，发现设定冲突、伏笔遗漏等问题并自动修正。
- 管理多个独立的故事项目，在不同项目间切换而不互相干扰。
- 将最终成果导出为 Word 或 Markdown 文档，用于后续排版、投稿或团队协作。

## 技术架构

- **前端**：Vite 5 + React 18 + TypeScript 5，Zustand 管理状态，`react-markdown` 渲染 Markdown，词级 Diff 展示修改对比。
- **后端**：Node.js + Express，负责 LLM 请求代理、配置管理、项目与资产文件持久化、对话历史存储、Word 文档导入导出（`mammoth` + `turndown` + `docx`）。
- **LLM**：OpenAI 兼容接口，支持 Function Calling，用于 Orchestrator 的 Agent 调度决策。
- **Agent 框架**：Orchestrator（任务拆解与调度）→ Subagent（角色职责与质量把控）→ Skill Router（能力选择）→ Skill（具体系统提示词与文件读写声明），Skill 以目录约定的方式组织，支持热插拔。

## 快速开始

### 环境要求

- Node.js >= 18
- npm

### 安装依赖

```bash
npm install
```

### 配置 LLM

首次启动前，在项目根目录创建 `.env.local` 并配置 LLM API Key（也可在启动后通过设置页面配置）：

```bash
VITE_DEEPSEEK_API_KEY=your_api_key_here
```

### 本地开发

```bash
npm run dev
```

该命令会同时启动前端（Vite，默认端口 5173）和后端（Express，默认端口 3001），前端通过 `/api/*` 代理访问后端。

### 生产构建与启动

```bash
npm run build
npm start
```

构建后由后端单端口（默认 3001）托管前端静态资源与 API 服务。

### 类型检查

```bash
npx tsc -b
```

## 目录结构

```
/storycrafter_3
├── server/                # 后端：Express API、LLM 代理、文件持久化、Word 导入导出
│   └── src/
│       ├── routes/        # API 路由
│       └── services/      # 业务逻辑（配置存储、项目存储、Skill 注册等）
└── web/                   # 前端：Vite + React SPA
    └── src/
        ├── api/           # 统一的后端请求层
        ├── components/    # UI 组件（布局、资产卡片、对话框等）
        ├── orchestrator/  # Orchestrator 调度引擎、Skill Router、上下文组装
        ├── skills/        # Subagent 与 Skill 定义（每个子目录对应一个 Subagent）
        ├── store/         # Zustand 全局状态（资产、对话、UI、阶段闸门）
        └── types/         # 类型定义（含四产品档案配置）
```

## 已知限制

- 暂无自动化测试覆盖。
- Orchestrator 的长对话上下文压缩当前为简单截断策略，超长创作流程下上下文质量有限。
- 文件关系系统（资产间引用关系可视化）当前仅预留接口，未实现完整功能。
