# 数据接口契约文档 (Data Interface Contract)

本系统摒弃复杂的JSON解析，所有Agent间的数据流转与状态持久化，**严格且仅依赖以下 6 个标准化 Markdown 文件**。Orchestrator 作为唯一的文件读写枢纽。

## 1. 核心资产文件清单 (Artifacts)

本系统的资产文件由 AgentRegistry 统一管理。以下为当前 MVP 的 6 个核心文件。  
远期加 Agent 时，该表自动扩展——新增 Agent 的 `writes` 数组中的文件即为新资产。

| 阶段 | 文件名 (Artifact ID) | 内容定义 | 生产者 | 消费者 |
| :--- | :--- | :--- | :--- | :--- |
| **阶段一** | `worldbuilding.md` | 世界观、规则、核心设定 | Orchestrator | Agent A, Agent B |
| **阶段一** | `characters.md` | 角色小传、动机、人物关系 | Orchestrator | Agent A, Agent B |
| **阶段一** | `plot_synopsis.md` | 故事梗概、核心冲突、主线走向 | Orchestrator | Agent A, Agent B |
| **阶段二** | `act_map.md` | 幕级地图（宏观结构与情绪弧线） | Agent A | Agent B |
| **阶段二** | `sequence_list.md` | 序列清单（中观情节块与功能定位）| Agent A | Agent B |
| **阶段三** | `scene_beat_outline.md`| 场景-节拍整体大纲（微观叙事原子）| Agent B | 最终交付/下游 |

## 2. 传输与注入协议 (Injection Protocol)

Orchestrator 在调用下游 Agent 时，必须将上游已生成的 MD 文件内容，通过严格的 XML 标签包裹后，注入给下游 Agent 的 Prompt 中。
**注入示例**：
```text
<worldbuilding>
{worldbuilding.md 的纯文本内容}
</worldbuilding>

<act_map>
{act_map.md 的纯文本内容}
</act_map>

## 3. 前端 IPC 接口定义 (Frontend IPC Protocol)

前端通过 Electron 的 `contextBridge` 暴露 API，实现渲染进程与主进程的隔离通信。

### 3.1 Preload 暴露 API

```typescript
interface StoryAPI {
  // 文件操作
  readFile(filename: string): Promise<string>
  writeFile(filename: string, content: string): Promise<void>
  listAssetFiles(): Promise<AssetFileInfo[]>

  // 快照管理
  saveApprovedSnapshot(filename: string): Promise<void>
  getApprovedSnapshot(filename: string): Promise<string | null>

  // 文件导入
  importFile(filePath: string): Promise<ImportResult>

  // 文件监听 (chokidar)
  onFileChange(callback: (info: FileChangeInfo) => void): void
  removeFileChangeListener(): void
}

interface AssetFileInfo {
  path: string
  filename: string
  stage: number               // 阶段编号（1/2/3/...），从 AgentSpec.phase 计算
  agentId: string             // 生产该文件的 Agent ID，对应 AgentRegistry
  group: string               // 前端分组标签，从 AgentSpec.group 读取
  exists: boolean
  hasApproved: boolean
}

interface FileChangeInfo {
  path: string
  event: 'change' | 'add' | 'unlink'
}
```

### 3.2 IPC 通道清单

| 通道名 | 方向 | 参数 | 返回值 | 用途 |
|--------|------|------|--------|------|
| `read-file` | renderer → main | filename: string | string | 读取 MD 文件内容 |
| `write-file` | renderer → main | filename, content | void | 写入 MD 文件 |
| `list-asset-files` | renderer → main | - | AssetFileInfo[] | 列出 6 个资产文件状态。可见性过滤需结合 `PHASE_ORDER`（见 stateMachine.ts）在渲染层计算 |
| `save-approved-snapshot` | renderer → main | filename: string | void | 保存确认快照 |
| `get-approved-snapshot` | renderer → main | filename: string | string\|null | 读取快照内容 |
| `import-file` | renderer → main | filePath: string | ImportResult | 导入 Word/Excel/MD |
| `write-import-to-asset` | renderer → main | filename, content | void | 导入内容写入资产卡片 |
| `file-changed` | main → renderer | FileChangeInfo | - | 文件变更推送 |

## 4. 快照文件规范 (Approved Snapshot)

### 4.1 文件命名

```
{original_filename}.approved.md
```

示例：
- `worldbuilding.md` → `worldbuilding.approved.md`
- `act_map.md` → `act_map.approved.md`

### 4.2 存储位置

与原始 MD 文件同目录（`story_craft_v3/` 根目录）。

### 4.3 文件格式

```markdown
---
snapshot_of: worldbuilding.md
approved_at: 2026-06-24T10:30:00+08:00
---

(此处为原始文件的纯文本内容)
```

### 4.4 生命周期

| 事件 | 动作 |
|------|------|
| 用户点击确认按钮 | 创建/覆盖对应 MD 文件的 `*.approved.md` |
| 系统回退到 STATE_1_CONCEPT | 清空当前阶段及之后的所有快照文件（含 Phase 2/3）；从 COMPLETED 回退时额外清空 Phase 1 快照 |
| 用户重新生成某卡片 | 旧快照保留，再次确认时更新 |
