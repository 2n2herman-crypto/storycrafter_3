### 步骤 1：新增类型 `ConversationTurn`
文件：[src/types/index.ts](../src/types/index.ts)

```ts
/** 传给引擎的轻量对话轮次（用于跨轮需求记忆，v5.5） */
export interface ConversationTurn {
  role: 'user' | 'assistant'
  content: string
}
```

### 步骤 2：contextAssembler 支持对话历史段
文件：[src/orchestrator/contextAssembler.ts](../src/orchestrator/contextAssembler.ts)

- `buildAgentPrompt` 增加可选参数 `history?: ConversationTurn[]`。
- 当 history 非空，在返回串**前置** `<conversation_history>` 段，逐行渲染「用户：/助手：」。
- 无 history 时行为与现状完全一致（向后兼容其它 Subagent 调用）。

```ts
export function buildAgentPrompt(
  context: string,
  instruction: string,
  history?: ConversationTurn[],
): string {
  const historyBlock = renderHistory(history)   // 空则返回 ''
  return [historyBlock, context,
    `<user_revision_instruction>\n${instruction}\n</user_revision_instruction>`]
    .filter(Boolean).join('\n\n')
}
```

### 步骤 3：engine 支持 history + 前置合并
文件：[src/orchestrator/orchestratorEngine.ts](../src/orchestrator/orchestratorEngine.ts)

1. `executeTool(subagent, instruction, history?)` 新增可选 `history`，透传给 `buildAgentPrompt`（[:156](../src/orchestrator/orchestratorEngine.ts)）。
2. `processUserInput(userInput, history, onEvent)` 新增 `history: ConversationTurn[]` 参数（放在 onEvent 前，或作为 options —— 采用位置参数 `(userInput, history, onEvent)`）。
3. **前置合并（机制 A）**：在 FC 循环前调用：
   ```ts
   const analyzer = getSubagent('user_requirements_analyzer')
   if (analyzer) {
     await this.executeTool(analyzer, MERGE_INSTRUCTION, history)
   }
   ```
   `MERGE_INSTRUCTION` 为 02 文档定义的合并指令常量。发一个 `tool_start`/`tool_complete` 事件让 UI 可见（可选）。
4. **历史前置（机制 B）**：FC 循环的 `messages` 初始化改为：
   ```ts
   const messages = [
     ...history.map(t => ({ role: t.role, content: t.content } as ChatCompletionMessageParam)),
     { role: 'user', content: userInput },
   ]
   ```

3. `npm run dev` 手动多轮对话验收：

## 风险与回退

- **额外成本**：每轮多 1 次 analyzer 调用。若成本敏感，可加开关或仅在检测到 history 非空时前置合并。
- **前置合并与 Orchestrator 重复选 analyzer**：Orchestrator 仍可能再选 analyzer；因 analyzer 幂等（高精度低召回 + 保留状态标记），重复执行不产生副作用，但会多一次调用。可在 orchestrator prompt 中提示"需求已在每轮自动整理，一般无需再手动调用需求整理者"。
- **回退**：机制 A、B 相互独立，可分别回退。全部回退即恢复 v5.3 行为。
