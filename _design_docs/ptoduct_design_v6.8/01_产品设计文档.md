# v6.8 产品设计文档 · 校验放宽 + 并发输出 + checker 精修范围约束

> 三项独立但互补的需求：① 放宽 scene_beats 机械校验，消除「写死导致无法输出」；② 多序列并发生成（闭合偏差 ③）；③ `story_checker` 审计修复约束为「只精修出错序列、仅序列问题时不动上游指导文件」。
> 开发方案见 [02_开发方案.md](./02_开发方案.md)，落地顺序见 [03_开发步骤.md](./03_开发步骤.md)。

## 1. 背景与目标

v6.7 已把 scene_beats 链路改成「建档 → 场景表 → 逐场景节拍块」三段式，并落地串行批量。联调中暴露三个产品问题：

1. **格式校验过严，反噬可用性**：场景表 6 项 + 节拍块 7 项机械校验里，多项是「质量偏好」而非「可解析性底线」（如视角人物非空、节拍 4 字段齐全、类型词库命中、相邻不得同类型）。DeepSeek 偶发漏一个字段或换一种类型措辞就被判违规 → 反复 retry → 耗尽 `MAX_RETRIES` 降级占位 → 用户看到「待补节拍」。校验本意是兜底，却成了阻断输出的主因。
2. **串行批量墙钟过长**：20+ 序列 × (1 场景表 + 8~15 场景节拍) 次 LLM 串行，用户等待明显。偏差 ③（并行）在 v6.7 用串行替代，v6.7.5 已设计并发方案但未落地。
3. **审计修复「伤及无辜」**：`story_checker` 报告某序列节拍有问题时，Orchestrator 可能调 `sequence_list`（上游）或对其他序列重跑 `scene_beats` 空靶（全量覆写），导致已验收序列被改坏、上游设定被无意篡改。用户期望：序列级问题只精修该序列，只有序列出问题时不得动上游指导文件。

三项需求的共同主题：**让系统在「质量保障」与「可用性/最小破坏」之间重新校准**——校验从「质量裁判」退回「可解析性底线」，并发提速，审计修复做到外科手术式精修。

## 2. 三项需求总览

| 需求 | 一句话 | 主要载体 | 收益 |
|---|---|---|---|
| ① 校验放宽 | 场景表去 2 项、节拍块去 4 项机械校验，只留可解析性底线 | [structuralChecks.ts](../src/skills/scene_beats/structuralChecks.ts) | 首次通过率↑，降级占位↓ |
| ② 并发多序列 | 全序列批量从串行 for 改并发池，429 退避重试 | [orchestratorEngine.ts](../src/orchestrator/orchestratorEngine.ts) `runBatchPipeline`/`runSingleStep` | 墙钟 ≈ 减半（并发 3） |
| ③ checker 精修约束 | 审计修复只改出错序列；仅序列问题时不改上游 | [story_checker SKILL](../src/skills/story_checker/story_checker/SKILL.md) + [orchestrator_v5.md](../src/llm/prompts/orchestrator_v5.md) + 引擎 Guard-3 | 审计不伤及无辜 |

> 与 v6.7.5 的关系：需求 ② 即 v6.7.5 已设计的并行专项方案，本次正式落地（吸收 [v6.7.5 方案](../ptoduct_design_v6.7.5/01_并行输出专项方案.md)）。需求 ①③ 是 v6.8 新增。

## 3. 需求 ①：校验放宽

### 3.1 设计原则

校验分两层：

- **可解析性底线**（保留）：输出无法被下游消费的，硬拦。如场景表不是合法表格、节拍块没有 `[BEAT ...]` 块头、B-ID 内嵌场景与目标场景不符。
- **质量偏好**（删除）：输出能解析但「不够好」的，交给 SKILL body 自然语言引导 + 软校验非阻塞 warning，不硬拦。如视角人物是否非空、节拍字段是否齐全、类型是否命中词库、相邻是否同类型。

原则：**机械校验只兜「下游崩坏」，不裁判「创作质量」**。质量交给 prompt 引导（第一道防线）和软校验 warning（透明提示，不阻断）。

### 3.2 场景表校验 `checkSceneTable`（6 项 → 5 项）

当前 6 项（按 [structuralChecks.ts:94-138](../src/skills/scene_beats/structuralChecks.ts#L94) 实际顺序）：

| # | 校验项 | 性质 | 本次 |
|---|---|---|---|
| 1 | 裸竖线（表格外 `\|`） | 可解析性 | 保留 |
| 2 | 能解析为合法 markdown 表格 | 可解析性 | 保留 |
| 3 | 列数 = 7 | 可解析性（下游列切片依赖） | 保留 |
| 4 | 至少 2 行数据 | 质量偏好 | **删除** |
| 5 | 每行列数一致（单元格内裸竖线错位） | 可解析性 | 保留 |
| 6 | 场景 ID 匹配 `SC-S{幕}-{序}-{nn}` | 可解析性（下游 extractSceneIds/beat_writer 依赖） | 保留 |
| 7 | 视角人物不为空 | 质量偏好 | **删除** |

> 用户原话「场景表中的 3、7 去了」对应此前给用户的 7 项编号（1裸竖线/2可解析/3至少2行/4每行一致/5场景ID/6列数7/7视角人物）。映射到代码：删 `table.rows.length < 2` 块（至少2行）和 `viewpoint` 空判（视角人物）。

保留 5 项全是「下游消费硬依赖」：列数=7 是 `sliceSceneRow`/列切片的前提；场景 ID 格式是 `extractSceneIds` + beat_writer `<target_scene>` 的前提；每行一致防单元格裸竖线错位。

### 3.3 节拍块校验 `checkBeatBlocks`（7 项 → 3 项）

当前 7 项（按 [structuralChecks.ts:192-220](../src/skills/scene_beats/structuralChecks.ts#L192) 实际顺序）：

| # | 校验项 | 性质 | 本次 |
|---|---|---|---|
| 1 | 无无法识别的非字段行（strayLines） | 质量偏好 | **删除** |
| 2 | 至少 1 个 `[BEAT ...]` 块 | 可解析性 | 保留 |
| 3 | B-ID 格式 `B-SC-...` | 可解析性（收口拼接依赖） | 保留 |
| 4 | 内嵌 SC-ID = 当前 `<target_scene>` | 可解析性（防孤儿节拍） | 保留 |
| 5 | 4 字段齐全（类型/动作/情绪/伏笔） | 质量偏好 | **删除** |
| 6 | 类型命中 `vset.beatTypeVocab` 词库 | 质量偏好 | **删除** |
| 7 | 相邻不得同类型（`vset.enforceAdjacentTypeRule`） | 质量偏好 | **删除** |

> 用户原话「节拍块中的 1、5、6、7 不需要」对应此前给用户的 7 项编号（1 strayLines/2 至少1块/3 B-ID/4 内嵌SC/5 字段齐全/6 词库/7 相邻不同）。映射到代码：删 strayLines 判、`BEAT_BLOCK_FIELDS` 字段齐全循环、词库命中判、相邻同类型判。

保留 3 项是「收口与定位硬依赖」：至少 1 块（否则该场景无节拍可拼）、B-ID 格式（`assembleSequenceDoc` 拼接 + 下游 writer 消费）、内嵌 SC-ID 一致（防节拍挂错场景）。

### 3.4 连带清理

- `BEAT_BLOCK_FIELDS` 常量：仅服务于「字段齐全」校验，删校验则删常量。
- `checkBeatBlocks` 的 `vset: ValidationSet` 形参：词库/相邻规则删除后不再消费 vset，**删除形参**，签名改为 `checkBeatBlocks(md, sceneId)`；调用方 `runSequencePipeline` 同步去 vset 实参。
- `parseBeatBlocks` 的 `strayLines` 返回：`checkBeatBlocks` 不再消费，但 `parseBeatBlocks` 是导出纯函数，保留 `strayLines` 返回字段不变（避免改返回类型波及其他潜在调用方；该字段只是不再被校验消费）。
- `ValidationSet.beatTypeVocab` / `enforceAdjacentTypeRule` 字段：守 INV-3（`types/product.ts` 零改），字段保留不动（仅 `checkBeatBlocks` 不再读，成为事实死字段；未来若需要可复用）。
- 软校验 `runPipelineSoftValidation`（场景数/节拍数越界非阻塞 warning）：**不动**，仍是档案 `countRange` 的透明提示。

### 3.5 用户可见效果

- 首次通过率显著上升：偶发漏「视角人物」或「情绪」字段、类型用了词库近义词、相邻两拍同类型 —— 不再判违规、不再 retry、不再降级占位。
- 质量不失控：SKILL body 仍用自然语言引导字段齐全/类型多样；软校验仍报越界 warning（透明、不阻断）。
- 可解析性不破：表格结构、ID 格式、块头/内嵌一致性仍硬拦，下游消费无回归。

## 4. 需求 ②：并发多序列生成

### 4.1 产品逻辑

- 对用户透明：仍是「铺所有序列」一次指令、一次 tool_call（偏差 ④ 收益不丢）。
- 墙钟改善：并发 3，N 序列耗时从 O(N) 降到约 O(N/3)。
- 失败隔离：单序列失败不拖垮整批（v6.7 既有，并发下保留）。
- 429 自愈：DeepSeek 限流时指数退避重试，不人工干预、不消耗格式重试预算。

### 4.2 方案要点（吸收 v6.7.5）

- `runBatchPipeline` 串行 `for...await` → `runWithConcurrency(seqIds, 3, worker)`，汇总逻辑逐字不动（按下标对齐）。
- `runSingleStep` 的 `sendMessage` → `sendWithRateLimit`（429 指数退避 1s→2s→4s，3 次，不消耗 `MAX_RETRIES`）。
- 并发上限 `PIPELINE_CONCURRENCY = 3`（保守防 429；联调无 429 可上调）。
- 硬前提 INV-5：`runSequencePipeline` 必须自包含（无跨序列共享可变状态）——已代码核实。

### 4.3 用户可见效果

- 批量铺序列时，资产面板短时间内集中出现多张骨架卡片（非逐张等待）。
- 长流程（20+ 序列）等待时间明显缩短。
- 偶发 429 不再中断批量，日志可见退避重试。

## 5. 需求 ③：checker 精修范围约束

### 5.1 问题

当前审计闭环（[orchestrator_v5.md:132-142](../src/llm/prompts/orchestrator_v5.md#L132)）：checker 报 FAIL → Orche「选择要修复的工具（sequence_list / scene_beats / subplot_manager）」。两个隐患：

- **伤及其他序列**：Orche 可能调 `scene_beats` 不带 target（空靶=全量批量覆写），把已验收序列一起重写。
- **伤及上游**：序列节拍本身的问题，Orche 可能误判为上游设定缺陷，去改 `worldbuilding`/`characters`/`act_map`/`sequence_list`/`foreshadowing`/`subplots`，污染已冻结的设计期基线。

### 5.2 设计：三层约束

**第一层 · checker 报告结构化（序列级定位 + 层级标注）**

`story_checker` SKILL 的「汇总发现」表加一列 **层级**，值为 `序列` / `上游`：

| 严重等级 | 层级 | 目标序列 | 问题描述 | 所在文件 | 建议修复方向 |
|---|---|---|---|---|---|
| critical | 序列 | S1-3 | 伏笔 F-02 缺失... | sequences/S1-3.md | 在 S1-3 中增加铺设 F-02 的场景 |
| major | 序列 | S2-1 | 主角行为与性格不符... | sequences/S2-1.md | 修改节拍动作 |
| minor | 上游 | — | 序列 S2-2 缺少钩子 | sequence_list.md | 建议补充可选钩子 |

- 序列级问题：`目标序列` 必填（`S{幕}-{序}`），`所在文件` 为 `sequences/<ID>.md`。
- 上游级问题：`目标序列` 为 `—`，`所在文件` 为上游设定文件。
- 引擎靠扫描报告末尾的机器可读标记判定：`<!-- AUDIT_SCOPE: sequence_only -->`（FAIL+全序列级）/ `has_upstream`（FAIL+含上游级）/ `pass`（PASS/WARNING）——不解析表格单元格（「层级」列是给人看的）。漏标默认放行（不拦上游）。

**第二层 · Orchestrator prompt 审计闭环约束**

[orchestrator_v5.md](../src/llm/prompts/orchestrator_v5.md) 审计闭环段改写：

- 序列级问题 → **必须**调 `scene_beats` 带 `target_sequence=<目标序列>` 精修该单序列；禁止调其他序列的 scene_beats；禁止空靶（全量覆写）。
- 仅当报告含上游级问题 → 才允许调对应上游工具（且只调报告点名的那个上游文件对应的工具）。
- 报告全是序列级问题 → **禁止**调任何上游设定工具（worldbuilding/characters/act_map/sequence_list/foreshadowing/subplots）。
- 需求一致性（维度 7）问题按既有规则（先 user_requirements_analyzer 确认意图）。

**第三层 · 引擎硬兜底 Guard-3（防 prompt 不守）**

走与 Guard-2 同款的 **executeTool dispatch 兜底**路径（不在 `visibleSubagents` 裁剪——因 FC 面在循环外一次性算定，审计修复轮发生在循环内后续轮，FC 面裁剪对审计轮无效；且 `state.auditRound` 局部变量 `executeTool` 不可达）。审计状态由实例字段承载：

- **实例字段**：`auditFixMode`（AUDIT_SCOPE 标记为 `sequence_only`/`has_upstream` 时 true，`pass`/漏标时 false）+ `auditScope`（标记 `sequence_only` 时设，否则 null）；`processUserInput` 开头 reset（不跨用户输入残留），story_checker 成功钩子设值。
- **Guard-3a（上游拦截）**：`auditFixMode && auditScope === 'sequence_only'` 时，上游设定 subagent（worldbuilding/characters/act_map/sequence_list/foreshadowing_tracker/subplot_manager）在 `executeTool` 被拒收。漏标或 `has_upstream` → 上游可见（默认放行，不误拦上游真问题）。
- **Guard-3b（scene_beats 必须带靶）**：`auditFixMode` 时 `scene_beats` 空靶 → `executeTool` 拒收，返回「审计修复轮必须带 target_sequence 精修单序列，禁止全量覆写」。

### 5.3 用户可见效果

- 审计修复只动出错序列文件，其余序列卡片 diff 为空。
- 序列级问题不会波及上游设定文件（设计期基线不被审计污染）。
- 真正的上游矛盾（如 act_map 与 sequence_list 映射断裂）仍能被修复——因为报告会标 `<!-- AUDIT_SCOPE: has_upstream -->`（或漏标默认放行），Guard-3a 不拦截上游工具。

## 6. 风险与边界

| 风险 | 影响 | 缓解 |
|---|---|---|
| 校验放宽致质量下滑 | 字段缺失/类型单调的节拍落盘 | SKILL body 自然语言引导 + 软校验 warning 透明提示；联调抽查 |
| 并发 429 限流 | 批量中断 | 默认并发 3 + `sendWithRateLimit` 指数退避 |
| Guard-3a 误判（checker 把上游问题标成序列级，或漏标 AUDIT_SCOPE） | 上游真问题修不了 | 默认放行策略（只有显式 `sequence_only` 才拦，漏标按 `has_upstream` 处理）；checker SKILL 层级判定原则 + prompt 指引；审计轮上限 3 轮兜底 |
| Guard-3b 误拒（Orche 想全量重铺） | 审计轮无法全量覆写 | 这是预期行为（审计轮就是精修，全量重铺应在非审计轮由用户主动发起） |

## 7. 不做的事（本轮边界）

- 不改场景层为非表格（场景表仍是 7 列表格，仅删 2 项校验）。
- 不改 `types/product.ts`（守 INV-3）：`ValidationSet` 死字段保留。
- 不引入 `ProductProfile.pipelineConcurrency` 可配置字段（暂用常量 `PIPELINE_CONCURRENCY`）。
- 不做自动降并发（429 退避重试已兜底；联调若 429 频繁再加）。
- 不改 writer 链路、不改 `validateOutput` 主体、不改 `assembleContext`。
