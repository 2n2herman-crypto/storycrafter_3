## 一、核心抽象：ProductProfile

### 1.1 类型定义（`src/types/product.ts` 新增）

```ts
export type ProductKind = 'novel' | 'screenplay' | 'long_drama' | 'short_drama'

/** 一层的规格：语义标签 + 数量约束 */
export interface LayerSpec {
  /** 该层在本产品中的显示语义（如短剧"幕"=全剧大阶段） */
  semantic: string
  /** 数量区间 [min, max]；子层相对父层的每单位数量 */
  countRange: [number, number]
  /** 硬约束描述（如短剧每幕恰 1 序列 → [1,1]） */
  note?: string
}

/**
 * validator 按产品切换的校验集合。
 * 注意：sceneIdRegex/beatIdRegex/sceneTableCols(7)/beatTableCols(6) 四产品**完全相同**
 * （文件结构红线不变），故它们**不进档案**、保留为 structuralChecks 内的共享常量。
 * ValidationSet 只承载**真正随产品变化**的项：节拍词库 + 相邻同类型约束开关。
 */
export interface ValidationSet {
  beatTypeVocab: string[]         // 允许的节拍主词集合（唯一强校验差异项）
  enforceAdjacentTypeRule: boolean // 是否强制"同场景相邻不得同类型"（小说关闭，剧作类开启）
}

export interface ProductProfile {
  kind: ProductKind
  displayName: string        // "小说" / "剧本" / "长剧脚本" / "短剧脚本"

  // === 四层语义 + 量级 ===
  act: LayerSpec             // 幕
  sequence: LayerSpec        // 序列（子层数量 = 每幕序列数）
  scene: LayerSpec           // 场景（子层数量 = 每序列场景数）
  beat: LayerSpec            // 节拍（子层数量 = 每场景节拍数）

  // === 叙事单位映射 ===
  sequenceToEpisode: 'none' | 'one_to_one' | 'one_to_many'
  // none=无集(小说/剧本); one_to_one=一序列一集(长剧); one_to_many=一序列多集(短剧)
  episodesPerSequence?: [number, number]  // one_to_many 时有效（短剧 8-15）

  // === 伏笔 ===
  foreshadowingMaxLifespan: number  // 伏笔最大寿命（以序列或集计）

  // === 成文 ===
  writerSubagentId: string   // 绑定的成文 Subagent id
  proseUnit: 'chapter' | 'sequence' | 'episode' | 'sequence_of_episodes'
  narrativeMode: 'unfold' | 'pulse'  // 展开式 / 脉冲式
  allowInnerMonologue: boolean       // 是否允许心理描写（小说 true，其余 false）
  outputAnnotations: ('behavior_track' | 'shot_breakdown' | 'foreshadow')[]

  /** 成文一次调用产出是否可能超单次 LLM 稳定上限 → 是则启用分段续写（长剧按场景、短剧按集） */
  proseSplitUnit: 'none' | 'scene' | 'episode'

  // === 运行时记忆窗口（Wave 6 消费，供 LRU/裁剪档案化）===
  behaviorTrackWindow: number    // behaviorTrack 保留条数（短剧 100 集需更大窗口）

  // === 校验 ===
  validation: ValidationSet
}
```

### 1.3 产品选择入口与锁定生命周期（回应 checker 第 1、2、7、8 条）

**入口机制（唯一真源，不留二义）**：产品由 **UI 显式选择器**选定，不靠 LLM 从对话推断。

```
新会话 → UI 顶部产品选择器（小说/剧本/长剧/短剧四选一，默认未选）
      → 用户点选 → chatStore.setProduct(kind) → engine.lockProfile(kind)
      → profileLock = PRODUCT_PROFILES[kind]（会话级锁定）
```

- `processUserInput` 签名扩展：`(userInput, history, onEvent)` → 由 engine 实例已持有的 `profileLock` 驱动，**不新增入参**（profileLock 在 `lockProfile()` 时落定，早于任何 sendMessage）。
- **废弃 02 §5.2 的"首产资产即锁定"分支**——它与"设计资产需档案注入才能生成"构成先有鸡先有蛋。**锁定时机收敛为单一：UI 选产品**。02 文档同步修正。

**硬闸门（Guard-0，补 checker 第 2 条的依赖倒置）**：

```
profileLock === null 时：
  · 设计区 + 成文区全部 Subagent 从 FC 面隐藏（buildFunctionSpec 过滤）
  · Orchestrator prompt 应答话术：引导用户"先在顶部选择产品方向"
  · 仅 reset_all 可用
→ 保证任何设计资产生成时，<product_profile> 必然已就绪 → 守红线③"参数由档案给、不靠 LLM 猜"
```

这条 Guard-0 是 SKILL.md 去硬编码数字（Wave 3）的**前置安全网**：去掉数字兜底后，绝不会出现 profileLock=null 却能触发设计的窗口。

**归一化与需求合并的顺序（补 checker 第 7 条）**：
- 用户投喂大文件时，`processUserInput` 每轮前置的 `user_requirements_analyzer` 合并**跳过**——当检测到本轮存在未归一化的 `_input_raw.md` 时，强制先走 `input_normalizer`，归一化产出种子资产后，才允许后续需求合并。避免整篇原文被当"需求"吞掉。

## 三、设计区各 Subagent 的档案化改造

### 3.1 改造原则

各设计 SKILL.md 正文**移除硬编码数字**，改为引用 `<product_profile>` 注入的区间与语义。示例（scene_designer）：

```
（改造前）场景数 N∈[3,6]
（改造后）场景数 N 取 <product_profile> 中 scene.countRange 给定的区间；
          场景语义参照 scene.semantic（如短剧模式下每行=一集）
```

### 3.2 `<product_profile>` 注入格式

Engine 在 resolveExtraContext 阶段为设计区 Subagent 注入：

```xml
<product_profile kind="short_drama" name="短剧脚本">
  <layer name="幕" semantic="全剧大阶段" count="3-5"/>
  <layer name="序列" semantic="多集弧" count="1-1" note="每幕恰含1序列"/>
  <layer name="场景" semantic="集(一集一场景)" count="8-15"/>
  <layer name="节拍" semantic="四拍微循环" count="4-6"/>
  <sequence_to_episode>one_to_many（8-15集）</sequence_to_episode>
  <beat_vocab>铺垫,触发,对抗,转折,收束,钩子,摩擦,尖峰,钉</beat_vocab>
  <foreshadowing_lifespan>10</foreshadowing_lifespan>
</product_profile>
```

### 3.3 每层的档案化要点

| Subagent | 读档案的字段 | 行为变化 |
|----------|------------|---------|
| act_map | `act.semantic` / `act.countRange` | 幕名称与数量随产品；短剧幕名用叙事阶段（Premise/Escalation/Dark Middle/Resolution） |
| sequence_list | `sequence.countRange` / `note` | 短剧强制每幕 1 序列；长剧序列=集 |
| scene_designer | `scene.countRange` / `scene.semantic` | 行数区间随产品（长剧 5-10、短剧 8-15）；短剧每行=一集 |
| beat_writer | `beat.countRange` / `validation.beatTypeVocab` | 节拍数与词库随产品；短剧启用四拍词库 |
| foreshadowing_tracker | `foreshadowingMaxLifespan` | 伏笔寿命窗口随产品 |
| story_checker | 全档案 | 审查标准随产品（短剧容忍扁平人物、快节奏） |

## 四、validator 档案化

### 4.1 现状问题
现有 `checkSceneTable` / `checkBeatTable`（`scene_beats/structuralChecks.ts`）把列数、ID 正则、类型词库、"相邻不得同类型"规则**硬编码**。四产品要不同词库/相邻约束 → 必撞校验。

> **抽象边界（回应 checker 第 9 条）**：列数(7/6)、ID 正则、场景表结构四产品**完全相同**（文件结构红线），故它们**保留为常量、不进档案**。ValidationSet 只承载真正随产品变的两项：`beatTypeVocab` + `enforceAdjacentTypeRule`。避免把不变字段塞进 per-product 档案造成维护面翻倍。

### 4.2 改造方案（守 INV-1 精神：最小侵入）

structuralChecks 从"读常量"改为"读传入的 ValidationSet"：

```ts
// 改造前：常量 + 无条件相邻检测
const BEAT_TYPE_KEYWORDS = ['铺垫','触发','对抗','转折','收束']

// 改造后：从档案传入
function checkBeatTable(md: string, scenesMd: string, vset: ValidationSet): CheckResult {
  const vocab = vset.beatTypeVocab
  // ...原校验逻辑不变；仅词库来源改为 vset，
  //    且"同场景相邻不得同类型"检测用 if (vset.enforceAdjacentTypeRule) 包裹（小说关闭）...
}
```

`runPipeline` 的 `buildStructuralCheckForStep()`（engine:129/132）绑定时，从 `profileLock.validation` 取 ValidationSet 传入。列数/ID 正则等不变项仍走常量。**校验逻辑本身不变，只是可变参数来源从常量改为档案**。

> **小说词库错配修复（回应 checker 第 10 条）**：小说档案已改用 `['铺垫','推进','转折','高潮','沉淀','留白']` 并 `enforceAdjacentTypeRule:false`——小说"叙事节拍"不套剧作五分张力法，也不强制相邻不同类型。

### 4.3 数量区间的软校验
场景数/节拍数落在 `countRange` 外时，产生**非阻塞 warning**（不 retry、不阻断落盘），沿用 v6.4 runSoftValidation 机制，扩展为读档案区间。

### 5.1 四个 writer Subagent

```
src/skills/novel_writer/

src/skills/screenplay_writer/

└── screenplay_writer/SKILL.md

src/skills/long_drama_writer/
├── subagent.md              # 长剧写作专家（单集弧+集间钩子/多线/视听化）
└── long_drama_writer/SKILL.md

src/skills/short_drama_writer/
├── subagent.md              # 短剧写作专家（脉冲式/4拍/每集反转/镜头分解）
└── short_drama_writer/SKILL.md
```

### 5.2 共享 vs 特化的内容划分

| 内容 | 处置 |
|------|------|
| 上游参照区段说明、双模式判定、TAG 格式、伏笔纪律、行为追踪注释、**伏笔状态注释** | **四 writer 共享**（措辞一致，可从公共片段复制） |
| 心理→视听转译 / 禁心理描写 | 仅 screenplay/long_drama/short_drama 有；**novel 相反**（鼓励心理描写、叙述声音） |
| 4 拍微循环 / 镜头分解 SHOT_BREAKDOWN / 脉冲式 | **仅 short_drama** |
| 单集弧 + 集间钩子 | 仅 long_drama / short_drama |
| 成文粒度、字数区间 | 各自按 `proseUnit` 定义 |

> **`<!--FORESHADOW:-->` 注释由 writer 生产（补 checker 第 5 条）**：四 writer 共享片段中新增一条——正文 END TAG 之后、与 `BEHAVIOR_TRACK` 注释并列，输出伏笔状态注释：
> ```
> <!-- FORESHADOW: F-id=plant|payoff@章节id -->
> ```
> 引擎 Wave 6 的 `foreshadowingState` 正则从此注释提取。**沿用现有 BEHAVIOR_TRACK 注释的既定模式**（`script_writer/SKILL.md:102` 已有先例），不新造机制。foreshadowing_tracker 的 `FORESHADOWING_START/END` TAG 是设计期资产，与此运行期注释是两回事，不冲突。

### 5.3 成文粒度（回应"字数太粗糙"）

以**叙事单元**为界，字数为副产物：

| 产品 | 一次调用产出 | 界定标准 | 分段策略 `proseSplitUnit` |
|------|------------|---------|--------------------------|
| 小说 | 一章 | 覆盖该章全部场景的完整叙事，字数自然浮动 | `none`（单次产完） |
| 剧本 | 一序列 | 覆盖该序列全部场次 | `none` |
| 长剧脚本 | 一集(=一序列) | 覆盖该集全部场景，单集完整弧线 | `scene`（超限时按场景续写） |
| 短剧脚本 | 一序列(=多集) | 覆盖该序列 8-15 集，每集 4 拍闭环 | `episode`（**必分段**，按集续写） |

**分段续写机制（回应 checker 第 3 条——短剧才是超限重灾区）**：

`proseSplitUnit` 声明该产品成文单元的续写粒度。引擎按此循环调用 writer、每段独立 START/END TAG 校验后 **REFINE 追加**进同一 `chapters/S{幕}-{序}.md`：

- **短剧**（`episode`）：一序列 8-15 集，体量远超长剧单集，一次产出必然撞 LLM 输出上限 → END TAG 被截断 → validator 失败。故**强制按集分段**：engine 逐集调用 short_drama_writer，`batchProgress` 记录已完成集号，支持断点续写。这是短剧成文的**默认路径**，非异常兜底。
- **长剧**（`scene`）：单集 5-10 场景，多数可一次产出；仅当超限时按场景分段。
- **小说/剧本**（`none`）：单次产完，不分段。

> 原风险表只给长剧配了缓解、漏了短剧，此处修正：短剧的分段是**设计内置的默认成文路径**，已写入档案 `proseSplitUnit:'episode'` 与批次策略。

### 5.4 FC 面互斥可见
`buildFunctionSpec` 依据 `profileLock.writerSubagentId`，成文区**只把绑定的那个 writer** 放进 FC 工具列表，其余三个 writer 对 Orchestrator 不可见 → LLM 不可能选错产品的写法。

## 六、输入归一化协议

### 6.1 input_normalizer Subagent

```
src/skills/input_normalizer/
├── subagent.md
└── input_normalizer/SKILL.md
```

- **reads**：`_input_raw.md`（用户投喂的原始文件，内部资产以 `_` 前缀隐藏）
- **writes**：动态——按识别结果写入 `worldbuilding.md` / `characters.md` / `act_map.md` / `sequence_list.md` / `user_requirements.md` 的种子版本
- **outputTags**：`<<<NORMALIZED_START>>>` / `<<<NORMALIZED_END>>>`

### 6.2 归一化分类逻辑

```
输入原文 → LLM 分类判定输入类型：
  ├─ 纯概念   → 只产 user_requirements 种子
  ├─ 梗概/大纲 → 产 act_map + sequence_list 种子
  ├─ 已有原文  → 反向抽取 worldbuilding + characters 种子（+ 标记已有 chapters 存量）
  ├─ 设定资料  → 产 worldbuilding + characters 种子
  └─ 结构化表格 → 产 act_map + sequence_list 种子
```

### 6.3 产品档案对归一化的约束
- 归一化在**产品已选定**后运行，按目标产品的层语义组织种子资产（如目标是短剧，则序列种子按"每幕1序列"约束生成）。
- 归一化产物是**建议性种子**，用户可在设计区继续修订。

### 6.4 多文件输入与写入链路（补 checker 第 8 条）
- **`_input_raw.md` 的写入链路**：现有 `importStore.confirmImportToAsset(targetPath)` 目前只提示"覆盖资产卡片功能待实现"（`importStore.ts:51`），尚未真正写盘。Wave 5 把 UI 投喂落到 `confirmImportToAsset('_input_raw.md')`：单文件直写、多文件追加合并（带 `<<< 来源:文件名 >>>` 分隔标记）。
- **持久化缺口显式承认**：`InMemoryFileManager` 刷新即丢（既有已知缺口）。v6.6 不解决持久化——归一化输入与其他资产同为内存态，刷新丢失属于全局已知限制，不在本方案范围。
- **增量**：新投喂文件时归一化只补充/修订相关资产，不清空已有。
- **执行顺序**：`input_normalizer` 必须先于每轮前置的 `user_requirements_analyzer` 合并（见 §1.3），否则原文会被误当需求吞掉。

## 七、文件结构不变性（红线复述）

| 不变项 | 说明 |
|--------|------|
| 资产路径 | `worldbuilding.md` / `characters.md` / `act_map.md` / `sequence_list.md` / `foreshadowing.md` / `subplots.md` / `sequences/S{幕}-{序}.md` / `chapters/S{幕}-{序}.md` 四产品完全一致 |
| 表格骨架 | 场景表 7 列 / 节拍表 6 列不变（列语义可随产品，列数不变） |
| ID 格式 | S{幕}-{序} / SC-... / B-... 不变 |
| TAG 包裹 | 各 START/END TAG 不变 |
| Pipeline 步序 | scene_designer → beat_writer → assemble 不变 |
| Phase Gate | 锁定机制不变，profileLock 并存 |

变化的**只有**：各层内容语义、数量规格、节拍词库、成文风格、成文 Subagent——全部由 ProductProfile 承载。
