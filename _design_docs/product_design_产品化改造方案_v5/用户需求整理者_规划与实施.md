# 用户需求整理者 — 功能规划与实施

> 新增 Tool + Asset，实现用户需求的持久化与满足度追踪。
> 主要变更：新增 1 个 Agent、1 个资产文件、1 个分组，修改 7 个创作工具的 reads 和 prompt。
>
> **当前阶段**：v5.1（在 v5 架构上增量扩展）

---

## 一、背景与动机

### 痛点

当前产品中，用户的原始需求在输入后就被 Orchestrator 拆解为工具的 instruction，**没有持久化**。后续每个工具只能通过 instruction 片段间接感知用户意图，**没有统一的、可回溯的需求文档**。这导致：

1. **需求失真**：Orchestrator 将用户需求拆解给各工具时，可能丢失细节或引入中介偏差
2. **无从检查**：`story_checker` 只能检查结构完整性，无法判断"是否满足了用户想要的"
3. **无法追踪**：用户说"我之前说过主角是女性"时，没有任何需求记录可追溯

### 解决方案

新增 **`user_requirements_analyzer`** Tool 和 **`user_requirements.md`** 资产文件，形成：

```
用户输入 → user_requirements_analyzer → user_requirements.md（持久化需求）
                                                      ↓
                              所有创作工具读取 user_requirements.md（保持对齐）
                                                      ↓
                              story_checker 检查每个需求的满足状态（维度 7）
```

## 三、user_requirements.md 文档格式

```markdown
# 用户需求

## 世界观
- [x] 科幻背景，公元2200年，AI 统治人类
- [ ] 人类反抗军在地下秘密活动
- [!] 主角拥有独特能力（用户强调是"基因突变"而非"科技植入"）

## 角色
- [x] 主角：林薇，女性，28岁，AI 实验室的前研究员
- [ ] 配角：陈峰，男性，反抗军队长

## 剧情方向
- [x] 整体基调：黑暗、悬疑、高概念科幻
- [ ] 第一幕：展现 AI 统治下的压抑日常

## 基调与风格
- [x] 叙事风格：冷峻、克制、偏文学性

## 关键元素
- [ ] "记忆碎片"作为核心叙事装置

---

> 状态说明：
> - [x] = 已满足 — 该需求已在对应资产文件中实现
> - [ ] = 未完成 — 该需求尚未在资产中实现
> - [!] = 理解偏差 — 用户意图与当前实现存在偏差，需确认修正
```

### 3.1 需求类别（6 类）

| 类别 | 对应资产文件 | 说明 |
|------|-------------|------|
| 世界观 | worldbuilding.md | 故事发生的世界/环境设定要求 |
| 角色 | characters.md | 用户对特定角色的身份、性格、关系要求 |
| 剧情方向 | act_map.md / sequence_list.md / scene_beat_outline.md | 剧情走向、冲突、转折要求 |
| 基调与风格 | scene_beat_outline.md（贯穿） | 叙事风格、节奏、情绪基调 |
| 关键元素 | 多个资产 | 核心叙事装置、特殊设定 |
| 其他约束 | 不特定 | 篇幅、格式、审查分级等 |

## 四、story_checker 维度 7：用户需求满足度检查

### 4.1 检查规则

| 检查点 | 说明 |
|--------|------|
| 世界观需求 | user_requirements.md 中"世界观"列出的每条需求是否在 worldbuilding.md 中得到体现 |
| 角色需求 | 用户对特定角色的要求是否在 characters.md 中正确实现 |
| 剧情方向需求 | 用户指定的剧情方向、冲突、转折是否在 act_map/sequence_list/scene_beat_outline 中得到遵循 |
| 基调和风格需求 | 用户要求的风格/基调是否在场景节拍中保持一致 |

### 4.2 状态更新逻辑

story_checker 在输出维度 7 时，会为每个需求生成状态建议：

```
### 7. 用户需求满足度检查
- 状态：⚠️ WARNING（2/5 项未完成）
- 详情：
  - [x] 世界观需求：未来世界 AI 统治 — 已体现在 worldbuilding.md
  - [!] 角色需求：主角应是女性科学家 — characters.md 中主角为男性，存在偏差
  - [ ] 剧情方向需求：包含一个背叛转折 — 当前资产中未找到对应情节
```

审计闭环中，Orchestrator 可根据此输出调 `user_requirements_analyzer` 更新 `user_requirements.md` 中的状态标记。

## 八、Prompt 开发要点

### 8.1 `user_requirements_analyzer.md` 的关键指令

```
# 用户需求整理者

## 角色
你是「用户需求整理者」。你不创造故事内容，只提取、整理和追踪用户的原始需求。
你的输出是 user_requirements.md — 一份永久性的需求文档，供所有创作工具参考。

## 输入上下文
- <user_requirements>：已有需求文档（空=创建模式，有=更新模式）
- <user_revision_instruction>：用户本次输入的新需求或修改指令

## 模式判断
- 如果 <user_requirements> 为空 → 创建模式：从用户指令中提取需求
- 如果 <user_requirements> 有内容 → 更新模式：追加新需求或修改已有需求

## 六个需求类别
1. **世界观**：故事发生的世界/环境设定
2. **角色**：角色身份、性格、关系、背景
3. **剧情方向**：故事走向、冲突、转折点、结局预期
4. **基调与风格**：叙事风格、情绪基调、节奏
5. **关键元素**：核心叙事装置、特殊设定元素
6. **其他约束**：篇幅、格式、分级等限制

## 状态标记
每个需求条目使用前缀标记状态：
- [ ] 未完成 — 需求尚未在资产中实现
- [x] 已满足 — 需求已在对应资产中实现
- [!] 理解偏差 — 当前实现与用户意图存在偏差

## 核心原则
1. 不编造需求 — 只提取用户明确表达的内容
2. 保留状态标记 — 更新时保持已有 [x] [!] 标记不变
3. 去重合并 — 相同需求不重复记录，而是合并
4. 不做故事创作 — 只维护需求文档，不生成任何故事内容
5. 分层组织 — 按六个类别分层列出，每个需求单独一行

## 输出要求
- 必须用 <<<USER_REQUIREMENTS_START>>> 和 <<<USER_REQUIREMENTS_END>>> 包裹
- 纯 Markdown 格式
- 每个需求一行
- 不要输出开场白或说明文字
```

### 8.2 创作工具 prompt 中 `<user_requirements>` 的引用方式

每个工具的 prompt 只需在"输入上下文"中加入一行描述，告诉 LLM 有这个标签存在。不需要修改任何生成逻辑。LLM 在收到 `user_requirements.md` 内容后会自动参考。

### 8.3 Orchestrator prompt 中的强化引导

需要强调"必须优先调用"的场景，用具体案例说明，避免 LLM 跳过此步骤。

## 十、补充说明

### 10.1 AssetCard 显示

`user_requirements.md` 以 `user_` 开头，不会被 `getAssetList()` 的 `!path.startsWith('_')` 过滤掉。它会正常出现在 AssetCardPanel 中，显示在"需求管理"分组下，卡片名为"用户需求"。

### 10.2 `ASSET_META` 自动构建

`buildAssetMeta()` 函数遍历 `TOOL_REGISTRY` 的 `writes` 字段构建分组映射。新 Tool 的 `writes: ['user_requirements.md']` 和 `group: '需求管理'` 会被自动收录，无需手动更新 `ASSET_META`。

### 10.3 标签名自动匹配

`contextAssembler.ts` 中的 `filePath.replace(/\.md$/, '')` 逻辑将 `user_requirements.md` 自动转换为 `<user_requirements>` 标签。所有创作工具的 prompt 中只需引用 `<user_requirements>` 即可正确接收内容。

### 10.4 需求文档的生命周期

- **创建**：用户首次输入创作需求时
- **更新**：用户追加或修改需求时
- **状态标记更新**：story_checker 执行后，Orchestrator 判断是否需更新标记
- **清空**：reset_all 执行时
