# v6.9 写作 Agent 优化 · 产品设计文档

## 1. 背景

v6.6 把短剧成文单元定为「一序列（多集）」、`proseSplitUnit=episode` 按集分段续写；v6.7 把 scene_beats 重构为串行批量三段式。但**成文 writer 侧仍停留在「单 Skill 直发、单序列单次、短剧逐集靠 Orche 多次调用」的旧形态**，三个痛点暴露：

- **短剧写不完**：writer 每次 tool_call 只产 1 集，一序列 8-15 集要 Orche 调 8-15 次。短剧常 20+ 序列，直接撞 `MAX_ROUNDS=10`/`MAX_TOOLS_PER_ROUND=5` 配额上限。
- **无批量**：writer 写多序列必须 Orche 逐序列点名，每个序列一次 tool_call，配额进一步吃紧。
- **文件名不达意**：`chapters/S1-1.md` 体现不出「该序列含 E01-E12 这十几集」，与「一序列=多集弧」的语义脱节，用户在资产面板难定位。

根因：writer 没有像 scene_beats 那样完成「序列级原子化 + 批量」改造，短剧的「逐集」粒度被错误地暴露给了 Orchestrator。

## 2. 设计目标

把成文 writer 升级为「**序列级原子化 + 多序列并发批量 + 文件命名产品特化**」：

1. **序列级原子化**——一次 tool_call 产出一个序列的完整正文（短剧内部逐集累积，对 Orche 是 1 次配额）。
2. **多序列并发**——空靶=全量序列并发写作，墙钟从 O(N) 降到 O(N/并发数)。
3. **文件命名产品特化**——短剧 `E01-Exx.md`、长剧 `E01.md`、小说/剧本沿用 `chapters/<seqId>.md`。

> 实现方式经用户拍板：**内部逐集累积**（非单次 LLM 产整序列）。短剧 8-15 集 × 800-1500 字 = 6400-22500 字，单次 LLM 调用易截断/崩坏；引擎在一次 tool_call 内部循环逐集调 writer、累积成整序列文件，既复用现有稳健的逐集逻辑，又把配额消耗从「Orche 多次」收敛为「1 次」。

## 3. 新链路产品逻辑

### 3.1 用户视角

用户在写作期说「把所有序列的正文写出来」（或点批量按钮）：

```
用户一句话（target_chapter 留空）
  → writer 被 Orche 选中
  → 引擎读 sequence_list 识别全部 N 个序列 + 算每序列全局集号区间
  → N 个序列文件【并发池】生成（一次 tool_call 内部，并发上限 3）
  → 每个序列内部：
      短剧：建档骨架 → 逐集循环调 writer（每集 1 次 LLM）→ 累积落盘 E01-Exx.md
      长剧/小说/剧本：建档骨架 → 单次调 writer 产整序列 → 落盘
  → 资产面板逐张出现 N 张正文卡片（建档骨架先现，再逐步填充）
```

用户若只想重做某序列：「重写 S1-3 的正文」→ 带序列号调用 → 只重跑该序列，其余不动。

### 3.2 每个序列文件的产出（按产品分化）

| 产品 | proseSplitUnit | 序列内产出方式 | 文件名 |
|---|---|---|---|
| 短剧 | episode | 引擎逐集循环调 writer（每集 1 次 LLM），`<current_draft>` 累积，集号 writer 内部 `## 第N集` 自管理 | `chapters/E01-E12.md` |
| 长剧 | scene | 单次调 writer 产整集（一序列=一集，5-10 场景） | `chapters/E05.md` |
| 小说 | none | 单次调 writer 产整章 | `chapters/S1-1.md`（不变） |
| 剧本 | none | 单次调 writer 产整序列正文 | `chapters/S1-1.md`（不变） |

> 短剧是本次重点：从「Orche 逐集调用 8-15 次」收敛为「Orche 1 次、引擎内部逐集循环 8-15 次」。writer SKILL 本身已是「单次产 1 集、读 `<current_draft>` 推断下一集编号」，**SKILL 零改动**，引擎循环复用。

### 3.3 为什么这样设计能解决三个痛点

- **写不完（配额）**：序列级原子化把短剧每序列的 Orche 配额从 8-15 次降为 1 次；批量并发把多序列再压成 1 次 tool_call。20+ 序列短剧从「数百次 tool_call」降为「1 次」。
- **无批量**：空靶=全量并发批量，对齐 scene_beats 的批量范式。
- **文件名不达意**：`E01-Exx.md` 直观体现序列所含集范围，资产面板可显示「第1-12集」。

## 4. 文件命名规则

### 4.1 全局集号映射

新函数 `buildEpisodeRangeMap(seqIds, profile, fm) → Map<seqId, [start, end]>`：

- 按 `sequence_list.md` 解析出的 seqIds 顺序遍历
- 每序列集数：
  - 短剧 `one_to_many`：从 `sequences/<ID>.md` 场景表数据行数解析（短剧一集一场景，场景数=集数）；解析失败回退 `episodesPerSequence[0]`（=8）
  - 长剧 `one_to_one`：固定 1
  - 小说/剧本 `none`：不适用（不进 map）
- 累计全局集号：`start = prevEnd + 1`，`end = start + episodeCount - 1`

例：短剧 3 序列，集数 [12, 10, 15] → S1-1→[1,12]、S1-2→[13,22]、S1-3→[23,37]。

### 4.2 路径解析

新函数 `resolveChapterPath(profile, seqId, episodeRangeMap) → string`：

| 产品 | 规则 | 示例 |
|---|---|---|
| 短剧 | `chapters/E${pad(start)}-E${pad(end)}.md` | `chapters/E01-E12.md` |
| 长剧 | `chapters/E${pad(start)}.md`（start=end） | `chapters/E05.md` |
| 小说/剧本 | `chapters/${seqId}.md`（不变） | `chapters/S1-1.md` |

`pad` = 两位零填充（`E01` 而非 `E1`），保证资产面板排序正确。

### 4.3 命名时机

写作期开始时所有 `sequences/*.md` 已冻结存在，`buildEpisodeRangeMap` 一次性算完全局集号映射。建档骨架（0 LLM）先按 `resolveChapterPath` 落盘占位，UI 立即出卡片——与 scene_beats 段①建档同构。

## 5. 数据流要点

- **触发**：UI/对话 → Orche 选 writer（空靶或带序列号）→ `executeTool` WRITER 分支 → 空靶 `runWriterBatchPipeline` / 带靶 `runWriterSequencePipeline`。
- **批量并发**：`runWriterBatchPipeline` 读 sequence_list 得 seqIds + `buildEpisodeRangeMap` → `runWithConcurrency(seqIds, WRITER_CONCURRENCY, runWriterSequencePipeline)` → 汇总单 ToolResult（`writes`=全部序列文件路径、`warnings`=失败清单+软校验）。
- **单序列**：`runWriterSequencePipeline(seqId)` ①建档骨架落 `resolveChapterPath` ②短剧逐集循环调 writer（每集注入 `<current_draft>`=`已产各集`、`<batch_progress>`）③收口覆写落盘 + `extractBehaviorTrack`/`extractForeshadowingState`/`updateBatchProgress`。
- **UI 刷新**：建档即 emit `tool_complete`（writes=[该序列路径]）→ `chatStore` 按 `event.writes` 路径 `refreshFile` 精准刷新；批次收尾 `refreshAllFiles` 兜底。多序列并发事件交错到达，按路径刷新互不干扰。
- **运行时记忆**：`batchProgress`/`foreshadowingState`/`behaviorTrack` 的 key 仍是 `seqId`（归约后），**不依赖文件名**——文件改名对这些记忆零影响。

## 6. 调度形态边界

| 调用形态 | target_chapter | 行为 |
|---|---|---|
| 批量 | 留空 | 读 sequence_list，全序列**并发**写作（一次 tool_call 内并发池），产 N 个文件 |
| 精修 | `S1-3` | 只对 S1-3 跑序列级 pipeline，覆写该序列文件，其余不动 |
| （兜底）| 集级 ID `S1-3-05` | `normalizeToSequenceId` 归约到 `S1-3` 后按精修处理 |

## 7. 兼容性

- **短剧 writer SKILL**：输出约定改（从「回填历史+新集」改为「只输出本集」、引擎负责追加），修复逐集覆写丢集 bug；角色能力/格式规范不变。
- **长剧/小说/剧本 writer SKILL**：零改动（单次产整序列，引擎单次调用）。
- **scene_beats**：零改动（v6.7.5 并发另算；v6.9 新增的 `runWithConcurrency` 供其未来复用）。
- **story_checker**：零改动（reads 是 `sequences/*.md`，不读 chapters，文件改名无感）。
- **foreshadowingState/batchProgress/behaviorTrack**：key 是 seqId 不依赖文件名，零影响。
- **chatStore 精准刷新**：按 `event.writes` 路径刷新，改名后路径变但逻辑不变。
- **assetStore**：`computeLabel` 通用 `chapters/(.+).md` 兜底已能显示「章节 E01-Exx」；本次加 `E\d+` 模式优化为「第1-12集」/「第5集」。
- **旧 `chapters/S1-1.md` 文件**（v6.6/v6.7 生成）：短剧重写时按新规则落 `E01-Exx.md`，旧文件残留需手动清理或 reset_all。**非短剧产品沿用旧名，无迁移问题**。

## 8. 与 v6.7.5 的关系

| 项 | v6.7.5（开发中） | v6.9（本专项） |
|---|---|---|
| 改造对象 | scene_beats 批量 | writer 批量 |
| 并发池 | `runWithConcurrency` | 复用同一 `runWithConcurrency`（v6.9 先落地则 v6.7.5 复用，反之亦然） |
| 调用点 | `runBatchPipeline`（scene_beats） | `runWriterBatchPipeline`（writer） |
| 互不触碰 | 不动 writer | 不动 scene_beats 调用点 |

两者共享通用并发池工具，但调用点各自独立，可分别落地、分别回滚。

## 9. 风险

| 风险 | 缓解 |
|---|---|
| DeepSeek 429 限流（并发 3 × 多序列 × 逐集 LLM 瞬时速率高） | 默认并发 3 + `sendWithRateLimit` 指数退避（1s→2s→4s，3 次），不消耗格式重试预算 |
| 短剧逐集循环某集反复失败 | 该集降级占位 `<!-- 待补第N集 -->` 不阻断整序列落盘，事件流报告；可单独精修重跑 |
| 全局集号映射依赖 sequences 场景表行数，场记残缺致集数误算 | 解析失败回退 `episodesPerSequence[0]`；建档前 `runPipelineSoftValidation` 已保证场记达标 |
| 多序列并发事件交错致 UI 错乱 | emit 单回调同步；`chatStore` 按 `event.writes` 路径精准刷新，`refreshAllFiles` 兜底 |
| 单序列失败拖垮整批 | `runWriterSequencePipeline` 返回 ToolResult 不抛；`runWithConcurrency` worker 总 resolve，失败进 `warnings` |
| 旧短剧 `chapters/S1-1.md` 残留与新 `E01-Exx.md` 共存 | 短剧首次 v6.9 写作前提示 reset_all 或引擎按 seqId 清理旧路径（见开发方案 Wave C） |
