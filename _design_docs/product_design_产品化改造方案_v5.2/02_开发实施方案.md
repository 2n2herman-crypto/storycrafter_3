# 02｜v5.2 开发实施方案

## 一、开发目标

重写 `scene_beats` Agent 的 System Prompt，使其：

1. 不再输出单张 15 列宽表；
2. 改为按序列分组的二层结构；
3. 严格体现 `Philosophy_story_structure.md` 中的场景/节拍定义；
4. 保持与现有资产文件、ToolRegistry、前端展示兼容；
5. 降低 TAG 丢失和 Markdown 表格断裂概率。

---

## 二、核心结构改造

### 旧结构

```text
<<<SCENE_BEAT_OUTLINE_START>>>
| 场景ID | 所属序列 | 场景功能 | 场景目标 | 冲突 | 场景结果 | 时空边界 | 视角人物 | 节拍序号 | 节拍类型 | 动作-反应 | 情绪/信息变化 | 承接钩子 | 关联伏笔 | 关联支线 |
| ... |
<<<SCENE_BEAT_OUTLINE_END>>>
```

问题：全局单表，15 列，每个节拍行重复场景信息。

### 新结构

```text
<<<SCENE_BEAT_OUTLINE_START>>>
# 场景节拍大纲

## S1-1：序列名称

### 场景表
| 场景ID | 场景功能 | 场景目标(Objective) | 冲突与障碍 | 场景结果(Outcome) | 时空边界 | 视角人物 |
| ... |

### 节拍表
| 所属场景 | 节拍序号 | 节拍类型 | 动作-反应描述 | 情绪/信息变化 | 关联伏笔 |
| ... |

## S1-2：序列名称
（重复同样结构）

<<<SCENE_BEAT_OUTLINE_END>>>
```

优势：

- 场景表只写场景级信息，不重复；
- 节拍表只写节拍级信息，降低列数；
- 序列标题形成天然分段，避免长表格连续生成；
- 更符合「场景是容器、节拍是内部张力单元」的概念。

---

## 三、Prompt 改写要点

### 1. 角色定义

从「创建或修改故事的场景节拍大纲」强化为：

- 严格遵循「序列 → 场景 → 节拍」层级；
- 场景是目标-冲突-结果结构；
- 节拍是动作-反应与情绪/权力位移；
- 不写完整剧本，只写提纲。

### 2. 输出要求

明确：

- 必须保留 `<<<SCENE_BEAT_OUTLINE_START>>>` / `<<<SCENE_BEAT_OUTLINE_END>>>`；
- 纯 Markdown；
- 按序列分组；
- 每个序列必须包含一张「场景表」和一张「节拍表」。

### 3. 场景层规则

来自哲学文档：

- 场景开始 = 角色带着明确目标进入；
- 冲突是动力源，无冲突则场景冗余；
- 场景结束 = 目标达成/被拒绝/改变；
- 结果决定下一个场景的入口状态；
- 时间/地点/人物群变化 = 新场景；
- 场景开始和结束必须有情感变化。

### 4. 节拍层规则

来自哲学文档：

- 节拍是场景内部最小叙事单位；
- A 说/做 X → B 以 Y 方式回应；
- 每个节拍必须带来情绪、权力、关系或认知位移；
- 情感方向改变 = 新节拍；情感持续 = 同一节拍延续；
- 节拍不能跨越场景边界。

### 5. 格式稳定规则

新增硬性格式约束：

- 不使用 HTML 表格；
- 表格单元格内不要使用 `|` 字符；
- 若需要列举，用中文顿号、分号或逗号；
- 每个序列结束后再开始下一个序列；
- 先完成全部内容，再输出 END TAG，不要中途解释。

## 四、开发步骤

### Step 1：建立 v5.2 方案文档

创建独立目录：

```text
product_design_产品化改造方案_v5.2/
```

写入：

- `README.md`
- `01_问题分析.md`
- `02_开发实施方案.md`
- `03_开发进度.md`

### Step 2：重写 `scene_beats.md`

修改文件：

```text
src/llm/prompts/scene_beats.md
```

保留：

- 原 Tool ID：`scene_beats`
- 原写入文件：`scene_beat_outline.md`
- 原 TAG：`<<<SCENE_BEAT_OUTLINE_START>>>` / `<<<SCENE_BEAT_OUTLINE_END>>>`

替换：

- 输出结构：单张 15 列宽表 → 序列分组 + 场景表 + 节拍表
- 核心原则：加入场景边界、节拍动作-反应、情绪/权力位移、因果链

### Step 3：验证

执行：

```bash
npx tsc -b
npm run build
```

预期：

- prompt 作为 raw markdown 导入，不应导致 TypeScript 错误；
- Vite build 能成功加载新的 prompt 文本；
- 无需测试框架。

### Step 4：人工使用验证

在产品中输入一个新故事需求，观察：

1. `scene_beats` 是否能稳定完成；
2. `scene_beat_outline.md` 是否出现按序列分组结构；
3. 每个序列是否有场景表与节拍表；
4. 场景表中目标/冲突/结果是否完整；
5. 节拍表是否每条都归属到明确场景；
6. 是否仍能被 story_checker 读取并检查。

## 五、暂不做的改动

本次不修改：

- `outputValidator.ts`：仍只校验 TAG；
- `toolRegistry.ts`：ToolSpec 不变；
- `story_checker.md`：目前检查原则仍适配场景/节拍密度；
- 前端组件：Markdown 渲染可直接展示新结构。

如果 v5.2 prompt 改造后仍出现高频失败，再进入 v5.2.1：增加 `scene_beats` 专用结构校验和更精确的错误重试提示。
