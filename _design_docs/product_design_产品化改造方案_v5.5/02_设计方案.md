# 02 设计方案

## 总览

两条机制协同，把「多轮逐渐明晰」的需求稳定沉淀进 `user_requirements.md`：

- **机制 A**：每轮在 FC 循环**之前**确定性地跑一次 analyzer（合并模式），不依赖 Orchestrator 选择。
- **机制 B**：把「最近 N 条对话窗口」喂给 analyzer 与 Orchestrator，使二者能解析指代性澄清。

数据流（v5.5）：

```
用户输入
  │
  ▼
chatStore.sendMessage(content)
  │  取 chatStore.messages 最近 N 条 → history[]
  ▼
engine.processUserInput(content, history, onEvent)
  │
  ├─【机制A+B】前置：executeTool(user_requirements_analyzer, 合并指令, history)
  │        └─ analyzer 结合 history + 已有 user_requirements.md + 本次输入 → 合并写回
  │
  ▼
FC 循环（Orchestrator，system prompt 内含 history 渲染）
  │  选择创作 Subagent → 生成资产（reads user_requirements.md，已含最新需求）
  ▼
后处理：analyzer 状态标记模式（沿用现状，标 ✅/⬜）
  ▼
返回 response
```

---

## 机制 A：前置确定性需求合并

### 位置
在 [processUserInput](../src/orchestrator/orchestratorEngine.ts) 的 FC 循环 `while` **之前**插入一次 analyzer 调用。

### 输入
- 已有 `user_requirements.md`（由 analyzer 的 `reads` 自动注入）
- 本次 `userInput`
- 最近 N 条对话窗口（机制 B 提供）

### 指令
新建一条「合并模式」instruction（区别于后处理的「状态标记模式」）：

> "结合最近的对话上下文，将用户本轮明确表达的新需求合并进 user_requirements.md。遵循高精度低召回：只记录用户明说的需求，解析指代时以对话上文为准，不推断未说出口的意图。保留所有已有的状态标记（✅/⬜/❌）不变。"

### 复用
直接复用现有 `executeTool(subagent, instruction, history)`（history 为新增参数）。analyzer 的 `reads/writes/outputTags` 不变，仍写 `user_requirements.md`。

### 幂等与成本
- 每轮固定 1 次额外 LLM 调用。可接受（analyzer 输出短、模型便宜）。
- 若本轮用户无新需求（如纯指令"继续"），analyzer 在高精度低召回原则下应原样返回文档，不产生噪声。SKILL.md 需明确此点。

## 机制 B：对话窗口注入

### 窗口大小
取 `chatStore.messages` 最近 **N = 6 条**（约 3 轮 user/system 往返）。理由：指代性澄清通常发生在紧邻的 1–2 轮内；6 条兼顾覆盖与 token 成本。以常量 `HISTORY_WINDOW = 6` 定义，便于调整。

### 数据结构
`chatStore.messages` 的 `ChatMessage.role` 是 `'user' | 'system'`。传给引擎时映射为轻量结构，避免与 OpenAI 的 `ChatCompletionMessageParam` 耦合：

```ts
// types/index.ts 新增
export interface ConversationTurn {
  role: 'user' | 'assistant'   // system → assistant（LLM 语义）
  content: string
}
```

`chatStore` 在回传前做映射：`role: 'system' → 'assistant'`，`role: 'user' → 'user'`，并**排除本次刚推入的 userMsg**（本次输入单独作为 `userInput` 传递，避免重复）。

### 注入点 1：analyzer（经 contextAssembler）
在 `buildAgentPrompt` 增加可选的对话历史段，渲染为独立 XML 标签：

```
<conversation_history>
用户：写个科幻故事
助手：好的，已记录科幻背景。主角有什么设定吗？
用户：一个女科学家
...
</conversation_history>

<user_requirements>...已有文档...</user_requirements>

<user_revision_instruction>...合并指令...</user_revision_instruction>
```

### 注入点 2：Orchestrator（FC 循环）
Orchestrator 的 `messages` 数组初始化时，把 history 映射为真实的 `role: 'user'/'assistant'` 消息**前置**于本次 `userInput`：

```ts
const messages: ChatCompletionMessageParam[] = [
  ...history.map(t => ({ role: t.role, content: t.content })),
  { role: 'user', content: userInput },
]
```

这样 Orchestrator 在选择 Subagent 与生成最终回复时都能理解上文（如"再改上面那个角色"）。

### token 预算
- 现有 `estimateTokens` + `compressMessages`（[orchestratorEngine.ts:54-78](../src/orchestrator/orchestratorEngine.ts)）在 FC 循环内已有截断保护，历史前置后仍受 `CONTEXT_LIMIT_CHARS = 22_000` 约束，无需新增机制。
- analyzer 侧窗口固定 6 条，量小，不做额外压缩。

## 角色映射一致性

| 来源 | chatStore 内 | 传引擎 (ConversationTurn) | OpenAI messages |
|------|-------------|--------------------------|-----------------|
| 用户消息 | `user` | `user` | `user` |
| 系统回复 | `system` | `assistant` | `assistant` |

analyzer 侧渲染为中文「用户：/助手：」纯文本，不涉及 OpenAI 角色。

## SKILL.md 调整（user_requirements_analyzer）

在现有「输入上下文」与「模式判断」中补充：

1. 新增可能出现的 `<conversation_history>` 标签说明：这是最近几轮对话，用于**解析指代**（如"那个""上面说的"），不是需求本身。
2. 合并模式强调：结合 `<conversation_history>` 理解 `<user_revision_instruction>` 中的指代，但仍遵循高精度低召回——只把能从对话中**明确锚定**的需求写入，无法锚定的指代不臆测。
3. 明确"本轮无新需求时原样返回文档"，避免每轮噪声累积。

## orchestrator_v5.md 调整

补一句说明：消息历史中可能包含此前几轮对话，选择 Subagent 与回复时应结合上文理解用户的指代与延续性意图。

## 不改动项

- 下游创作 Skill（worldbuilding / characters / …）：已 `reads` `user_requirements.md`，自动读取到最新合并结果。
- 前端 UI、outputValidator、fileManager：无需改动。
- 后处理状态标记逻辑：保留现状（机制 A 负责内容合并，后处理负责状态标记，职责不重叠）。
