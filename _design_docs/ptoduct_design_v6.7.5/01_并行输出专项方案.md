> v6.7 已落地（代码确认），偏差 ③ 用串行批量替代；本专项正式实现并行。

## 1. 代码现状核实（改造起点）


**并行化前提已逐条代码核实**：

| 前提 | 代码证据 | 结论 |
|---|---|---|
| `runSequencePipeline` 自包含 | 只读 `profileLock`/`fileManager`/`history`；只写 `sequences/${seqId}.md`（[line 1039-1043](../../src/orchestrator/orchestratorEngine.ts#L1039)）；不写 `foreshadowingState`/`batchProgress`（二者仅 writer 路径 [line 718-719](../../src/orchestrator/orchestratorEngine.ts#L718) 与 reset 写） | ✓ 无跨序列共享可变状态 |

| 汇总不依赖顺序 | `results.forEach((r, i) => seqIds[i])` 按下标（[line 1118](../../src/orchestrator/orchestratorEngine.ts#L1118)） | ✓ `runWithConcurrency` 保持下标对齐即可 |

> 关键：`runSequencePipeline` 必须保持自包含是可并行的硬前提，定为 **INV-5**。后续任何改动不得让序列间产生隐式依赖（如共享累加器、读他序列产物等）。

## 2. 目标

- 多序列并发执行，墙钟从 O(N) 降到 O(N / 并发数)
- 对 Orchestrator 仍是 1 次 tool_call（偏差 ④ 收益不丢）
- 429 限流退避重试，不人工干预
- 单序列失败不拖垮整批（v6.7 已有，保留）
- **汇总逻辑、runSequencePipeline、runSingleStep 主体、校验、落盘、UI 事件全不动**

## 3. 方案设计

### 3.1 核心改动：串行 for → 并发池（唯一硬改动）

[orchestratorEngine.ts:1108-1112](../../src/orchestrator/orchestratorEngine.ts#L1108) 的串行循环替换为并发池，**汇总逻辑 1114-1126 逐字不动**：

```ts
// ② 并发池（偏差③闭合：串行 for → runWithConcurrency）
const results = await this.runWithConcurrency(seqIds, PIPELINE_CONCURRENCY,
  (seqId) => this.runSequencePipeline(subagent, pipe, seqId, instruction, history))

// ③ 汇总：与 v6.7 逐字相同（forEach + 下标，runWithConcurrency 保证下标对齐）
```

`tool_start` 仍只 emit 一次（批次开始，[line 1102](../../src/orchestrator/orchestratorEngine.ts#L1102)）——并行不改为每序列 emit。

### 3.2 新增 runWithConcurrency（无第三方依赖）

```ts
/**
 * 有界并发池：最多 limit 个 worker 同时跑，超出排队。
 * out[i] 与 items[i] 下标对齐（无论完成顺序），保证 runBatchPipeline 汇总 seqIds[i] 正确。
 */
private async runWithConcurrency<T, R>(
  items: T[], limit: number, worker: (item: T) => Promise<R>,
): Promise<R[]> {
  const out: R[] = new Array(items.length)
  let next = 0
  const runners = Array.from({ length: Math.min(limit, items.length) }, async () => {
    while (next < items.length) {
      const i = next++
      out[i] = await worker(items[i])
    }
  })
  await Promise.all(runners)
  return out
}
```

### 3.3 并发上限

```ts
/** 全序列并行并发池上限。默认 3（保守防 429）。 */
const PIPELINE_CONCURRENCY = 3
```

默认 3 而非 4：DeepSeek 测试 key 速率限制较紧。联调无 429 可上调。暂不引入 `ProductProfile.pipelineConcurrency` 可配置字段（守 INV-3，避免改 `types/product.ts`）；如未来需产品级可调再加。

### 3.4 429 限流退避（增强，可选但推荐）

**注入点**：`runSingleStep` 的 `this.llm.sendMessage`（[line 968](../../src/orchestrator/orchestratorEngine.ts#L968)）。当前 catch（[line 990](../../src/orchestrator/orchestratorEngine.ts#L990)）是 `catch {}` 无类型识别，429 被当普通错误直接 break 浪费重试预算。

新增私有方法包 sendMessage，429 退避重试**不消耗**格式重试预算（MAX_RETRIES 仍专留给格式错误）：

```ts
/** 429 指数退避重试（复用 classifyLLMError 识别 rate_limit）；非 429 直接抛交原流程。 */
private async sendWithRateLimit(systemPrompt: string, userContent: string): Promise<string> {
  const MAX_429_RETRIES = 3
  for (let i = 0; i < MAX_429_RETRIES; i++) {
    try {
      return await this.llm.sendMessage(systemPrompt, userContent)
    } catch (e) {
      if (classifyLLMError(e).type !== 'rate_limit' || i === MAX_429_RETRIES - 1) throw e
      const backoff = 1000 * Math.pow(2, i) // 1s → 2s → 4s
      this.emit('tool_retry', {
        toolId: '', toolName: '', skillId: '', skillName: '',
        attempt: i + 1, maxAttempts: MAX_429_RETRIES,
        message: `429 限流，${backoff}ms 后重试 (${i + 1}/${MAX_429_RETRIES})`,
      })
      await new Promise((r) => setTimeout(r, backoff))
    }
  }
  throw new Error('unreachable')
}
```

`runSingleStep` [line 968](../../src/orchestrator/orchestratorEngine.ts#L968) `this.llm.sendMessage(...)` → `this.sendWithRateLimit(...)`。退避后仍 429 则抛出，被外层 catch 当普通错误 break → 该 step null → 降级占位/abort（v6.7 既有路径）。

> 复用 [utils/llmError.ts](../../src/utils/llmError.ts) 的 `classifyLLMError`（已识别 `rate_limit`），零新增依赖。

> **不实现自动降并发**：429 退避重试已能兜底；自动降并发（3→2→1）增加状态复杂度，联调若 429 仍频繁再加。两种实现都不破坏汇总契约。

## 6. 风险（基于代码实际）

| 风险 | 依据 | 缓解 |
|---|---|---|
| DeepSeek 429 限流 | 并发 3 × 多序列 × (1+8~15) LLM 瞬时速率高 | 默认并发 3 + `sendWithRateLimit` 指数退避（1s→2s→4s，3 次） |
| `history` 共享被写 | runSequencePipeline 传 `history` 只读 | 代码核实 runSequencePipeline 不改 history ✓ |
| 事件交错致 UI 错乱 | 多序列 tool_retry/complete 交替到达 | emit 单回调同步；`chatStore` 按 `event.writes` 资产路径精准刷新，`refreshAllFiles` 兜底 |
| token 消耗峰值升高 | 并发使调用时间集中 | 总量与串行相同；受 `budget` 硬上限约束（若启用） |
| 单序列失败拖垮整批 | 并发下某序列 abort | `runWithConcurrency` worker 异常会被 Promise.all 捕获？→ **需 worker 内 try 兜底**（见下） |

> **最后一个风险需补**：当前 `runSequencePipeline` 返回 ToolResult（不抛），所以 worker 不会 reject。但 `sendWithRateLimit` 退避耗尽后会 throw，被 runSingleStep catch 后返回 null（不抛出）。整条链路不抛异常 → worker 总是 resolve。**确认无需额外 try**。但实现时仍建议 worker 包 try 转 ToolResult 失败兜底，防御未来改动引入异常。
