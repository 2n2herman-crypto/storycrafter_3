# 02 开发方案 — 7.0 用户化专项

## 1. 架构总览

### 1.1 目标架构

从「浏览器内存 SPA + bundle 焊死 key/skill」演进为「本地 Node 后端 + Vite 前端」前后端分离单机应用。

```
storycrafter_3/
├── package.json          # workspace root，dev 脚本起双进程
├── server/               # Node 后端
│   ├── package.json      # express + tsx + typescript
│   ├── tsconfig.json     # NodeNext / strict
│   └── src/
│       ├── index.ts      # express 入口
│       ├── routes/       # API 路由（llm/config/projects/assets/chat/skills/import/export/relations）
│       ├── services/     # configStore / projectStore / skillRegistry / llmProxy
│       └── util/         # fsQueue（按 projectId 串行）
│   └── data/             # 持久化数据（gitignore）
│       ├── config.json
│       ├── projects.json
│       └── projects/<id>/{metadata.json, chat_history.json, execution_log.jsonl, assets/*.md}
├── web/                  # 前端（M2 从 src/ 迁移）
│   └── src/
│       ├── api/          # 统一后端调用层（CLAUDE.md 强制）
│       ├── orchestrator/ # OrchestratorEngine + HttpFileManager
│       ├── skills/       # 内置 skill（M5 起双源 overlay 的内置源）
│       ├── store/        # zustand stores
│       └── ...
└── shared/               # M5 前后端共享（types / frontmatter 解析器）
```

### 1.2 复用 v6 后端专项

`product_design_v6_后端专项/` 8 份文档已详细设计：后端架构（01）、持久化（02）、Skill 运行时（03）、LLM Proxy（04）、API 契约（05）、迁移路径（06）、开放问题（07）。**详细到可实施**（含接口定义 + 代码骨架）。

v7.1 主体直接复用这 8 份文档，不重新设计后端。本方案只补充 7.0 特有部分：Word 模块整合、文件关系预留、前端美化、版本划分与验收。

### 1.3 进程模型

- **Dev**：双进程（Vite 5173 + Express 3001），root `npm run dev` 用 `concurrently` 起，Vite `/api/*` proxy 到 3001
- **Prod**：单进程单端口，`express.static` 托管 `web/dist/`

## 2. 技术选型

| 维度 | 选型 | 理由 |
|---|---|---|
| 后端框架 | Express + tsx | v6 后端专项已定，体量最小 |
| 存储 | 文件系统（.md + .json + .jsonl） | 单用户无需 DB，assets 真实 .md 可手编 |
| Word 导入 | `mammoth` + `turndown` | 纯 JS 无原生模块，docx→HTML→markdown |
| Word 导出 | `docx` 库 | 纯 JS，程序化生成 .docx |
| MD 导出 | 纯前端 Blob+`<a download>` | 零依赖，资产已在 assetStore |
| 前端组件库 | 继续自研（不引入 Radix 等） | 现有 CSS Module + CSS 变量体系足够 |
| 项目 ID | `crypto.randomUUID()` | 避开中文/空格跨平台问题 |

## 4. Word 模块技术方案

### 4.1 端点设计（独立命名空间，不扩展 assets 端点）

导入/导出是**格式转换服务**，不是项目资产 CRUD——资产写入仍走 `PUT /api/projects/:id/assets/_input_raw.md`。

| Method | Path | 请求 | 响应 |
|---|---|---|---|
| POST | `/api/import/docx` | multipart `file`(.docx)，≤10MB | `{markdown, filename}` |
| POST | `/api/export/docx` | JSON `{markdown, filename}` | binary docx |

### 4.2 导入链路（复用现有 importStore，零改 `_input_raw` 流程）

```
.docx → FileImporter(accept=.md,.docx) → BottomPanel.handleFileImport 分流
  → importDocx(file) → POST /api/import/docx → {markdown}
  → showImportPreview(markdown) → confirmImportToAsset
  → appendInputRaw 写 _input_raw.md（经 HttpFileManager）
  → 下轮 processUserInput 前置 input_normalizer 归一化 [零改]
```

### 4.3 导出链路

- **MD**：纯前端 `web/src/utils/exportMd.ts`（Blob+`<a download>`）
- **Word**：`web/src/api/importExport.ts` → `POST /api/export/docx` → Blob 下载
- **UI**：MVP 只在 `CurrentPanel` 头部加按钮组（作用于当前选中资产），单 AssetCard 级导出推后

### 4.4 FileImporter.tsx 改动

- `accept=".md"` → `".md,.docx"`
- 拖拽校验放行 `.docx`
- 提示文案「拖拽 .md / .docx」
- 分流在 `BottomPanel.handleFileImport`（FileImporter 只传 File）

## 5. HttpFileManager 与 path 校验放宽（关键技术决策）

### 5.1 HttpFileManager

`web/src/orchestrator/fileManager.ts` 新增 `HttpFileManager implements FileManager`，六方法逐字对齐 `readFile/writeFile/clearFile/listAssetFiles/clearAll/clearByPrefix`。底层换 HTTP 调用，`OrchestratorEngine`/`assetStore` 不感知。保留 `InMemoryFileManager` 作 fallback。

### 5.2 ⚠️ path 校验放宽

**冲突**：v6 `05_api_contract` 规定资产 path 只允许 `[A-Za-z0-9_.-]+\.md`（不含子目录），但 v6.1 起实际存在：
- `sequences/<ID>.md`
- `chapters/<ID>.md`
- `_seq/<ID>/*.md`（pipeline 临件）

**修正**：M3 必须放宽为「允许单层/多层子目录 + `..`/绝对路径黑名单 + `path.resolve`+`startsWith(projectRoot)` 兜底」，否则 `HttpFileManager` 一上线就写不进 pipeline 产物。这是 v6 设计与现状的偏差，v7.1 修正。

## 7. v7.2 前端美化方案

### 7.1 暗色主题（半天）
- `web/src/styles/global.css` `:root` 旁加 `[data-theme="dark"]` 覆盖 ~25 个 `--color-*`
- `diff.css` diff 变量加暗色版
- 新建 `web/src/hooks/useTheme.ts`（读写 localStorage、设 `document.documentElement.dataset.theme`）
- `HeaderBar` 加 ☀/🌙 toggle
- **降级**：diff 暗色对比度不足 → 单独暗色 diff 变量 + WCAG AA

### 7.2 响应式（1–1.5 天，当前 0 个 @media）
三档断点：
- ≥1280px `wide`：四栏现状（PanelResizer 可拖）
- 768–1279px `medium`：两栏（左对话+资产 Tab，右当前+基线 Tab）
- <768px `narrow`：单栏 + 顶部 Tab

- 新建 `web/src/hooks/useResponsive.ts`（`matchMedia` 返回 mode）
- `MultiColumnLayout.tsx` 接受 `layoutMode` prop 按模式渲染
- `global.css` 加 `@media` 断点调整 `--header-height`/字号
- **降级**：时间紧先只做 `wide`+`narrow` 两档

### 7.3 视觉调优（1 天）
- `global.css` 加 spacing scale（`--space-1`..`--space-8`）+ 字体层级变量，替换各 Module 硬编码
- 空状态：`AssetCardPanel`/`ChatHistory` 无内容时占位文案+图标
- 加载骨架：`CurrentPanel` `isLoading` 显示 Skeleton（新建 `Skeleton.tsx`）
- AssetCard hover/状态圆点微动画

### 7.4 组件库选型：继续自研
不引入 Radix 等。v7.1 新增 Settings/项目选择器需 Modal/Dropdown 时手写两个轻量 primitive。

## 8. 关键不变式（INV，v7.1 后端改造必须保持）

## 9. 风险与降级

### v7.1
| 风险 | 降级 |
|---|---|
| HttpFileManager 破坏 INV | 六方法逐字对齐 + 保留 `InMemoryFileManager` 作 fallback |
| path 校验过严写不进 sequences/chapters | M3 放宽为子目录 + resolve/startsWith 兜底 |

| mammoth/docx 大文件慢 | 后端 10MB 上限；docx 表格 fidelity 降级为纯文本行 |

### v7.2
| 风险 | 降级 |
|---|---|
| 响应式破坏四栏 diff 工作流 | 先 wide+narrow 两档；narrow 下 diff 上下堆叠 |
